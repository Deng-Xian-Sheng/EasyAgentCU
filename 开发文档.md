## 1. 项目目标与边界

EasyAgentCU 是一个运行在 Deepin V25 X11 虚拟机中的桌面智能体。用户在宿主机浏览器中访问它的 Web 界面，输入任务后，系统调用支持视觉的多模态大模型，让模型通过截图理解当前电脑画面，再通过工具调用操作这台电脑，最终完成任务。

本项目第一版必须满足下面这些硬性边界：

前端使用 Next.js、TailwindCSS、radix-ui、shadcn/ui、prompt-kit。  
后端使用 Golang。  
模型接口使用 OpenAI 兼容的 Chat Completions API，而不是 OpenAI Responses API。  
项目最终交付形态是一个 Go 可执行二进制，启动后监听 `0.0.0.0`，前端静态资源由该二进制直接提供。  
运行环境固定为 Deepin V25 + X11，不支持 Wayland。  
桌面控制默认不做沙箱限制，bash 可直接调用系统命令。  
系统默认单用户部署，不做登录、权限、多租户。  
聊天记录、截图、附件、运行日志都保存在本机磁盘。  
每个会话同一时刻只允许一个运行中的 Agent 任务。

这里明确一个关键实现决策：前端虽然用 Next.js 开发，但发布时不运行 Node 服务，而是导出为纯静态站点，再嵌入 Go 二进制。这样既满足技术栈要求，也满足“通过可执行二进制运行”的需求。

## 2. 最终技术方案

## 2.1 前端方案

前端采用 Next.js App Router 开发，但只做静态导出能力允许的客户端页面，不依赖 Next Server API。所有数据请求都直接打到 Go 后端。

前端依赖固定如下：

| 类别        | 选型                               |
| ----------- | ---------------------------------- |
| 框架        | Next.js 15 + React 19 + TypeScript |
| 样式        | TailwindCSS                        |
| 基础组件    | radix-ui + shadcn/ui               |
| AI 界面组件 | prompt-kit                         |
| 图标        | lucide-react                       |
| 状态管理    | zustand                            |
| 表单校验    | zod                                |
| 提示消息    | sonner                             |
| VNC 客户端  | noVNC                              |

前端不使用 Next API Route，不使用服务端渲染业务逻辑，不使用服务端 actions。页面统一由客户端加载后通过 REST + SSE 获取数据。

## 2.2 后端方案

后端采用 Go 1.24+。

后端依赖固定如下：

| 类别      | 选型                                 |
| --------- | ------------------------------------ |
| HTTP 路由 | go-chi/chi/v5                        |
| 跨域      | go-chi/cors                          |
| 日志      | rs/zerolog                           |
| 数据库    | SQLite + modernc.org/sqlite          |
| ID        | google/uuid                          |
| 重试      | cenkalti/backoff/v4                  |
| 图片处理  | disintegration/imaging + fogleman/gg |
| 配置      | 直接读取环境变量，自定义 config 包   |
| 并发控制  | 标准库 context / sync / channels     |

后端直接调用系统命令完成桌面控制与截图，不引入复杂桌面自动化框架。固定依赖的系统命令如下：

`xdotool`：鼠标、键盘、窗口激活  
`xrandr`：获取屏幕分辨率与缩放后逻辑尺寸  
`scrot` 或 `import`：截图  
`x11vnc`：把 X11 桌面暴露成 VNC 服务  
`websockify`：把 VNC TCP 转成 WebSocket，供 noVNC 使用  
`bash`：模型自由执行命令  
`ps`, `pgrep`, `pkill`：进程检查与清理

推荐强制使用 `scrot` 做截图，`import` 作为备用。文档后续统一按 `scrot` 编写。

## 2.3 模型交互方案

模型使用支持图片输入与工具调用的 OpenAI 兼容 Chat Completions API。第一版按“非流式模型请求 + 前端事件流”实现。也就是：

模型接口本身不依赖 token 级流式输出。  
后端在任务执行过程中，把阶段文本、工具调用、进展、截图、最终答复通过 SSE 实时推给前端。  
这样可以避开很多国产 OpenAI 兼容接口在“流式工具调用”上的不稳定行为，可靠性更高。

模型请求格式固定采用：

1. 用户文本消息。
2. 需要看图时，追加一个包含说明文字和 `image_url(base64)` 的用户消息。
3. 模型返回 `tool_calls`。
4. 后端执行工具。
5. 以 `role: tool` 回传工具执行结果文本。
6. 再按需要追加新的截图用户消息。
7. 循环直到模型输出最终文本。

## 3. 运行拓扑

系统运行时的真实拓扑不是“浏览器直接连模型”，而是下面这条链路：

```text
宿主机浏览器
   │
   ├── HTTP / SSE ───────────────► EasyAgentCU Go 服务
   │                                 │
   │                                 ├── SQLite / 文件系统
   │                                 ├── OpenAI 兼容模型 API
   │                                 ├── xdotool / xrandr / scrot / bash
   │                                 └── 任务调度与 Agent 主循环
   │
   └── noVNC(WebSocket) ─────────► websockify ─► x11vnc ─► X11 桌面(:0)
```

这里有两个并行的画面来源：

第一种是“实时画面”，由 noVNC 显示。它用于用户观看与接管操作。  
第二种是“截图画面”，由后端主动采集、压缩、加标注、持久化。它用于模型分析，也用于会话中的历史进展回放。

这两种画面必须分开实现，不能混为一体。原因很简单：

实时画面适合交互，但不适合作为稳定的模型输入。  
模型分析需要明确的一帧图片、明确的分辨率、明确的时间点、明确的鼠标坐标。  
而历史回放也必须基于已保存的截图帧，否则进度条无法回看。

## 4. 关键实现决策

## 4.1 只做一个页面路由

前端静态导出后，为了避免动态路由处理复杂化，整个应用只保留一个页面入口 `/`。当前会话 ID 通过查询参数承载，例如：

`/`
`/?c=conversation_xxx`

这样静态导出最稳，Go 服务只需要提供一个前端入口和一组 API。

## 4.2 对话与运行分离

“会话”是长期存在的数据。  
“运行”是一次用户发送消息后触发的 Agent 执行过程。

一个会话里会有多轮消息。  
每次用户发送一条新消息，都会创建一个新的 run。  
run 生命周期只有四种状态：

`queued`  
`running`  
`waiting_user`  
`completed`  
`failed`  
`cancelled`

助手消息的可视化内容，本质上都是某个 run 的渲染结果，而不是一次性纯文本。

## 4.3 屏幕窗口必须先显示，再开始推理

收到用户消息后，前端要立即插入一条新的助手消息壳子。这个壳子默认先显示“实时屏幕窗口”，哪怕此时模型还没拿到截图，哪怕后端还在准备第一轮推理。

这条规则必须在前后端两侧都落实：

前端提交消息成功后，立刻插入本地临时 assistant-run 消息。  
后端创建 run 后马上发出 `run.created` 和 `screen.live_ready` 事件。  
前端一旦收到 `screen.live_ready`，立刻让对应消息进入“实时画面模式”。

## 4.4 模型自由调用工具，但工具面固定

不做工作流编排器，不做固定流程树，不做“先浏览器后 bash”这类强约束。  
但必须给模型一个稳定的工具面。第一版工具面固定为：

屏幕观察相关工具  
鼠标键盘相关工具  
等待与窗口聚焦相关工具  
bash 工具

后续再扩展，但第一版不要混入 Playwright、浏览器 DOM、MCP、RPA 等第二套控制体系。否则上下文和错误恢复会变乱。

## 5. 屏幕截图与坐标体系

这是整个项目最关键、最容易做错的部分。必须一次设计对。

## 5.1 三种尺寸同时存在

系统里同时存在三种坐标尺寸：

第一种是桌面真实逻辑尺寸，也就是 X11 当前桌面的宽高，例如 `1920x1080`。  
第二种是模型输入截图尺寸，也就是为了节约 token 和网络传输而缩放后的图片尺寸，例如 `1600x900`。  
第三种是前端展示尺寸，也就是浏览器里屏幕窗口实际渲染成的 CSS 尺寸，例如 `896x504`。

模型返回的坐标，必须明确约定为“模型输入截图尺寸空间”的坐标，而不是前端展示尺寸，也不是浏览器像素坐标。

## 5.2 坐标映射规则

每次截图时，后端必须记录下面这些元数据：

`desktop_width`  
`desktop_height`  
`model_image_width`  
`model_image_height`  
`scale_x = desktop_width / model_image_width`  
`scale_y = desktop_height / model_image_height`

模型返回 `(x, y)` 后，执行工具前统一映射成：

`desktop_x = round(x * scale_x)`  
`desktop_y = round(y * scale_y)`

模型永远不需要知道浏览器里展示了多大，也不需要知道前端缩放比例。  
前端点历史进展时，只切换图片与时间点，不参与工具执行坐标换算。

## 5.3 截图文件必须保存三份

每个进展节点保存三份文件：

第一份是原始截图 `raw.png`，分辨率等于桌面逻辑尺寸。  
第二份是模型输入图 `model.png`，缩放后、可加轻度压缩、用于 base64 发给模型。  
第三份是带标注的展示图 `annotated.png`，在图上画出鼠标位置圈、箭头和“鼠标指针”四字，用于模型观察与前端历史回放。

实时画面不用这三份文件，它直接走 noVNC。

## 5.4 鼠标鲁棒性优化必须落地到实现

不是只写在提示词里，而是要形成固定机制。

每次提供给模型的截图，后端都同时附带：

1. 一张带鼠标标注的截图。  
2. 一段文本元数据，明确写出当前鼠标坐标、桌面分辨率、模型截图分辨率。

模型提示词中再强制要求：  
凡是点击小控件、输入框、关闭按钮、页签、复选框、下拉箭头等高精度目标，必须先移动，再看图确认，再点击。不能盲点。

## 6. 页面骨架与原型还原要求

这一部分先定义整体外观骨架，组件细节在下一部分展开。

整个应用是深色主题。背景不是纯黑，而是偏暖的深灰黑，主背景建议从 `#161616` 到 `#1d1d1d` 做极轻微纵向渐变。所有边框都很弱，接近 `rgba(255,255,255,0.08)`，不能出现亮色边框。页面整体视觉要接近你给的原型：安静、厚重、圆角大、留白大。

页面分成左右两栏。

左栏是会话侧边栏，展开状态宽度固定 `384px`。背景略深于主内容区，顶部有一个窄工具栏，里面只有折叠按钮。折叠后宽度变成 `64px`，只保留图标。左栏内部从上到下依次是：新聊天按钮、搜索聊天按钮、“你的聊天”标题、会话列表。会话项是圆角矩形，当前激活项底色比普通项亮一层，文字白色，其余项是低对比度灰白。整个左栏自身可滚动。

右栏是主内容区。顶部固定一个高度约 `72px` 的头部，只显示产品名 `EasyAgentCU`，左对齐，字重中等，字号大约 `24px` 到 `28px`。头部底部有一条很淡的分隔线。头部以下是消息滚动区和底部输入区。

空会话状态要还原原型里的居中首页效果：  
中间是一块大圆角输入框，宽度大约 `960px`，高度大约 `160px` 到 `180px`。输入框下方是一排任务分类胶囊按钮，再下面是一组推荐任务列表项。整体垂直居中略偏上。这个状态下右侧没有任何消息，只显示首页输入区和推荐区。

有对话时，消息区依旧居中显示，消息列最大宽度固定 `960px`。  
用户消息是右对齐的气泡，背景比页面稍亮，圆角很大，靠右悬浮。  
助手消息不是传统单气泡，而是一组“消息段落块”的容器，内部可出现阶段文本、bash 工具块、进展列表、屏幕窗口、最终回答。  
底部输入框固定吸附在页面底部中央，不跟随消息一起滚走。输入框下方有一句小号免责声明：`EasyAgentCU 也可能会犯错。请核查重要信息。`

## 7. 前端页面与路由设计

## 7.1 路由清单

虽然最终只保留一个实际页面路由，但前端要按“页面级状态”来设计，而不是只做一个巨型组件。

实际路由如下：

`/`  
主应用页面。空会话、已选会话、运行中、等待用户操作、查看历史截图、接管全屏，全部都发生在这里。

`/?c=<conversation_id>`  
主应用页面的会话选中态。前端根据查询参数加载指定会话。

除此之外不再新增独立业务路由。搜索聊天、图片预览、全屏接管都做成当前页上的面板或浮层。

## 7.2 页面级状态

`/` 页面必须支持 6 种稳定状态：

第一种是“未选择会话的首页空态”。显示大输入框、分类胶囊按钮、推荐任务列表。对应你给的首页原型。

第二种是“已选择会话但无消息”。右侧头部仍显示 `EasyAgentCU`，消息区为空，底部输入框可直接发消息。

第三种是“会话正常浏览态”。显示用户消息和助手消息历史。

第四种是“运行中态”。在最后一条助手消息中显示实时屏幕窗口或进展截图窗口，同时顶部或中部显示阶段文本、bash 调用、进展文本。

第五种是“等待用户操作态”。最后一条助手消息切换为回答文本，并显示“我已经操作”按钮。屏幕窗口可点开进入全屏接管。

第六种是“全屏接管态”。以全屏浮层展示 noVNC 画面，用户可直接操作远程桌面。顶部提供退出按钮，底部提供“我已经操作”按钮。

## 8. 原型图还原说明

## 8.1 整体视觉

整体采用暗色主题。页面不是纯黑，而是高灰低饱和的深色背景。推荐全局颜色：

`--bg-app: #151515`  
`--bg-sidebar: #111111`  
`--bg-panel: #1c1c1e`  
`--bg-panel-2: #242426`  
`--bg-bubble-user: #2b2b2d`  
`--bg-input: #2a2a2c`  
`--text-primary: rgba(255,255,255,0.92)`  
`--text-secondary: rgba(255,255,255,0.68)`  
`--text-muted: rgba(255,255,255,0.46)`  
`--border-soft: rgba(255,255,255,0.08)`  
`--border-strong: rgba(255,255,255,0.14)`

圆角必须偏大。侧栏项、用户气泡、输入框、屏幕窗口外壳都要有明显圆角，不能尖锐。阴影很弱，主要靠层次和亮度差区分区域，而不是强阴影。

字体使用系统中文无衬线。标题和关键文字用中等字重，正文使用常规字重。不要出现高饱和品牌色，整体以灰、白、少量浅蓝灰为主。

## 8.2 布局尺寸

桌面宽屏下采用固定双栏布局。

左侧边栏展开宽度 `384px`。  
主内容区最小宽度 `960px`，占剩余全部空间。  
聊天消息列最大宽度 `960px`，左右居中。  
顶部栏高度 `72px`。  
底部输入区整体高度约 `190px`，包含输入框和免责声明。  
底部输入框本体宽度 `min(960px, calc(100vw - 120px))`。  
消息区上下留白都较大，首屏看起来疏朗。

## 8.3 首页空态还原

对应你给的首页图：

右侧主内容区中间偏上，放置一个大输入框。输入框是深灰圆角矩形，宽约 `960px`，高约 `160px`。左上角 placeholder 为“描述任务”。左下角有一个加号按钮，圆形或仅图标，表示添加文件。右下角是发送箭头按钮，灰色禁用态时半透明。

输入框下方先放一排胶囊按钮。按钮边框细、圆角完整、高度约 `48px`。文案如“建议 / 报告 / 操作 / 电子表格 / 演示文稿”。当前选中项底色略亮，图标与文字白色，未选中项透明底配细描边。

胶囊按钮下方是一组推荐任务列表。每一项都是一行，左侧是小图标和标题，右侧是圆形播放样式按钮。项与项之间有横向分隔线，整体宽度比输入框略窄，约 `700px` 左右，居中。

## 8.4 会话态还原

对应你给的聊天原型：

左侧边栏顶部是两个大按钮区域。“新聊天”和“搜索聊天”不是纯图标，而是图标加文字，左对齐。按钮底色默认透明，hover 时出现浅底。下方是“你的聊天”标题，字号比正文略小但比辅助文字略大。再下面是会话列表，每个会话项一行，高度约 `52px`，当前选中项有深灰背景和大圆角。

右侧头部只显示 `EasyAgentCU`，左对齐，距离左边界约 `28px`。头部下方一条极细分割线，贯穿内容区。

消息区内，用户消息总在右边，最大宽度约 `520px`。使用圆角气泡，四边内边距充足，文字居中偏左，看起来像悬浮胶囊。

助手消息总在左边，但不是单一气泡。它是一个宽内容容器，内部的不同块垂直排列，中间留出 `12px` 到 `16px` 间距。比如先出现“我正在点击”，下面出现屏幕窗口，再下面可能出现 bash 工具块，或者只出现最终回答文本。

## 9. 前端组件清单

基础层：shadcn/ui、radix-ui、prompt-kit 原生组件。

## 10. 组件详细规格

## 10.1 AppShell

作用是整页布局容器。

左侧固定边栏，右侧主区。边栏支持展开与折叠。折叠时只保留图标列，宽度 `64px`；展开时宽度 `384px`。切换动画时长 `200ms`，使用 `ease-out`。主内容区跟随平滑变化，不出现跳闪。

实现基于普通 `div + CSS Grid` 即可，不需要复杂布局库。  
左栏使用 `overflow-y-auto`。  
右栏由 `topbar + chat body + composer footer` 三部分组成。

## 10.2 Sidebar

侧栏由以下子区块组成：

顶部工具栏。高度 `64px`，右上角放折叠按钮。展开态下按钮靠右，折叠态下居中。按钮为幽灵按钮，hover 出现圆角浅底。图标用 Lucide 的 `PanelLeftClose` / `PanelLeftOpen`。

快捷操作区。包含“新聊天”“搜索聊天”。每一项高度 `44px`，左侧图标，右侧文字。展开态显示文字，折叠态只显示图标并用 Tooltip 提示。基础组件使用 shadcn `Button` + `Tooltip`。

会话分组标题。文案是“你的聊天”。样式为小号半透明文字，左边距与按钮对齐。

会话列表。每项单行显示标题，超长省略。当前选中项背景为 `rgba(255,255,255,0.08)`，普通 hover 为 `rgba(255,255,255,0.04)`。边距小，圆角 `18px`。点击切换会话。右键菜单暂不做。

列表滚动使用 shadcn `ScrollArea`。

## 10.3 EmptyHome

这是首页空态核心组件，对应原型里中央大输入框和下方建议区。

上半部分是大输入框。  
中间是一排分类胶囊。  
下半部分是推荐任务列表。  
整个块在主内容区横向居中，纵向位置约在可视高度的 `38%` 附近，不要完全居中，否则太靠下。

输入框本体基于 `prompt-kit` 的 `PromptInput` 二次封装。  
但默认 prompt-kit 样式不满足原型，需要覆盖成大尺寸版本：

输入框圆角 `32px`。  
背景 `--bg-input`。  
内部上下 padding 很大。  
文本域最小高度 `108px`。  
左下角是上传按钮。  
右下角是发送按钮。  
placeholder 为“描述任务”。

分类胶囊建议基于 shadcn `Button` 自定义，而不是直接用 PromptSuggestion，因为这里更像筛选器，不是纯建议 prompt。推荐数据固定写死前端：

`建议`  
`报告`  
`操作`  
`电子表格`  
`演示文稿`

推荐任务列表每项使用自定义 `SuggestionRow`，左侧图标和标题，右侧圆形按钮。标题最多单行，超长省略。点击任意一项，直接把文案填入输入框，不自动发送。

## 10.4 Composer

底部输入区是整个产品最常用的交互部件，必须单独封装，不要散落在页面里。

组成如下：

最外层固定在底部。宽度和消息列一致。上方可有很淡的背景模糊。  
中间是一块大圆角输入容器。  
容器内部上层是附件预览区，下层是文本输入区和按钮区。  
容器下方是免责声明。

输入组件使用 `prompt-kit` 的 `PromptInput`。  
文件拖拽和选择使用 `prompt-kit` 的 `FileUpload`。  
但附件展示必须自定义成 `AttachmentChip` 列表，支持图片缩略图和文件名。

默认状态下按钮为“发送”。  
当前会话存在运行中的 run 时，发送按钮切换为“停止”。  
点击“停止”调用后端取消接口。

输入规则如下：

回车发送。  
Shift+Enter 换行。  
支持粘贴图片和文件。  
支持拖拽文件。  
支持同时携带文本和多个附件。  
发送后清空输入框和本地附件缓存。  
运行中仍允许用户继续输入，但点击发送前要先提示“当前任务仍在运行，是否停止并发送新消息”，第一版可以做成简单阻止，不允许并发发送。

免责声明文字固定：

`EasyAgentCU 也可能会犯错。请核查重要信息。`

样式用小号、居中、低对比度。

## 10.5 UserMessage

用户消息组件尽量简单。用 `prompt-kit` 的 `Message` 思路，但建议自定义实现，因为原型里用户消息没有头像，也不需要标准 message row。

渲染规则：

整体右对齐。  
文本消息显示为单个深灰气泡。  
有图片附件时，图片卡片显示在用户文本上方。  
多附件纵向排列。  
图片卡片支持点击放大。  
文本区支持 markdown 但默认按纯文本展示即可。

用户上传的带标注图片需要忠实还原原型：  
图片卡片宽度约 `420px` 到 `560px`，圆角 `20px`，背景稍亮，内部图片居中显示。下方可有一个小下载按钮或查看原图按钮，但第一版非必需。

## 10.6 AssistantRunMessage

这是最核心的复杂组件。一个运行中的助手消息不是一段文本，而是一个“运行容器”。它内部根据 run 状态渲染不同块。

容器宽度跟消息列一致，但实际内容块偏左布局。容器内部允许以下块按顺序出现：

阶段回答文本  
bash 工具调用块  
进展文本时间线  
屏幕窗口  
最终回答文本  
用户接管操作面板

这些块的显示规则不是任意组合，而是固定状态机，后面会写。

## 10.7 PhaseTextBlock

用于展示阶段性回应，例如“收到您的最新说明，我会按您圈出的输入框进行操作”。

它看上去像一段左对齐的正文，不加外层卡片。字体比普通说明略大，行高舒展。最大宽度建议 `760px`。这块在屏幕窗口上方出现。

一旦最终回答文本出现，这块必须从界面移除，不再显示。

这块可以直接用 `prompt-kit` 的 `Markdown` 渲染内容，但要关闭默认 prose 强样式，改成更轻的聊天正文样式。

## 10.8 BashToolBlock

bash 调用块用于展示模型调用 bash 的过程和结果。推荐基于 `prompt-kit` 的 `Tool` 组件思路，但不要直接使用默认外观，要封装成更贴近原型的样式。

块结构分三层：

头部显示 `bash` 标签、状态徽标、执行时长。  
中部显示命令正文，使用 `prompt-kit` 的 `CodeBlock`。  
底部按需显示标准输出、标准错误、退出码。输出超长时默认折叠，支持展开查看。

状态只有四种：

`pending`  
`running`  
`success`  
`error`

`running` 时头部右侧显示小型 loader。  
`error` 时底部 stderr 区域用暗红色弱提示，但不要鲜红刺眼。

一旦最终回答文本出现，这个块也必须从界面移除。

## 10.9 ProgressTimeline

进展文本位于屏幕窗口上方，默认收起，只显示最新一条。

这个组件不能直接用 prompt-kit 现成组件，必须自定义。建议内部使用 shadcn `Collapsible`。

收起态：  
只显示一行，左侧是状态词或小图标，中间是最新进展文本，右侧是展开箭头。  
整行可以点击展开。  
文本超长单行省略。  
风格接近原型里的“我正在点击 >”“等待页面加载 …”。

展开态：  
显示纵向时间线。每一项左边是小圆点，圆点之间用竖线连接。右边是进展文本。最新项高亮。  
每一项都可点击。点击后，下方屏幕窗口切换到该进展对应的截图。  
当前选中的进展项需要有浅底高亮。  
展开区域最大高度约 `360px`，内部可滚动。

每条进展数据必须绑定 `progress_id` 和 `screenshot_id`，不能只靠数组下标。

## 10.10 ScreenWindow

这是屏幕画面显示组件，是产品的视觉中心。

外层看起来像一个桌面窗口。由三部分组成：

顶部是可选标题区。显示当前进展文案，比如“等待页面加载…”或“我正在点击”。  
中间是画面区。显示截图或 noVNC 实时画面。  
底部是时间进度条。左到右是历史节点，最右侧固定为“直播”。

整体外框是大圆角矩形，背景比页面稍亮，内边距明显。内部真实画面区域再包一层较小圆角。比例固定 16:9，默认最大宽度约 `980px`，高度自适应。页面窄时整体缩小，但始终保持比例。

画面区有两种模式：

历史模式。显示某个进展对应的截图。图片 `object-contain`，上下左右居中。  
实时模式。显示 noVNC 画布。右下角可有一个很淡的“实时”标识。

用户点开全屏时，如果当前是实时模式，进入 noVNC 全屏接管。  
如果当前是历史模式，进入图片预览全屏，不可操作桌面。

## 10.11 ScreenProgressBar

这个组件不能直接用原生 slider 外观，要自定义成原型里的“时间线 + 节点”。

底部是一条细横线，颜色浅灰。  
线上的每个进展节点渲染为一个小圆点。  
当前选中节点变大并高亮。  
最右端固定一个“直播”节点，文字显示“直播”或“实时”。  
可以点击任意节点跳转。  
可以拖动滑块头左右移动，但第一版直接做“点击节点切换”已经足够稳定；拖动可以通过 Radix Slider 做二期增强。若第一版实现拖动，必须支持吸附到最近节点。

点击时间线中的某个点后，进展时间线列表也要同步高亮对应项。  
点击进展时间线列表某项后，底部时间条也要同步切换。

## 11. 助手消息显示状态机

这一段必须严格执行，否则界面会乱。

定义一个助手 run 消息包含这些逻辑字段：

`phase_text`  
`bash_blocks[]`  
`progress_items[]`  
`screen_mode`，值为 `live | history | hidden`  
`final_answer`  
`needs_user_action`，布尔值

前端渲染规则如下：

当 run 刚创建但还没有截图时，显示 `screen_mode=live` 的屏幕窗口，可以只显示加载态 noVNC 容器。此时如果还没有阶段文本，就显示一条简短 loading 文案，例如“正在思考”。

当 run 有阶段文本且还没有最终回答时，显示阶段文本。  
当 run 有 bash 工具块且还没有最终回答时，显示 bash 工具块。  
当 run 有进展项且还没有最终回答时，显示进展时间线和屏幕窗口。此时屏幕窗口默认显示实时画面，用户点击历史节点时切到历史模式。

当 `final_answer` 出现后，必须立即隐藏：

阶段文本  
bash 工具块  
进展时间线  
屏幕窗口

只保留最终回答文本。  
这条规则必须写进前端条件渲染里，不允许同时显示“最终回答”和“进展屏幕”。

当 `needs_user_action=true` 时，在最终回答文本下方显示“我已经操作”按钮。点击后自动发送一条用户消息：`我已经操作`。

## 12. 全屏接管设计

## 12.1 打开方式

用户点击实时屏幕窗口时，打开全屏浮层。  
只有实时模式可以进入可操作接管。历史截图模式点开后只能看大图，不可操作。

## 12.2 全屏浮层样式

浮层覆盖整个 viewport，背景接近黑色。  
顶部有窄工具栏，左侧是“退出全屏”，中间显示当前状态，右侧可显示连接状态。  
中间是最大化的 noVNC 画面，保持比例，黑边填充。  
底部固定一个主按钮“我已经操作”。

## 12.3 行为要求

进入全屏后，前端不拦截鼠标键盘，直接让 noVNC 接收操作。  
用户点击“我已经操作”后：

先退出全屏。  
再自动向当前会话发送一条用户消息，内容固定为 `我已经操作`。  
发送成功后，当前 run 若处于 `waiting_user`，后端应继续执行新的 run 或继续原流程，后面在后端部分定义。

## 13. 基于现有库的组件选型结论

这里给出明确结论，程序员照着选，不需要再判断。

`PromptInput`：使用 prompt-kit，作为 Composer 核心输入框底座。  
`FileUpload`：使用 prompt-kit，处理拖拽与选择文件。  
`Markdown`：使用 prompt-kit，用于阶段文本和最终回答文本。  
`CodeBlock`：使用 prompt-kit，用于 bash 命令与输出。  
`ChatContainer`：可以使用 prompt-kit，但由于本项目有固定底部输入区和复杂消息结构，建议只借鉴逻辑，不直接整套接入。  
`Tool`：只借用思路，不直接用默认样式。封装成 `BashToolBlock`。  
`Loader`：使用 prompt-kit，显示运行中小动画。  
`Dialog`：使用 shadcn/ui，做图片预览和全屏接管容器。  
`Collapsible`：使用 shadcn/ui，做进展文本展开收起。  
`ScrollArea`：使用 shadcn/ui，做侧栏和长进展滚动。  
`Tooltip`：使用 shadcn/ui，做折叠侧栏图标提示。  
`Slider`：使用 Radix Slider，自定义时间轴拖动。第一版也可暂缓拖动，仅保留点击节点切换。  
`Button`、`Input`、`Badge`、`Separator`、`DropdownMenu`：使用 shadcn/ui。

## 14. 前端数据模型设计

## 14.2 页面渲染节点模型

虽然前端内部存的是归一化数据，但消息区渲染时不要直接遍历 `messagesById`。  
应该先把会话中的消息整理成一条有序渲染链。

定义一个页面渲染节点：

```ts
export type RenderNode =
  | { type: "user-message"; id: ID; messageId: ID }
  | { type: "assistant-run"; id: ID; assistantMessageId: ID; runId: ID }
```

生成规则很简单：

会话消息按 `createdAt` 升序排列。  
用户消息生成 `user-message` 节点。  
助手消息必须从 `runId` 找到对应 run，生成 `assistant-run` 节点。  
找不到 run 的助手消息不能直接丢弃，要渲染一个错误占位块，并触发后台补拉取。

这样即使 SSE 中途断开，页面仍能用 HTTP 重建渲染结构。

## 15.2 conversation-store

负责会话列表和会话消息索引。

```ts
interface ConversationStore {
  conversationIds: ID[]
  conversationsById: Record<ID, ConversationSummary>
  conversationMessageIds: Record<ID, ID[]>
  messagesById: Record<ID, MessageRecord>
  upsertConversation(conversation: ConversationSummary): void
  upsertConversations(conversations: ConversationSummary[]): void
  upsertMessage(message: MessageRecord): void
  upsertMessages(messages: MessageRecord[]): void
  setConversationMessages(conversationId: ID, messageIds: ID[]): void
  appendMessageToConversation(conversationId: ID, messageId: ID): void
  removeConversation(id: ID): void
  clear(): void
}
```

`conversationMessageIds` 是渲染顺序的主来源。  
不要每次都从 `messagesById` 扫描过滤，那样消息多了以后会反复重算。

## 15.3 run-store

负责运行时相关的复杂实体，包括 bash、进展、截图。

```ts
interface RunStore {
  runIds: ID[]
  runsById: Record<ID, RunRecord>
  bashBlocksById: Record<ID, BashBlockRecord>
  progressItemsById: Record<ID, ProgressItemRecord>
  screenshotsById: Record<ID, ScreenshotRecord>

  upsertRun(run: RunRecord): void
  upsertRuns(runs: RunRecord[]): void

  upsertBashBlock(block: BashBlockRecord): void
  upsertProgressItem(item: ProgressItemRecord): void
  upsertScreenshot(screenshot: ScreenshotRecord): void

  appendBashBlockToRun(runId: ID, bashBlockId: ID): void
  appendProgressToRun(runId: ID, progressId: ID): void
  appendScreenshotToRun(runId: ID, screenshotId: ID): void

  setRunSelectedProgress(runId: ID, progressId?: ID): void
  setRunSelectedScreenshot(runId: ID, screenshotId?: ID): void
  setRunScreenMode(runId: ID, mode: "live" | "history" | "hidden"): void
  mergePhaseText(runId: ID, text: string): void
  setFinalAnswer(runId: ID, text: string): void
  setRunStatus(runId: ID, status: RunStatus): void
  clear(): void
}
```

这里有三个关键点：

第一，`phaseText` 必须支持覆盖式更新，也支持增量拼接。因为后端可能发送 `set`，也可能未来扩展成 `delta`。  
第二，`selectedProgressId` 和 `selectedScreenshotId` 是纯前端 UI 状态，但放进 `RunRecord` 最方便联动。  
第三，历史截图切换和实时画面切换都由 `screenMode` 驱动，不要在组件内部临时维护一套重复状态。

## 16. 前端初始化流程

浏览器打开页面后的启动顺序必须固定，不能边想边写。

## 16.1 启动时序

页面加载后按下面顺序执行：

第一步，读取 URL 查询参数中的 `c`。  
第二步，调用 `GET /api/conversations` 拉取会话列表。  
第三步，如果 URL 中存在 `c`，并且该会话存在，则加载该会话。  
第四步，如果 URL 中不存在 `c`，则进入首页空态，不主动创建会话。  
第五步，如果加载了某个会话，则并行执行：

`GET /api/conversations/:id/messages`  
`GET /api/conversations/:id/runs`  
`GET /api/conversations/:id/events/stream`

第六步，把消息和 run 全量归一化后，再渲染消息区。  
第七步，SSE 连接建立后再接受实时更新。

页面一刷新，不需要依赖本地缓存恢复完整数据。可以保留少量 UI 偏好，比如侧栏折叠状态，但消息历史必须以服务端为准。

## 16.2 当前会话切换

点击侧栏会话项时流程如下：

先更新 URL 查询参数。  
再清理当前会话的 SSE 连接。  
再并行拉取新会话的 messages 和 runs。  
拉取完成后建立新的 SSE 连接。  
最后把消息区滚动到底部。

切换会话时不要保留上一个会话的展开进展项和全屏状态。  
这些 UI 状态要重置。

---

## 17. 用户发送消息的前端事件流

这是第一个真正关键的链路。

## 17.1 发送前校验

点击发送按钮或回车提交时，前端必须按下面顺序判断：

当前是否已有运行中的 run。  
如果有，直接提示“当前任务仍在运行，请先停止当前任务”。第一版不允许并发。  
输入文本是否为空且附件列表是否为空。两者都为空则禁止发送。  
如果附件中存在大于后端限制的文件，直接提示失败。  
如果图片预览 URL 是本地 blob，不必先上传单独附件接口，直接走消息发送的 multipart/form-data 即可。

## 17.2 发送过程

推荐接口为 `POST /api/conversations/:id/messages`，使用 `multipart/form-data`。  
如果当前没有会话，先调用 `POST /api/conversations` 创建会话，再发消息。

前端执行时序如下：

第一步，必要时创建会话。  
第二步，构造本地临时用户消息，立即插入列表。  
第三步，清空输入框和附件。  
第四步，请求服务端发送消息。  
第五步，成功后用服务端返回的正式用户消息替换临时消息。  
第六步，服务端同时会返回新建的 assistant 消息壳和 run 基础信息，前端立即插入。  
第七步，若当前会话 SSE 未连接，则立刻连接。  
第八步，等待 SSE 推送后续运行事件。

临时消息结构需要加一个本地字段：

```ts
type OptimisticStatus = "pending" | "confirmed" | "failed"
```

但这个字段不必进入全局实体模型，可以只在发送 hook 内部维护，再映射到渲染层。

## 17.3 创建助手消息壳子的前端规则

服务端一旦接受消息，前端必须马上让界面里出现一个助手消息容器。  
这条容器默认规则如下：

`status = queued`  
`phaseText = ""`  
`finalAnswer = ""`  
`screenMode = live`  
`liveReady = false`

渲染时看到 `screenMode=live` 且 `liveReady=false`，显示一个带 loading 的屏幕窗口骨架，占住画面空间。  
这样用户不会觉得系统“没反应”。

## 18. 助手运行时的前端渲染逻辑

## 18.1 AssistantRunMessage 组件输入

这个组件只接收 `runId` 和 `assistantMessageId`。  
其余全部数据自己从 store 中取，避免父组件层层透传。

建议写一个 selector：

```ts
interface AssistantRunViewModel {
  run: RunRecord
  bashBlocks: BashBlockRecord[]
  progressItems: ProgressItemRecord[]
  screenshots: ScreenshotRecord[]
  selectedScreenshot?: ScreenshotRecord
  latestProgress?: ProgressItemRecord
}
```

## 18.2 视图推导规则

`selectedScreenshot` 的推导规则必须固定：

如果 `run.screenMode === "live"`，则不选历史图。  
如果 `run.screenMode === "history"` 且 `selectedScreenshotId` 存在，则取它。  
否则默认取最新进展对应截图。  
若连最新进展截图都没有，则取最后一张 screenshot。  
再没有就显示空骨架。

`latestProgress` 取 `progressItems` 的最后一项。  
`bashBlocks`、`progressItems`、`screenshots` 都必须按时间升序排序后再渲染。

## 18.3 显示优先级

这一段是组件最关键的条件分支。

当 `run.finalAnswer` 非空时：

只显示最终回答文本。  
若 `run.needsUserAction` 为真，则在回答下方显示 `HumanActionPanel`。  
绝对不显示阶段文本、bash、进展时间线、屏幕窗口。

当 `run.finalAnswer` 为空时：

若 `run.phaseText` 非空，则显示 `PhaseTextBlock`。  
若 `bashBlocks.length > 0`，则显示一组 `BashToolBlock`。  
若 `run.liveReady` 为真或 `progressItems.length > 0`，则显示 `ProgressTimeline + ScreenWindow`。  
若以上都没有，但 `run.status` 仍是 `queued/running`，则显示默认 loading 文案“正在思考”。

## 19. SSE 事件协议在前端的处理方式

这一部分先从前端角度定义。后端接口文档部分会按这个协议落地。

## 19.1 连接方式

每个已选会话建立一个 SSE 连接：

`GET /api/conversations/:id/events/stream`

服务端持续推送该会话内的消息和 run 事件。  
前端只需要一条连接，不要为每个 run 单独开 SSE。

使用 `EventSource` 即可，不要自己封装 fetch streaming，SSE 更稳。

## 19.2 事件基础格式

所有 SSE data 必须是统一 JSON 对象：

```json
{
  "event_id": "evt_xxx",
  "conversation_id": "conv_xxx",
  "run_id": "run_xxx",
  "seq": 12,
  "type": "run.progress.created",
  "data": {}
}
```

前端收到事件后不要直接假设顺序绝对正确，而要做两层保护：

第一层，`event_id` 去重。  
第二层，对每个 `run_id` 记录最后处理的 `seq`。若收到的 `seq` 小于等于已处理序号，则丢弃。若出现跳号，比如当前是 12，收到 15，说明中间事件丢失，要立即触发该 run 的全量补拉接口。

这一步非常重要。只靠 SSE 连接本身，页面在弱网下很容易丢中间状态。

## 19.3 前端需要支持的 SSE 事件类型

第一版固定支持下面这些事件：

```ts
type SSEEventType =
  | "conversation.updated"
  | "message.created"
  | "run.created"
  | "run.status.changed"
  | "run.live.ready"
  | "run.phase_text.set"
  | "run.bash.started"
  | "run.bash.updated"
  | "run.bash.completed"
  | "run.progress.created"
  | "run.screenshot.created"
  | "run.answer.final"
  | "run.waiting_user"
  | "run.failed"
  | "run.cancelled"
  | "heartbeat"
```

这里不用做过细的“delta token”事件。  
第一版的设计目标是可靠，不是炫技。

## 19.4 每类事件的前端应用逻辑

### `conversation.updated`

更新侧栏会话标题、更新时间、预览文本、状态。

### `message.created`

把正式消息插入会话消息列表。  
如果之前存在本地临时消息，需要根据客户端生成的 `client_message_id` 或内容哈希进行替换，避免重复渲染两条。

### `run.created`

创建或更新 `RunRecord`。  
同时确保对应 assistant message 已存在。  
若不存在，先插入一个最小 assistant message 占位。

### `run.status.changed`

只更新 `run.status` 和 `updatedAt`。  
若状态变成 `completed/failed/cancelled` 且没有最终回答，则保留已有块并显示错误或完成提示。

### `run.live.ready`

把 `liveReady` 改为 `true`，记录 `liveSessionId`。  
若当前 run 还没有 `finalAnswer`，并且用户没有手动切到历史模式，则把 `screenMode` 设为 `live`。

### `run.phase_text.set`

覆盖 `phaseText`。  
这里采用覆盖，不做拼接。因为阶段回答是完整的一段，不是 token 流。

### `run.bash.started`

创建 bash block，状态为 `running`。  
若当前已经存在同 ID 的 block，则更新为运行中。

### `run.bash.updated`

更新 stdout、stderr、duration。  
输出区域是否展开由 UI store 决定，不受事件影响。

### `run.bash.completed`

更新状态为 `success` 或 `error`，写入退出码和结束时间。

### `run.progress.created`

插入新的进展项。  
若当前 run 处于 `live` 模式，不改变它。  
若当前 run 处于 `history` 模式且用户没有手动锁定到旧进展，仍可自动跳到最新截图；  
但为了交互稳定，建议只在用户没有明确选中过历史进展时自动切换。实现上可以用 `selectedProgressId` 是否为空判断。

### `run.screenshot.created`

插入新的截图记录。  
如果该截图绑定了新的 progress，前端下一轮渲染时自然能显示出来。  
若当前 run 还没有 `selectedScreenshotId`，就把它设成最新截图。

### `run.answer.final`

这是状态切换的分水岭。  
收到后立即：

设置 `finalAnswer`。  
设置 `screenMode = hidden`。  
清除 `selectedProgressId` 和 `selectedScreenshotId`。  
保留原数据，但不再显示 phase/bash/progress/screen。  
之后滚动消息区到底部。

### `run.waiting_user`

把 `status` 改为 `waiting_user`。  
设置 `needsUserAction = true`。  
设置 `screenMode = hidden`。  
这一事件通常应和 `run.answer.final` 同时或先后到达。  
前端必须允许它们任意顺序到达。实现上采用幂等合并，不依赖先后顺序。

### `run.failed`

状态改为 `failed`，写入 `errorMessage`。  
若还没有 `finalAnswer`，界面显示一个错误回答块，例如“任务执行失败：xxx”。

### `run.cancelled`

状态改为 `cancelled`。  
若最后一条 run 处于当前会话且还在显示实时屏幕，则切成隐藏，并提示“已停止”。

## 20. SSE 断线重连与补偿机制

这是可靠性要求，不是可选项。

## 20.1 重连策略

浏览器 SSE 断开后，前端要自动重连。  
第一次重连等待 `1s`，第二次 `2s`，第三次 `4s`，上限 `15s`。  
当前会话切换时，旧连接必须立即关闭，不参与重连。

重连成功后，不能假设本地状态一定正确，必须做一次补偿同步。

## 20.2 补偿同步

重连成功后，立即调用：

`GET /api/conversations/:id/runs?since=<last_known_update_time>`  
或者更简单，第一版直接重新拉整个会话的 runs 列表。

若发现最后一个 run 的 `updatedAt` 晚于本地，则用服务端数据覆盖本地。  
覆盖原则不是清空重建，而是按主键 upsert，避免界面闪烁。

## 20.3 seq 跳号处理

每个 run 维护一个 `lastSeq`。  
当收到的事件序号不是 `lastSeq + 1` 时：

先把该事件放入短暂缓冲区。  
立即请求 `GET /api/runs/:id` 全量补拉。  
补拉成功后，以服务端快照覆盖该 run 相关所有实体。  
再清空缓冲区。

第一版也可以更简单：一旦跳号，直接忽略当前事件并全量补拉该 run。  
不要尝试自己在前端做复杂事件重排。

## 21. 前端 HTTP 请求封装规范

`frontend/lib/api.ts` 必须封装统一请求层，不允许组件直接 `fetch("/api/...")`。

推荐封装：

```ts
export async function apiFetch<T>(
  path: string,
  init?: RequestInit
): Promise<T> {
  const res = await fetch(path, {
    ...init,
    headers: {
      ...(init?.headers ?? {}),
    },
    credentials: "same-origin",
  })

  if (!res.ok) {
    const text = await res.text().catch(() => "")
    throw new Error(text || `HTTP ${res.status}`)
  }

  return res.json() as Promise<T>
}
```

对于 `multipart/form-data` 请求，不要手动设置 `Content-Type`，交给浏览器。  
错误统一交给上层 hook 捕获，再用 `sonner` 提示。

## 22. 前端业务 Hook 设计

组件不要直接拼装复杂请求逻辑，必须抽成 hook。

推荐定义下面这些 hook：

`useConversations()`  
负责会话列表加载、选择会话、URL 同步。

`useConversationDetail(conversationId)`

`useConversationStream(conversationId)`  
负责 SSE 建连、断线重连、事件分发。

`useSendMessage(conversationId)`

负责实时桌面连接地址、连接状态、全屏接管逻辑。

## 23. 前端交互细节要求

## 23.1 自动滚动策略

消息区使用“聪明跟底部”逻辑，不要粗暴每次新消息都强制滚到底。  
规则如下：

用户当前在底部附近时，新增消息或事件更新后自动滚到底。  
用户已经手动向上滚动查看历史时，不要打断。  
此时右下角显示一个“回到底部”按钮。  
最终回答出现时，无论当前是否在底部，都建议轻微提示，但不强制抢滚动。

可以借用 prompt-kit `ChatContainer` 的思路，但本项目建议自行封装，以便兼容固定底部输入区。

## 23.2 历史进展和实时画面的切换

默认 run 开始后，屏幕窗口优先显示实时模式。  
用户点击某个进展后，切到历史模式。  
此时若后端又推送了新进展，不应强制把用户拉回实时模式。  
只有用户点击底部“直播/实时”节点，才重新回到实时模式。

所以前端必须区分两种行为：

系统自动更新最新数据。  
用户主动锁定历史画面。

推荐在 run 上增加一个临时前端字段：

```ts
isHistoryPinnedByUser?: boolean
```

但第一版为了简化，也可以通过 `screenMode === "history"` 且 `selectedProgressId` 非空来推断。

## 23.3 停止按钮行为

底部 Composer 在会话存在运行中任务时，右下角按钮从发送图标切换为停止图标。  
点击停止后：

按钮立即进入 loading。  
调用 `POST /api/runs/:id/cancel`。  
收到成功后，等 SSE 把 run 状态改成 `cancelled`。  
若 5 秒内没收到 SSE 更新，主动补拉该 run。

## 23.4 “我已经操作”按钮行为

用户点这个按钮时，前端不要调用特殊接口。  
统一走“发送一条用户消息”的普通消息接口，内容固定写死为 `我已经操作`。  
这样整个会话历史是完整的，也利于模型继续理解上下文。

## 25. 前端必须避免的实现错误

这一段是强约束。

不要把阶段文本、bash、进展、屏幕窗口分别建成多条 assistant 消息。  
正确做法是：它们都是同一个 assistant run 消息里的不同块。

不要把当前屏幕截图 URL 直接写死在组件本地 state 里。  
必须从 run 的 `selectedScreenshotId` 推导。否则进展点击联动会乱。

不要把实时 noVNC 画布作为模型输入图来源。  
模型输入图只能来自后端主动采集并保存的 screenshot 记录。

不要在前端推断 run 是否完成，必须以后端事件和接口返回为准。  
只要 `finalAnswer` 或 `status=completed/failed/cancelled/waiting_user` 没明确到达，就视为未结束。

不要把 SSE 事件直接写成“收到什么就 setState 什么”的散乱逻辑。  
必须统一经过事件分发器，按 event type 更新对应 store。

下一部分进入后端工程目录、配置项、数据库表结构与文件存储规范。回复“继续”。

## 27. Go 模块职责划分

为了让程序员直接开工，这里把每个核心文件该做什么写死。

## 27.1 `cmd/easyagentcu/main.go`

职责只有四个：

加载配置。  
初始化日志。  
初始化数据库并执行迁移。  
创建 app 并启动 HTTP 服务。

它不包含任何业务逻辑，不直接操作 SQLite，不直接注册每一条路由。

## 27.2 `internal/app/app.go`

负责组装整个应用依赖，包括：

配置对象。  
日志实例。  
数据库连接。  
各 service。  
AgentManager。  
HTTP router。  
静态文件服务。

它是依赖注入入口，但不使用复杂 DI 框架，直接手写构造函数。

## 27.3 `internal/config/config.go`

负责读取环境变量并做默认值填充、类型转换、合法性校验。  
配置一旦加载完成，必须变成只读结构体，后续全局传递。

## 27.4 `internal/db/*`

`db.go` 打开 SQLite，设置 pragma。  
`migrate.go` 执行数据库初始化和版本迁移。  
`queries/*.go` 每个文件只负责一个聚合根或一组相关表。

所有 query 层方法都要接受 `context.Context`。  
所有写操作都必须允许接收事务对象。

## 27.5 `internal/service/*`

这是 API 视角的服务层。比如：

创建会话。  
发消息。  
列出消息。  
列出 run。  
取消 run。  
流式事件订阅。

这里负责事务边界和数据拼装，但不直接执行模型主循环。  
例如 `message_service.go` 在收到用户消息后，会：

写入 user message。  
写入 assistant message 壳。  
写入 run。  
提交事务。  
异步通知 AgentManager 启动 run。

## 27.6 `internal/agent/*`

这是整个系统的心脏。

`manager.go` 负责每个会话只允许一个运行中的任务，维护运行 goroutine 生命周期。  
`runner.go` 负责单个 run 的执行循环。  
`prompt.go` 负责系统提示词与开发者提示词模板。  
`context_builder.go` 负责构造 messages 和图片输入。  
`model_client.go` 负责调用 OpenAI 兼容接口。  
`tool_registry.go` 定义模型可见工具 schema。  
`tool_executor.go` 负责按 tool call 执行具体工具并写回结果。  
`event_emitter.go` 负责把运行事件写入数据库并广播给 SSE 订阅者。  
`snapshot_builder.go` 负责截图、鼠标标注、图片缩放、图片 base64 载荷准备。

## 27.7 `internal/tools/*`

每个目录负责一类工具，不要交叉。

`tools/desktop` 管鼠标键盘截图窗口焦点。  
`tools/bash` 管 bash 调用。  
`tools/liveview` 管 noVNC、x11vnc、websockify 的检查与连接地址。  
`tools/files` 管附件和截图的磁盘写入、读取、删除。

## 29. 数据库选型与连接策略

数据库固定为 SQLite。  
原因不是“简单”，而是这个项目部署在单台虚拟机、单用户、本地文件存储环境里，SQLite 最匹配。

## 29.1 SQLite 连接初始化

打开数据库后立即执行：

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
PRAGMA temp_store = MEMORY;
```

Go 层建议使用单个 `*sql.DB`，但限制连接池：

`SetMaxOpenConns(1)`  
`SetMaxIdleConns(1)`  
`SetConnMaxLifetime(0)`

这样最稳，避免 SQLite 在多连接下出现不必要锁争用。  
因为本项目吞吐不高，性能完全足够。

## 29.2 迁移策略

第一版不要引入复杂迁移框架。  
直接在 `migrations/001_init.sql` 保存完整建表 SQL，程序启动时检查 `schema_migrations` 表，没有则执行初始化。

以后新增版本时再加：

`002_xxx.sql`  
`003_xxx.sql`

`schema_migrations` 结构固定：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 30. 数据库表结构设计

下面是第一版必须落地的表。

## 30.1 conversations

一个会话就是左侧栏的一条聊天记录。

```sql
CREATE TABLE IF NOT EXISTS conversations (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'idle',
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  archived INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX IF NOT EXISTS idx_conversations_updated_at
ON conversations(updated_at DESC);
```

字段规则：

`id` 使用 UUID。  
`title` 初始可取第一条用户消息前 30 个字符，后续允许后台自动更新。  
`status` 只允许 `idle/running/waiting_user`。  
`archived` 第一版不暴露删除接口时也保留，便于后续软删除。

## 30.2 messages

消息表统一存用户消息和助手消息壳。

```sql
CREATE TABLE IF NOT EXISTS messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL,
  text_content TEXT NOT NULL DEFAULT '',
  run_id TEXT,
  client_message_id TEXT,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation_created
ON messages(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_messages_run_id
ON messages(run_id);
```

字段规则：

`role` 只允许 `user` 或 `assistant`。  
用户消息的 `run_id` 为空。  
助手消息的 `run_id` 必填，并与 runs 表一对一。  
`client_message_id` 用于前端临时消息去重。第一版建议支持。

不要把阶段文本、最终回答塞在 `messages.text_content` 里。  
`messages.text_content` 对用户消息有意义，对助手消息壳可为空。  
真正的阶段文本、最终回答放在 `runs` 表。

## 30.3 attachments

附件既支持用户上传图片，也支持普通文件。

```sql
CREATE TABLE IF NOT EXISTS attachments (
  id TEXT PRIMARY KEY,
  message_id TEXT NOT NULL,
  kind TEXT NOT NULL,
  file_name TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  file_size INTEGER NOT NULL,
  storage_path TEXT NOT NULL,
  thumbnail_path TEXT,
  image_width INTEGER,
  image_height INTEGER,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_attachments_message_id
ON attachments(message_id);
```

字段规则：

`kind` 只允许 `image/file`。  
`storage_path` 存相对 `DATA_DIR` 的相对路径，不存绝对路径。  
图片附件若生成缩略图，则写 `thumbnail_path`。

## 30.4 runs

每次用户发消息触发一个 run。  
一个 run 对应一个 assistant message 壳。

```sql
CREATE TABLE IF NOT EXISTS runs (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  assistant_message_id TEXT NOT NULL,
  status TEXT NOT NULL,
  phase_text TEXT NOT NULL DEFAULT '',
  final_answer TEXT NOT NULL DEFAULT '',
  needs_user_action INTEGER NOT NULL DEFAULT 0,
  user_action_label TEXT,
  live_ready INTEGER NOT NULL DEFAULT 0,
  live_session_id TEXT,
  error_message TEXT,
  seq_counter INTEGER NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  started_at DATETIME,
  finished_at DATETIME,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (assistant_message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_runs_assistant_message_id
ON runs(assistant_message_id);

CREATE INDEX IF NOT EXISTS idx_runs_conversation_created
ON runs(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_runs_status
ON runs(status);
```

字段规则：

`status` 只允许 `queued/running/waiting_user/completed/failed/cancelled`。  
`phase_text` 存当前最新阶段文本，不存历史版本。  
`final_answer` 存最终回答正文。  
`needs_user_action` 与 `status=waiting_user` 可以一起出现，但逻辑上不强绑定，方便容错。  
`seq_counter` 用于生成该 run 的 SSE 事件序号。

## 30.5 bash_blocks

一个 run 中可能多次调用 bash。

```sql
CREATE TABLE IF NOT EXISTS bash_blocks (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  status TEXT NOT NULL,
  command_text TEXT NOT NULL,
  stdout_text TEXT NOT NULL DEFAULT '',
  stderr_text TEXT NOT NULL DEFAULT '',
  exit_code INTEGER,
  started_at DATETIME,
  finished_at DATETIME,
  duration_ms INTEGER,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_bash_blocks_run_created
ON bash_blocks(run_id, created_at ASC);
```

字段规则：

`status` 只允许 `pending/running/success/error`。  
stdout 和 stderr 都允许截断保存，但数据库中应保留截断后的完整展示内容，不要只存摘要。

## 30.6 screenshots

这是最关键的表。  
每一张保存下来的截图都在这里登记。

```sql
CREATE TABLE IF NOT EXISTS screenshots (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  progress_id TEXT,
  raw_path TEXT NOT NULL,
  model_path TEXT NOT NULL,
  annotated_path TEXT NOT NULL,
  desktop_width INTEGER NOT NULL,
  desktop_height INTEGER NOT NULL,
  model_image_width INTEGER NOT NULL,
  model_image_height INTEGER NOT NULL,
  mouse_x INTEGER,
  mouse_y INTEGER,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_screenshots_run_created
ON screenshots(run_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_screenshots_progress_id
ON screenshots(progress_id);
```

注意这里 `progress_id` 可以为空。  
原因是有些截图是纯中间检查帧，还未来得及生成进展文案。  
但一旦某个截图对应一个明确进展，后续应该回填其 `progress_id`。

## 30.7 progress_items

每条进展是一段人类可读文本，绑定某张截图。

```sql
CREATE TABLE IF NOT EXISTS progress_items (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  idx INTEGER NOT NULL,
  text_content TEXT NOT NULL,
  screenshot_id TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE,
  FOREIGN KEY (screenshot_id) REFERENCES screenshots(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_progress_items_run_idx
ON progress_items(run_id, idx ASC);

CREATE UNIQUE INDEX IF NOT EXISTS idx_progress_items_run_idx_unique
ON progress_items(run_id, idx);
```

字段规则：

`idx` 从 1 开始递增。  
一条进展必须绑定一张截图。  
这是前端时间线和时间条的主数据来源。

## 30.8 run_events

为了支持 SSE 重连补偿、调试、问题排查，事件必须持久化。  
不要只在内存广播。

```sql
CREATE TABLE IF NOT EXISTS run_events (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  run_id TEXT NOT NULL,
  seq INTEGER NOT NULL,
  event_type TEXT NOT NULL,
  payload_json TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_run_events_conversation_created
ON run_events(conversation_id, created_at ASC);

CREATE UNIQUE INDEX IF NOT EXISTS idx_run_events_run_seq_unique
ON run_events(run_id, seq);
```

这张表非常有价值：

前端断线后可以补拉。  
后端问题可重放。  
线上调试时能知道某一步到底发过什么事件。

数据库只存元数据。

## 31.1 数据根目录结构

`DATA_DIR` 下固定采用下面结构：

```text
data/
  app.db
  tmp/
  attachments/
    2026/
      03/
        msg_xxx/
          att_xxx_original.png
          att_xxx_thumb.png
  runs/
    conv_xxx/
      run_xxx/
        screenshots/
          shot_001_raw.png

          shot_001_annotated.png
          shot_002_raw.png
          ...
        logs/
          bash_block_xxx_stdout.txt
          bash_block_xxx_stderr.txt
```

推荐仍以数据库文本字段保存相对路径，例如：

`attachments/2026/03/msg_xxx/att_xxx_original.png`  
`runs/conv_xxx/run_xxx/screenshots/shot_001_annotated.png`

不要把文件都堆在一个目录里。  
消息多了以后目录会很难维护。

## 31.2 附件保存规则

用户上传附件后，服务端保存原文件。  
若是图片，再额外生成缩略图。

命名规则固定：

原文件：`<attachment_id>_original.<ext>`  
缩略图：`<attachment_id>_thumb.webp`

缩略图宽度建议 `480px`，等比缩放。  
图片原图不做二次压缩覆盖，避免用户上传的标注图失真。

## 31.3 截图保存规则

每次截图生成一个 screenshot_id，例如 `shot_xxx`。  
然后固定生成三个文件：

`<screenshot_id>_raw.png`  
`<screenshot_id>_model.png`  
`<screenshot_id>_annotated.png`

文件必须写入 run 专属目录：

`runs/<conversation_id>/<run_id>/screenshots/`

任何地方都不要直接覆盖旧截图。  
截图是时间点快照，必须不可变。

## 31.4 临时目录用途

`TEMP_DIR` 只用于：

上传时的临时 multipart 文件。  
截图处理中间产物。  
模型 base64 临时缓存。  
bash 临时重定向文件。

任务结束后尽量清掉。  
但真正的业务附件和截图一旦入库，必须移动到正式目录。

## 31.5 静态访问 URL 规则

后端需要通过 HTTP 把附件和截图暴露给前端。  
推荐统一挂在：

`/files/*`

例如：

`/files/attachments/2026/03/msg_xxx/att_xxx_original.png`  
`/files/runs/conv_xxx/run_xxx/screenshots/shot_001_annotated.png`

数据库中存的是相对路径。  
API DTO 中返回的是前端可直接访问的 URL。  
URL 拼装规则在 DTO 层处理，不要在数据库层处理。

## 31.6 文件删除策略

第一版不做前台删除会话功能时，可以不实现物理删除。  
但代码层面要预留一个后台清理函数：

删除 conversation 时，先删数据库记录。  
再递归删除 `attachments` 和 `runs/<conversation_id>/` 目录。  
失败时记录错误日志，不影响数据库删除事务提交。

## 32. API 总体设计

## 32.1 基本原则

第一版后端 API 统一采用下面几类协议：

普通读接口使用 `application/json`。  
写接口使用 `application/json` 或 `multipart/form-data`。  
事件推送使用 `text/event-stream`。  
文件访问使用普通静态文件 HTTP。  
实时桌面 WebSocket 由 Go 反向代理到 `websockify`，前端同源访问。

整套 API 不做鉴权。  
整套 API 只服务单用户、单实例。  
整套 API 统一挂在 `/api` 前缀下。  
文件静态访问统一挂在 `/files`。  
noVNC 静态资源统一挂在 `/novnc`。  
VNC WebSocket 统一挂在 `/api/live/ws`。

## 32.2 有状态与无状态划分

从 HTTP 协议角度看，所有请求都是无状态的。  
从产品行为角度看，接口分三类：

第一类是“纯查询接口”，例如列会话、列消息、列 run、健康检查。这类接口不改变系统状态。

第二类是“状态变更接口”，例如创建会话、发送消息、取消 run。这类接口会写数据库、触发异步任务、改变当前会话状态。

第三类是“长连接接口”，也就是 SSE 事件流和 VNC WebSocket。它们本身不保存浏览器 Session，但连接建立后会持续订阅当前系统状态变化，所以逻辑上属于弱状态连接。

程序员实现时不要误解成“因为没有登录就没有状态”。  
本项目最大的状态就是会话、run、桌面当前画面和 Agent 当前执行位置。

## 32.3 统一返回格式

普通 JSON 接口统一返回：

成功：

```json
{
  "requestId": "req_01JXXXX",
  "data": {}
}
```

失败：

```json
{
  "requestId": "req_01JXXXX",
  "error": {
    "code": "invalid_argument",
    "message": "text or files is required",
    "details": {}
  }
}
```

不要混用有的接口直接返回对象、有的接口返回包裹对象。  
所有 JSON 接口都统一包一层 `requestId + data/error`。

## 32.4 请求 ID

每个 HTTP 请求进入后端时都生成一个 `requestId`，写入：

响应 JSON。  
日志。  
错误日志。  
模型调用日志上下文。

SSE 事件不复用 HTTP `requestId`，SSE 自己使用 `eventId`。

## 33. 错误码规范

错误码固定如下，前后端都按这个集合处理。

| HTTP 状态码 | error.code               | 场景                                     |
| ----------- | ------------------------ | ---------------------------------------- |
| 400         | `invalid_argument`       | 参数缺失、格式错误、非法状态切换         |
| 404         | `not_found`              | 会话、消息、run、文件不存在              |
| 409         | `conflict`               | 会话已有运行中任务，不能并发发送         |
| 409         | `run_busy`               | 指定 run 正在执行中且不允许新操作        |
| 413         | `payload_too_large`      | 上传文件过大、图片超过限制               |
| 415         | `unsupported_media_type` | 不支持的文件类型                         |
| 422         | `desktop_unavailable`    | X11、VNC、截图命令不可用                 |
| 424         | `upstream_model_error`   | 模型接口失败或返回异常                   |
| 499         | `client_cancelled`       | 客户端主动取消连接，可只记日志不对外暴露 |
| 500         | `internal_error`         | 未分类服务端错误                         |

错误 `message` 给前端看，要写成人能读懂的话，不要把 Go 错误堆栈直接暴露出去。  
详细栈信息只写日志。

处理逻辑必须按下面顺序实现：

第一步，校验 conversation 存在。  
第二步，检查该会话是否存在 `queued/running/waiting_user` 的 run。只要存在就返回 409。第一版不允许并发。  
第三步，若请求携带 `clientMessageId`，先查 `(conversation_id, client_message_id)` 是否已存在。若存在，直接返回那次已创建的 `userMessage + assistantMessage + run`，不要重复创建。  
第四步，在一个事务里完成：
创建用户消息。  
保存附件元数据。  
保存附件文件到磁盘。  
创建助手消息壳。  
创建 run，初始状态 `queued`。  
更新 conversation 的 `status=running`、`updated_at=now`。  
若会话标题还是默认“新聊天”或空标题，用本次用户文本前 30 个字符更新标题。  
第五步，事务提交。  
第六步，异步触发 AgentManager 启动该 run。  
第七步，立即返回 JSON，不等待模型完成。

这里有一个落地细节：  
文件保存要么在事务前完成、失败则不入库；要么先进临时目录，事务成功后再 move 到正式目录。推荐第二种，最稳。

## 36.9 `GET /api/runs/:runId`

用途是单个 run 补拉。  
返回结构和 `GET /api/conversations/:conversationId/runs` 中的单项完全一致。

处理逻辑：

不存在返回 404。  
存在则返回完整 run 聚合。

## 36.10 `POST /api/runs/:runId/cancel`

请求体为空即可。

成功响应：

```json
{
  "requestId": "req_xxx",
  "data": {
    "runId": "run_xxx",
    "status": "cancelling"
  }
}
```

处理逻辑：

第一步，查 run 是否存在。  
第二步，若状态已经是 `completed/failed/cancelled`，直接返回当前状态，幂等成功。  
第三步，若状态是 `queued/running/waiting_user`，通知 AgentManager 取消该 run 的 context。  
第四步，不要在 handler 里直接改数据库为 `cancelled`。真正状态落库由 runner 在收到取消信号后统一处理并发出 SSE。  
第五步，立即返回 `cancelling`。

这样可以避免 handler 和 runner 同时写状态导致竞态。

## 36.11 `GET /api/conversations/:conversationId/events/stream`

这是当前会话的 SSE 事件流。

响应头固定：

```http
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no
```

服务端推送格式固定为：

```text
id: evt_01JXXXX
event: message
data: {"eventId":"evt_01JXXXX","conversationId":"conv_xxx","runId":"run_xxx","seq":12,"type":"run.progress.created","data":{"progressItem":{...}}}

```

前端只监听默认 `message` 事件即可。  
真正业务分发看 `data.type`。

处理逻辑：

建立连接后，立即发一条 `heartbeat`。  
之后把该会话的事件订阅注册到内存广播器。  
每 15 秒发一次心跳，防止代理层断开。  
客户端断开连接时，及时注销订阅。  
这个接口不主动回放历史事件。断线补偿走 runs 全量补拉。

## 38. 处理逻辑的后端实现约束

## 38.1 handler 层不直接返回数据库模型

每个 handler 都必须走 DTO 转换。  
数据库里是 `storage_path`，前端要的是 `url`。  
数据库里是 `text_content`，前端要的是 `text`。  
不要把内部字段原样透给前端。

## 38.2 发消息接口的事务边界

`POST /messages` 必须保证下面三件事要么一起成功，要么一起失败：

用户消息入库。  
助手消息壳入库。  
run 入库。

附件文件若已从临时目录移到正式目录，但事务失败，必须补偿删除。  
不要留下孤儿文件。

## 38.3 SSE 与数据库写入顺序

任何会话事件都必须先写数据库，再发内存广播。  
顺序不能反。

正确顺序是：

更新业务表。  
插入 `run_events`。  
提交事务。  
事务成功后广播给在线 SSE 订阅者。

这样即使广播时进程崩溃，重连后仍能靠数据库状态补齐。  
如果反过来，前端可能看到一个从未真正落库的瞬态事件。

## 38.4 conversation 状态同步规则

后端要在下面节点更新会话状态：

创建新 run 后，conversation 状态改成 `running`。  
run 进入 `waiting_user` 后，conversation 状态改成 `waiting_user`。  
run 进入 `completed/failed/cancelled` 后，若该会话没有别的活动 run，则改成 `idle`。

虽然第一版不允许并发 run，但代码仍应按“检查是否还有其他活动 run”写，便于以后扩展。

## 39.1 `GET /files/*`

用途是前端展示用户附件、截图和缩略图。  
只允许读取 `DATA_DIR` 下白名单目录中的文件。  
必须做路径清洗，防止 `../` 穿越。

推荐逻辑：

`path.Clean` 后确认仍在 `attachments/` 或 `runs/` 下。

必须限制目录前缀。

前端 noVNC 直接连 `/api/live/ws`。

代理要求：

连接建立前检查 `VNC_ENABLED=true`。  
若未启用，直接拒绝。  
若本地 `websockify` 不可达，返回 502。  
代理过程不做消息级解析，只做字节转发。

## 40. 桌面控制与实时画面实现总原则

这一部分决定 Agent 是否真正“能操作电脑”，所以实现上必须保守而稳定。

第一版统一采用 X11 原生命令链，不接入 Playwright、不接入 Selenium、不接入 OCR 框架、不接入第二套浏览器自动化体系。所有桌面操作都通过本机 X11 会话执行，所有实时画面都从当前 X11 桌面采集。

固定技术路径如下：

桌面动作执行使用 `xdotool`。  
屏幕信息读取使用 `xrandr` 和 `xdotool getmouselocation`。  
截图使用 `scrot`。  
截图后处理、缩放、鼠标标注使用 Go 图片库。  
实时桌面画面使用 `x11vnc + websockify + noVNC`。  
Go 服务负责进程守护、健康检查、WebSocket 代理、文件持久化。  
所有外部命令都必须继承同一套 X11 环境变量。

这一版有一个非常重要的工程约束：虽然“每个会话同一时刻只允许一个运行中的 Agent 任务”已经足够满足需求，但由于整个系统只有一个真实桌面，所以后端还必须加一个全局桌面执行锁。也就是不管来自哪个会话，同一时刻只能有一个 run 进入桌面动作执行区。这样可以彻底避免两个任务抢鼠标和键盘。

## 41. X11 运行环境约束

EasyAgentCU 不是在纯服务环境里运行，而是在已经登录进入 Deepin V25 图形桌面的用户会话中运行。只有在这个前提下，`xdotool`、`scrot`、`x11vnc` 才能稳定工作。

Go 服务启动后，所有需要访问 X11 的子进程都必须带上这两个环境变量：

```text
DISPLAY=:0
XAUTHORITY=/home/<user>/.Xauthority
```

其中 `DISPLAY` 来自配置项 `X11_DISPLAY`。  
`XAUTHORITY` 若配置为空，则优先按下面顺序自动推断：

先读取环境变量 `XAUTHORITY`。  
再尝试 `${HOME}/.Xauthority`。  
再尝试 `/run/user/<uid>/gdm/Xauthority` 这类路径不推荐，因为 Deepin 场景不稳定。

程序启动期必须执行一轮桌面自检。自检内容不是只检查命令存在，而是要确认当前显示会话真的可用。推荐顺序如下：

先执行 `xrandr --current`，确认能取到 `current WxH`。  
再执行 `xdotool getmouselocation --shell`，确认能拿到鼠标位置。  
再执行一次 `scrot` 到临时目录，确认截图成功。  
再检查 `x11vnc` 和 `websockify` 命令可执行。  
最后删除自检截图。

只要其中任意一步失败，程序直接启动失败。不要让系统在用户第一次发消息时才发现桌面不可用。

## 42. 外部命令执行封装

所有系统命令都必须走统一封装，不能在业务代码里直接 `exec.Command(...)` 到处写。

推荐定义一个公共执行器：

```go
type CmdRunner interface {
    Run(ctx context.Context, name string, args []string, opts CmdOptions) (CmdResult, error)
}

type CmdOptions struct {
    Env          map[string]string
    Dir          string
    Timeout      time.Duration
    CaptureStdout bool
    CaptureStderr bool
}

type CmdResult struct {
    ExitCode int
    Stdout   string
    Stderr   string
    Duration time.Duration
}
```

这个执行器要负责四件事。

第一，统一注入 X11 环境变量。  
第二，统一记录日志，包括命令名、参数、耗时、退出码。  
第三，支持超时和进程组杀死。  
第四，把常见错误转成可分类错误，例如 `desktop_unavailable`、`command_not_found`、`timeout`。

超时处理不能只 kill 主进程，必须 kill 整个进程组。因为 bash、websockify 这类命令经常会再派生子进程。Go 侧需要给命令加：

```go
cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}
```

超时后执行：

```go
syscall.Kill(-cmd.Process.Pid, syscall.SIGKILL)
```

这样才不会残留僵尸子进程。

## 43. DesktopService 设计

桌面动作不要散落在工具执行器里，必须集中到一个服务里。

推荐接口如下：

```go
type ScreenInfo struct {
    DesktopWidth     int
    DesktopHeight    int
    ModelImageWidth  int
    ModelImageHeight int
    ScaleX           float64
    ScaleY           float64

type MouseLocation struct {
    X int

    Screen int
    Window string
}

type ScreenshotFiles struct {
    RawPath       string
    ModelPath     string
    AnnotatedPath string
}

type ScreenshotBundle struct {
    ID            string
    ScreenInfo    ScreenInfo
    Mouse         MouseLocation
    Files         ScreenshotFiles
    CreatedAt     time.Time
}

type DesktopService interface {
```

## 45. 截图实现

## 45.1 原始截图采集

原始截图一律截整个桌面，不截局部窗口。  
推荐命令：

```bash
scrot /tmp/easyagentcu-shot-xxxx.png
```

执行前先创建唯一临时文件名。  
截图成功后，把临时文件交给 Go 代码继续处理。

这里不要启用 scrot 自带鼠标指针渲染。原因有两个。

第一，不同桌面主题下指针样式和大小不一致。  
第二，需求里明确要求“圈和箭头指向鼠标指针的位置，并写上‘鼠标指针’四个字”，这必须由我们自己的标注层来做。

## 45.2 模型输入图尺寸规则

模型输入图不一定等于原始桌面尺寸。  
缩放规则固定如下：

若原始尺寸宽高都不超过 `MODEL_IMAGE_MAX_WIDTH` 和 `MODEL_IMAGE_MAX_HEIGHT`，则不放大，只保持原尺寸。  
若超出，则按等比缩放，把图压到限制框内。  
推荐默认限制为 `1600x900`。

缩放后记录：

`desktop_width`  
`desktop_height`  
`model_image_width`  
`model_image_height`  
`scale_x = desktop_width / model_image_width`  
`scale_y = desktop_height / model_image_height`

这些值必须和 screenshot 记录一起入库。

## 45.3 文件生成顺序

每次截图生成固定三份文件。

第一份 `raw`。原始整屏 PNG，不做标注，不改尺寸。  
第二份 `model`。给模型发送的那张图，等比缩放。  
第三份 `annotated`。给前端历史进展回放和模型视觉理解辅助使用的标注图。

推荐处理顺序如下：

先落 `raw`。  
再从 `raw` 生成 `model`。  
再从 `model` 生成 `annotated`。  
因为标注图应该和模型看到的坐标空间一致，这样模型看到的箭头、圈、坐标文本与它实际要输出的坐标空间完全一致。

也就是说，箭头和“鼠标指针”标注应画在 `model` 尺寸空间里，而不是画在 `raw` 上再缩放。否则细线条可能因为再次缩放变模糊。

## 45.4 图片编码策略

`raw` 固定保存 PNG。  
`annotated` 固定保存 PNG。  
`model` 默认保存 PNG；若编码后文件过大，导致 base64 预计超过 `MAX_IMAGE_BASE64_MB`，则自动退化为 JPEG。

具体规则如下：

先编码 PNG。  
若文件大小乘以 `4/3` 后超过配置上限，则改用 JPEG 质量 90。  
若仍超限，依次降到 85、80、75。  
降到 75 仍超限，则继续降低模型输入尺寸，例如按 0.9、0.8 比例再次缩小，直到满足上限。  
整个过程必须记录日志。

这样可以同时满足“尽量清晰”和“不能超过模型接口上限”。

## 46. 鼠标标注图实现

这是项目需求中的明确重点，必须具体实现，不能只写在提示词里。

## 46.1 标注元素

每张 `annotated` 图都必须包含三个视觉元素。

第一个元素是一个圆圈，圆心就是当前鼠标坐标。  
第二个元素是一根箭头，箭头指向圆圈。  
第三个元素是文字标签“鼠标指针”。

颜色统一使用高对比的红色系。推荐：

圆圈和箭头主色：`#ff3b30`。  
文字白色描边后填充 `#ff3b30` 或直接白字配深色底。  
线宽随图片尺寸自适应，不能太细。

## 46.2 标注布局算法

不要把箭头和文字永远画在固定位置，否则遇到鼠标靠边时会被裁切。

建议算法如下：

先把鼠标坐标从桌面空间映射到模型图空间。  
再根据鼠标所在象限，选择标签落点方向。  
若鼠标在画面左上区域，标签放右上。  
若鼠标在画面右上区域，标签放左上。  
若鼠标在画面左下区域，标签放右下。  
若鼠标在画面右下区域，标签放左下。  
再做边界裁切修正，确保标签框完全落在图片内。

圆圈半径建议为：

```text
max(18, min(modelWidth, modelHeight) * 0.018)
```

箭头长度建议为圆圈半径的 3 到 4 倍。  
文字标签建议带浅色半透明底，避免压在复杂背景上看不清。

## 46.3 标注内容附加到模型文本上下文

除了给模型图片本身，后端还必须在发图时附带一段说明文字，例如：

```text
这是当前屏幕截图。图中已经用红圈和箭头标出了鼠标指针位置，标签文字为“鼠标指针”。
当前桌面逻辑分辨率为 1920x1080。
发送给你的这张模型截图分辨率为 1600x900。
当前鼠标在桌面坐标系中的位置为 (286, 102)。
你返回的坐标必须基于模型截图 1600x900 的坐标空间。
```

这样就实现了需求中说的“双管齐下”：既提供视觉标注，又提供明确坐标文本。

## 47. 动作执行实现

## 47.1 MoveMouse

命令：

```bash
xdotool mousemove --sync <x> <y>
```

这里的 `<x> <y>` 必须是桌面逻辑坐标，不是模型坐标。  
所以工具执行器在调用 DesktopService 前，必须先完成一次坐标映射。

`--sync` 必须保留，确保命令返回时鼠标已经到位。  
执行完成后默认等待 `80ms` 到 `120ms`，避免桌面渲染还没跟上就继续下一步。

## 47.2 Click

左键、中键、右键分别映射为 `1/2/3`。

命令：

```bash
xdotool click --delay 80 1
```

点击前不自动移动鼠标。  
模型若要点击某坐标，应先单独调用 `move_mouse`，再调用 `click`。这和需求中的鲁棒性设计一致。

## 47.3 DoubleClick

命令：

```bash
xdotool click --repeat 2 --delay 120 1
```

双击前也不自动移动。  
仍然要求模型先 move，再观察确认，再 double_click。

## 47.4 Scroll

滚动不使用鼠标拖拽，而是使用滚轮按钮号。  
上滚是 `4`，下滚是 `5`。

例如向下滚 5 次：

```bash
xdotool click --repeat 5 --delay 80 5
```

前端或模型参数中的 `amount` 统一解释为“滚轮刻度数”，不是像素。  
这样最稳定，也最容易和网页/应用兼容。

## 47.5 TypeText

命令：

```bash
xdotool type --delay 20 --clearmodifiers -- "<text>"
```

必须加 `--clearmodifiers`，否则当系统残留 Ctrl/Shift/Alt 修饰键时，输入会错乱。  
文本参数一定要走 argv 传递，不能拼接 shell，避免转义问题。

对于特别长的文本，建议按 200 到 500 字一段切块输入，每段后等待 `50ms`。  
否则某些应用输入框可能丢字。

## 47.6 KeyPress

快捷键使用：

```bash
xdotool key --clearmodifiers ctrl+l
xdotool key --clearmodifiers alt+Tab
xdotool key --clearmodifiers Return
```

工具层参数建议统一成字符串数组，例如：

```json
["ctrl", "l"]
```

执行时拼成 `ctrl+l`。  
这样 schema 更稳定，也更容易校验。

## 47.7 Drag

实现顺序固定：

先 `mousemove` 到起点。  
再 `mousedown 1`。  
再 `mousemove --sync` 到终点。  
最后 `mouseup 1`。

命令序列：

```bash
xdotool mousemove --sync <fromX> <fromY>
xdotool mousedown 1
xdotool mousemove --sync <toX> <toY>
xdotool mouseup 1
```

每一步之间留出很短的等待，例如 `50ms`。  
不要用一步到位的简化命令，否则某些应用里拖拽识别不稳定。

## 47.8 Wait

等待使用 Go 自己的 `time.Sleep` 即可，不需要调用外部命令。  
支持毫秒级。  
模型参数若给秒数，后端转换为 `time.Duration`。

## 48. 实时画面实现

## 48.1 总体结构

实时画面不依赖截图记录。  
它是一条独立链路：

```text
X11 桌面(:0) -> x11vnc -> websockify -> Go WebSocket 代理 -> 浏览器 noVNC
```

这里的目标不是录屏，而是让用户始终能看到当前真实桌面，并且在需要时可以接管操作。

## 48.2 x11vnc 启动参数

推荐固定命令：

```bash
x11vnc \
  -display :0 \
  -rfbport 5900 \
  -localhost \
  -forever \
  -shared \
  -nopw \
  -xkb \
  -noxdamage \
  -repeat
```

参数含义按实现结果理解即可：

`-localhost` 表示只监听本机。外部浏览器不直接连它。  
`-forever` 表示客户端断开后不退出。  
`-shared` 表示允许多个观察者连接。  
`-nopw` 因为只绑定本机，真正暴露面在 Go 服务层。  
`-noxdamage` 可以减少某些合成器环境下的残影与黑块。  
`-xkb` 和 `-repeat` 有助于键盘事件稳定。

## 48.3 websockify 启动参数

推荐命令：

```bash
websockify 127.0.0.1:6080 127.0.0.1:5900
```

这里只负责把 TCP VNC 转成 WebSocket。  
noVNC 静态文件由 Go 服务或前端包自己提供，不依赖 `websockify --web`。

## 48.4 LiveViewService 设计

推荐接口：

```go
type LiveViewService interface {
    EnsureStarted(ctx context.Context) error
    HealthCheck(ctx context.Context) error
    WebSocketTarget() string
    SessionID() string
}
```

第一版 `SessionID` 固定返回 `desktop-default`。  
因为整个系统只有一个真实桌面，没有每个 run 独立桌面会话。

`EnsureStarted` 的逻辑必须是幂等的。  
也就是多次调用不会重复启动多个 `x11vnc` 或 `websockify` 进程。

推荐做法是：

程序启动时调用一次 `EnsureStarted`。  
每次 run 创建时再调用一次，作为保险。  
内部先探测端口和进程是否健康，健康则直接返回。  
不健康才重启。

## 48.5 进程守护

不要把 `x11vnc` 和 `websockify` 完全交给 systemd 外部守护。  
EasyAgentCU 自己也必须能发现它们异常退出并自动重启。

推荐实现：

Go 内部保存两个受管进程句柄。  
每个进程 `Start()` 后起一个 goroutine `Wait()`。  
一旦异常退出，记录日志，进入指数退避重启。  
退避序列建议 `1s -> 2s -> 4s -> 8s`，上限 `30s`。  
重启成功后恢复健康状态。

这样用户不需要手动修复实时画面链路。

## 48.6 Go WebSocket 代理

为了让浏览器始终访问同源地址 `/api/live/ws`，Go 服务必须提供 WebSocket 代理。  
这里需要新增一个后端依赖：

```text
github.com/gorilla/websocket
```

实现原则是：

浏览器连到 `/api/live/ws`。  
Go 在服务端把这个连接升级为 WebSocket。  
再作为客户端去连 `ws://127.0.0.1:6080/`。  
之后双向拷贝二进制帧和文本帧。  
任一侧断开，则关闭另一侧。

这个代理不需要理解 VNC 协议内容。  
它只是转发。

## 49. 前端实时画面接入细节

前端 `NoVncLiveView` 组件只负责三件事：

获取 `/api/live-view` 配置。  
创建 noVNC `RFB` 实例并连接 `/api/live/ws`。  
向父组件暴露连接状态，例如 `connecting/connected/disconnected/error`。

默认连接参数建议：

关闭本地光标绘制冲突。  
允许剪贴板但第一版不对外暴露。  
全屏接管态下允许发送键盘和鼠标。  
普通嵌入态只读显示也可以允许点击，因为需求里点击实时窗口要进入全屏，实际操作放到全屏再进行。

当连接失败时，屏幕窗口不要整个消失。  
而是显示一个固定大小的深色占位框，中心显示“实时画面连接失败”，并保留重试按钮。  
这样页面结构不会跳动。

## 50. bash 工具实现

## 50.1 工具定位

bash 在本项目中不是附属工具，而是模型的兜底能力。  
当纯鼠标键盘操作低效、页面复杂、系统级任务需要执行时，模型可以直接用 bash。

因为项目运行在用户专门准备的虚拟机里，第一版不对 bash 做权限限制。  
但为了稳定性，仍然必须做下面三件事：

限制单次执行超时。  
限制 stdout/stderr 最大返回量。  
把执行过程完整记录为 bash block，并通过 SSE 推给前端。

## 50.2 BashService 接口

```go
type BashExecResult struct {
    ExitCode   int
    Stdout     string
    Stderr     string
    Duration   time.Duration
    TimedOut   bool
    Truncated  bool
}

type BashService interface {
    Run(ctx context.Context, command string) (*BashExecResult, error)
}
```

## 50.3 执行方式

固定使用：

```bash
/bin/bash -lc "<command>"
```

不要直接 `sh -c`。  
也不要逐词拆成 argv，因为模型给出的就是一整段 bash 命令。

执行时要继承 EasyAgentCU 的环境变量，至少包括：

`DISPLAY`  
`XAUTHORITY`  
`HOME`  
`PATH`

这样模型在 bash 中调用 `xdotool`、`curl`、`python`、`ls` 等命令时环境一致。

工作目录建议设为当前用户的 home 目录。  
因为很多桌面场景下用户文件、下载目录、桌面目录都在 home 下，模型更容易找到资源。

## 50.4 输出截断策略

`stdout + stderr` 合计不能无限增长。  
建议读取时就做流式限长，而不是等执行完再整体读入内存。

推荐上限就是配置项 `BASH_MAX_OUTPUT_KB`。  
超过后：

继续允许进程执行。  
但不再继续累积内存输出。  
最终在截断位置追加：

```text
...[output truncated]
```

同时返回 `Truncated=true`。

## 50.5 超时处理

bash 超时后，必须杀死整个进程组，不允许留后台子进程。  
结果写成：

`ExitCode = -1`  
`TimedOut = true`  
`Stderr` 末尾补一行 `command timed out`

bash block 最终状态记为 `error`。

## 50.6 bash block 和 SSE 的生命周期

bash 一旦开始执行，后端立刻创建 `bash_blocks` 记录，状态是 `running`，并发送 `run.bash.started`。  
执行过程中若输出很长，可以每隔 500ms 到 1000ms 刷新一次数据库和 SSE，发送 `run.bash.updated`。  
执行结束后统一更新状态、退出码、耗时，并发送 `run.bash.completed`。

这样前端能像原型一样看到 bash 调用块逐步出现和变化。

## 51. 截图、实时画面、动作之间的协同顺序

这一段决定执行稳定性，必须固定。

当 run 刚开始时：

先确保 LiveViewService 正常。  
一旦正常，发 `run.live.ready`。  
然后进入模型循环。  
也就是说，用户发消息后界面能先出现实时窗口，再等模型开始分析。

当模型需要新截图时：

先取当前桌面逻辑尺寸。  
再取当前鼠标坐标。  
再截 `raw`。  
再生成 `model` 和 `annotated`。  
再入库并发 `run.screenshot.created`。  
若本次截图会对应一条进展，再创建 `progress_item` 并发 `run.progress.created`。

当模型要求移动鼠标并检查位置时：

执行 move。  
等待短暂稳定时间。  
重新截图。  
模型查看新的带标注图。  
若确认位置正确，再继续 click。  
这个流程在工具和提示词两层都要支持，后面 Agent 部分会把它写死到执行策略里。

## 52. 错误处理与降级策略

## 52.1 截图失败

只要截图命令失败，这一步不能静默跳过。  
因为模型的视觉上下文就断了。

处理规则：

当前工具步骤立即失败。  
记录错误日志。  
run 状态改为 `failed`。  
发送 `run.failed`，错误文案对前端可读，例如“屏幕截图失败，无法继续分析当前桌面”。

不要让模型在没有新截图的情况下盲操作。

## 52.2 实时画面失败

实时画面失败不一定需要让整个 run 失败。  
因为模型操作依赖的是截图，不是 VNC。

处理规则：

若 `x11vnc/websockify` 在 run 开始前不可用，且截图、自检都正常，则 run 仍可继续。  
但前端实时窗口显示“实时画面不可用”。  
同时后端继续后台重启 live view 进程。  
若重启成功，可在后续 run 或本次 run 中再次发 `run.live.ready`。

不过为了符合你给的产品体验，推荐生产部署时把 live view 视为强依赖，在程序启动期就确保可用。

## 52.3 桌面命令偶发失败

比如 `xdotool` 因为窗口短暂切换导致返回非零。

处理规则不要过度激进。  
对于动作命令，可以允许一次轻量重试。  
推荐：

`mousemove/click/type/key/drag/scroll` 失败时，等待 `300ms` 再重试一次。  
第二次仍失败，则工具失败。

但截图和 bash 不建议自动重试太多。  
截图失败一般意味着桌面环境出了问题；bash 失败一般是业务错误，不应盲重试。

这部分虽然不是 API 代码，但必须在实现中默认考虑，否则运行非常不稳定。

系统必须禁用自动锁屏、自动睡眠、自动熄屏。

## 55. Agent 运行架构

这一部分定义后端最核心的执行链路。实现时不要偏离。

整体结构如下：

`MessageService` 负责创建用户消息、助手消息壳和 run。  
`AgentManager` 负责启动、取消、跟踪 run。  
`Runner` 负责单个 run 的模型循环。  
`ModelClient` 负责调用 OpenAI 兼容 Chat Completions API。  
`ToolExecutor` 负责执行模型返回的工具调用。  
`EventEmitter` 负责落库并广播运行事件。  
`ContextBuilder` 负责给模型构造消息上下文。  
`SnapshotBuilder` 负责截图、缩放、标注、base64。

一个 run 从创建到结束，只允许由一个 `Runner` goroutine 持有。  
一个会话同一时刻只允许一个 `queued/running` run。  
整个系统同一时刻只允许一个 run 实际操作桌面。这个全局桌面锁必须落实在 `AgentManager` 或 `DesktopService` 层。

## 56. 对前文的一处修正

前文写过“会话存在 `waiting_user` 的 run 时，发送新消息返回冲突”。这里修正为更可靠的规则：

`queued` 和 `running` 才是阻塞新消息的活动态。  
`waiting_user` 不阻塞。它表示上一轮已经暂停，正在等用户下一条消息。  
因此用户点击“我已经操作”后，前端发送普通用户消息 `我已经操作`，后端应允许创建新的 run。

也就是说，状态语义改为：

`queued/running`：当前 Agent 还在执行，不能并发发新任务。  
`waiting_user`：上一轮已经停下，允许用户继续发消息，新的消息会创建新的 run。  
`completed/failed/cancelled`：结束态。

这个修正必须同步落实到 `POST /api/conversations/:id/messages` 的冲突判断逻辑。

## 57. Run 状态机

run 的后端状态机固定如下：

`queued -> running -> completed`  
`queued -> running -> waiting_user`  
`queued -> running -> failed`  
`queued -> running -> cancelled`  
`queued -> cancelled`

不允许从结束态回到运行态。  
不允许复用旧 run 继续执行。  
用户后续任何补充说明，都是新建一个 run，而不是恢复旧 run。

状态切换时必须同步更新：

`runs.status`  
`runs.updated_at`  
必要时更新 `started_at/finished_at`  
`conversations.status`  
对应的 SSE 事件

## 58. AgentManager 设计

推荐接口：

```go
type AgentManager interface {
    StartRun(ctx context.Context, runID string) error
    CancelRun(ctx context.Context, runID string) error
    IsRunActive(runID string) bool
}
```

内部维护两个层面的并发控制：

第一层是 `activeRuns map[runID]*RunningHandle`，用于取消和避免同一个 run 被重复启动。  
第二层是 `desktopExecMutex`，用于全局桌面互斥。

`RunningHandle` 推荐包含：

```go
type RunningHandle struct {
    RunID      string
    Ctx        context.Context
    Cancel     context.CancelFunc
    StartedAt  time.Time
}
```

`StartRun` 逻辑固定如下：

先查 run 是否存在且状态为 `queued`。  
若已在 `activeRuns` 中，直接返回幂等成功。  
创建 run 级 context。  
注册到 `activeRuns`。  
起 goroutine 执行 `Runner.Run(ctx, runID)`。  
goroutine 退出时，无论成功失败取消，都从 `activeRuns` 删除。

`CancelRun` 逻辑固定如下：

若 run 不在 `activeRuns` 中，返回幂等成功。  
若存在，则调用其 `Cancel()`。  
不要在这里直接写数据库状态为 `cancelled`。真正状态由 runner 收到取消信号后统一落库。

## 59. Runner 的职责边界

`Runner` 只负责单个 run 的完整生命周期。  
它必须是一个可重复测试的纯业务编排器，不直接持有 HTTP、SSE 连接。

推荐结构：

```go
type Runner struct {
    Runs          RunRepository
    Messages      MessageRepository
    Conversations ConversationRepository
    Emitter       EventEmitter
    ModelClient   ModelClient
    Context       ContextBuilder
    Tools         ToolExecutor
    LiveView      LiveViewService
    Snapshot      SnapshotBuilder
    Logger        zerolog.Logger
    Config        config.Config
}
```

`Run(ctx, runID string)` 的职责只有这些：

把 run 从 `queued` 切到 `running`。  
确保实时画面可用，并尽早发 `run.live.ready`。  
抓取初始截图。  
构造第一轮模型上下文。  
循环调用模型，执行工具，补充截图，再继续下一轮。  
在完成、等待用户、失败、取消时做统一收尾。

## 60. 模型交互的关键技术决策

这里做一个重要且必须照做的决定：

第一版不向模型暴露“截图工具”。  
截图由 harness 自动完成，不作为函数工具让模型自己调用。

原因很简单：

OpenAI 兼容 Chat Completions 的 function calling 无法像 Responses API `computer` 工具那样原生在工具输出里直接塞图片。  
如果把截图也设计成工具，实际仍然需要后端额外再补一条带图片的消息给模型。  
既然如此，不如直接由 harness 在 run 开始和每轮工具批执行后自动截图，这样更稳定，也更接近真正的 computer-use loop。

所以这一版的“屏幕观察相关能力”不是显式函数，而是“自动注入当前截图 + 当前鼠标坐标 + 当前分辨率说明”。

## 61. 模型接口约定

固定调用 OpenAI 兼容 Chat Completions API：

```http
POST {MODEL_BASE_URL}/chat/completions
Authorization: Bearer {MODEL_API_KEY}
Content-Type: application/json
```

请求参数固定建议：

```json
{
  "model": "Qwen3.5-397B-A17B",
  "messages": [],
  "tools": [],
  "tool_choice": "auto",
  "parallel_tool_calls": false,
  "temperature": 0.2,
  "max_tokens": 4096,
  "stream": false
}
```

说明如下：

`parallel_tool_calls` 固定为 `false`。因为桌面动作有强顺序性，并发工具调用几乎一定会把桌面搞乱。  
`stream` 固定为 `false`。第一版只做“事件流式 UI”，不做“模型 token 流式输出”。  
`temperature` 保持低值，优先稳定和可复现。  
`max_tokens` 不要太小，否则模型可能在多轮工具调用后输出不完整。

兼容性细节必须处理：

有些厂商返回 `message.content = null`。允许。  
有些厂商返回 `tool_calls[].id` 为空。后端要自己补一个 UUID。  
有些厂商返回 `function.arguments` 前后带代码块标记。后端要先 strip 再解析。  
如果 `arguments` 不是合法 JSON，不直接 fail run，而是生成一个工具错误结果回给模型，让它重试。

## 64. 工具执行返回与错误分类

工具执行不能“出错就立即结束整个 run”。  
必须区分可恢复错误和致命错误。

可恢复错误包括：

工具参数缺失。  
坐标超出当前截图范围。  
窗口标题搜索不到。  
按键名称非法。  
bash 命令返回非零但环境本身正常。

这类错误要包装成工具结果回给模型，例如：

```json
{
  "ok": false,
  "error": {
    "code": "invalid_tool_arguments",
    "message": "x must be within current model image width 1600"
  }
}
```

模型收到后有机会自行修正。

致命错误包括：

截图失败。  
桌面环境失效。  
模型上下文构建失败。  
数据库写事件失败。  
run 被取消。  
全局桌面不可用。

这类错误直接结束 run，状态设为 `failed` 或 `cancelled`。

## 65. ContextBuilder 设计

## 65.1 基本思路

给模型的上下文分成两部分：

第一部分是“会话文本历史”。  
第二部分是“当前 run 的实时工具链上下文”。

不要每一轮都从数据库把整个历史全量重放给模型。  
要做稳定裁剪。

## 65.2 文本历史选取规则

第一版固定规则如下：

从当前会话中，按时间倒序取最近 12 条用户/助手轮次。  
再反转为正序放入模型消息列表。  
其中：

用户消息保留文本；若有图片附件，把图片也作为多模态内容一并放入。  
用户普通文件附件不直接发给模型图片输入，而是把文件名、MIME、可访问本地路径写入文本说明。模型如需读取，可调用 `run_bash`。  
助手历史消息不放阶段文本和进展，只放每个旧 run 的最终回答；若旧 run 是 `waiting_user`，则放它当时对用户的接管说明；若旧 run 是 `failed/cancelled`，可放一条简短系统化说明。

这样做的目标是保留多轮任务延续语义，同时不把旧的视觉步骤、旧截图、旧工具日志塞爆上下文。

## 65.3 当前 run 的实时上下文

当前 run 不走数据库重建，而是在 runner 内存里维护一份 `modelMessages`。

推荐结构：

```go
type ModelMessage struct {
    Role       string
    Content    any
    ToolCalls  []ToolCall
    ToolCallID string
    Name       string
}
```

runner 在启动时先把“系统提示词 + 裁剪后的会话历史”构造成基础消息。  
之后每一轮模型输出、工具结果、系统自动截图消息，都只追加到这份内存消息链上。

第一版不支持进程崩溃后恢复中途 run。  
如果服务意外退出，启动后应把所有 `queued/running` run 标为 `failed`，原因写“service restarted during execution”。这一点放到部署部分再写，但这里的架构前提要明确。

## 66. 自动截图消息格式

每次 run 开始，以及每次工具批执行完成后，runner 会自动把“当前截图”作为一条合成消息追加给模型。  
这条消息角色用 `user`，因为 OpenAI 兼容 Chat Completions 对图片输入最稳定的写法就是 `user content[]`。

但为了避免模型把它误解为新的用户要求，消息文本必须固定写明这是系统自动提供的信息。

模板如下：

```text
[系统自动提供的环境观察，不是新的用户要求]
请基于这张最新屏幕截图继续执行当前任务。
图中已经用红圈、箭头和“鼠标指针”标出了鼠标位置。
当前桌面逻辑分辨率：1920x1080
当前模型截图分辨率：1600x900
当前鼠标桌面坐标：(286, 102)
你返回的所有坐标必须基于 1600x900 这张模型截图的坐标空间。
如果目标很小，请先移动鼠标，再等待下一张截图确认位置，然后再点击。
```

对应消息结构示例：

```json
{
  "role": "user",
  "content": [
    {
      "type": "text",
      "text": "[系统自动提供的环境观察，不是新的用户要求]\n请基于这张最新屏幕截图继续执行当前任务。\n..."
    },
    {
      "type": "image_url",
      "image_url": {
        "url": "data:image/png;base64,...."
      }
    }
  ]
}
```

为兼容更多国产 OpenAI 兼容服务，第一版不传 `detail` 字段。

## 68. Runner 主循环

这是必须按代码实现的核心流程。

推荐伪代码如下：

```go
func (r *Runner) Run(ctx context.Context, runID string) error {
    // 1. 进入 running
    // 2. 确保 live view 可用，必要时发 live.ready
    // 3. 捕获初始截图，入库并发 screenshot.created
    // 4. 构造基础模型消息
    // 5. for step := 1; step <= maxSteps; step++ {
    //      a. 调用模型
    //      b. 若返回 tool_calls:
    //           - 追加 assistant(tool_calls) 到 modelMessages
    //           - 顺序执行每个工具
    //           - 追加每个 tool result 到 modelMessages
    //           - 如果执行过程中 finish_task/request_user_action 已终止，则 return
    //           - 如果本轮执行了桌面动作，或开启了每步截图，则自动截图
    //           - 追加合成截图消息到 modelMessages
    //           - continue
    //      c. 若返回普通 assistant 文本:
    //           - 作为 finish_task 的兜底，写 final_answer 并 completed
    //           - return
    //      d. 若既无工具也无文本:
    //           - 向模型补一条工具错误/格式错误提示，重试一次
    //    }
    // 6. 超过最大步数则 failed
}
```

## 69. 单轮模型调用的实现

## 69.1 正常轮次

每一轮请求都用当前 `modelMessages` 全量调用一次模型。  
因为 Chat Completions 是无状态 API，不能只发增量。

模型返回后取第一条 `choice.message`。  
处理规则：

若有 `tool_calls`，进入工具执行分支。  
若 `tool_calls` 为空但 `content` 有文本，则进入文本兜底分支。  
若两者都没有，视为异常输出。

## 69.2 文本兜底分支

虽然系统提示词要求模型用 `finish_task` 或 `request_user_action`，但兼容模型不一定完全遵守。  
因此必须有兜底逻辑：

若 assistant 返回普通文本且无工具调用，默认把它当作最终回答，等价于 `finish_task(answer=text)`。  
但如果这段文本明显是在要求用户操作，比如包含“请您登录”“请手动完成验证码”“完成后点击我已经操作”，则可作为 `request_user_action` 兜底。  
第一版可以简单保守：普通文本一律当 `finish_task`。真正的接管暂停尽量通过提示词引导模型走结构化工具。

## 69.3 无效输出修复

当模型返回：

无工具调用。  
无普通文本。  
或者工具参数 JSON 解析全部失败。

不要直接 fail run。  
先追加一条内部修复消息再重试一次：

```text
你刚才的输出无效。请只做以下两种事情之一：
1. 返回合法的工具调用
2. 使用 finish_task 或 request_user_action 完成本轮
不要输出空响应。
```

若修复后仍无效，再把 run 标为 `failed`。

## 70. 工具批执行策略

## 70.1 顺序执行

一个 assistant 消息可能返回多个 `tool_calls`。  
后端必须严格按数组顺序执行。  
不得重排。  
不得并发执行。

## 70.2 每轮工具数量上限

为了防止模型一次吐出几十个小动作，把桌面带飞，第一版限制：

单轮最多执行 8 个工具调用。  
超过则只执行前 8 个，并给模型一个工具错误结果：

```json
{
  "ok": false,
  "error": {
    "code": "too_many_tool_calls",
    "message": "at most 8 tool calls are allowed in one turn"
  }
}
```

## 70.3 哪些工具算“桌面动作”

下面这些工具执行后，下一轮前必须自动截图：

`move_mouse`  
`click_mouse`  
`double_click_mouse`  
`scroll_mouse`  
`type_text`  
`press_keys`  
`drag_mouse`  
`activate_window`  
`wait`

`run_bash` 是否自动截图，按配置决定。  
第一版建议统一自动截图，只要 `AGENT_SCREENSHOT_ON_EVERY_STEP=true`。这样最稳定。

`set_phase_text`、`report_progress`、`finish_task`、`request_user_action` 不触发自动截图。

---

## 71. `report_progress` 的调用时机约束

系统提示词里虽然要求模型“看到新截图后描述进展”，但光靠提示词还不够，runner 也要给出稳定环境。

具体做法：

run 开始后的初始截图入库后，不立即自动创建 progress。要让模型自己调用 `report_progress`。  
每次工具批执行后自动截图，也不自动创建 progress。仍由模型在看到新截图后的下一轮自己决定是否 `report_progress`。  
这样进展文本就是真正基于模型视觉理解，而不是工具名拼接出来的假进展。

同时，为了避免 UI 太空，runner 在以下情况下可以插入极短默认阶段文本，但不插入 progress：

run 刚启动，还没从模型拿到任何输出时，可给前端默认显示“正在思考”。  
这只是前端骨架文案，不落库。

---

## 72. `finish_task` 与 `request_user_action` 的后端落地

这两个工具不是普通工具，而是终止型工具。  
一旦执行，就不再继续当前 run 的模型循环。

## 72.1 `finish_task`

执行顺序固定如下：

更新 `runs.final_answer`。  
更新 `runs.status=completed`。  
更新 `runs.finished_at`。  
发送 `run.answer.final`。  
发送 `run.status.changed`。  
更新 `conversations.status=idle`。  
发送 `conversation.updated`。  
return。

注意前端的 UI 规则要求一旦 final answer 出现，就隐藏 phase/bash/progress/screen。  
所以 `run.answer.final` 必须先发，状态更新事件随后发也可以。

## 72.2 `request_user_action`

执行顺序固定如下：

更新 `runs.final_answer`。  
更新 `runs.needs_user_action=true`。  
更新 `runs.user_action_label`，默认“我已经操作”。  
更新 `runs.status=waiting_user`。  
更新 `runs.finished_at`。  
发送 `run.waiting_user`。  
发送 `run.status.changed`。  
更新 `conversations.status=waiting_user`。  
发送 `conversation.updated`。  
return。

这时旧 run 就停住了。  
用户点“我已经操作”后，新建的是下一条用户消息和下一次新的 run。

## 73. 用户接管与安全策略

为了满足需求文档和 OpenAI 的 computer use 安全建议，系统提示词和后端规则都要明确这些停手场景。

模型必须调用 `request_user_action` 的场景：

验证码、滑块、人机验证。  
短信验证码、邮箱验证码、一次性口令。  
支付提交、下单确认、金融转账。  
对外发送邮件、聊天消息、帖子、评论、工单、表单最终提交。  
删除文件、删除数据、清空记录、修改权限。  
安装新软件、执行来源不明脚本、打开浏览器开发者工具粘贴代码。  
密码修改的最终确认。  
明显可疑的钓鱼页面或页面上的恶意提示。

对于登录这件事，第一版建议保守处理：

可以自动打开登录页、定位输入框、甚至填写非敏感用户名。  
但一旦涉及密码、验证码、扫码确认，统一 `request_user_action`。

这样虽然比最激进自动化更保守，但可靠且安全。

## 74. 鼠标点击鲁棒性在 Agent 逻辑中的落地

前面桌面部分已经解决了“如何提供带标注截图”。  
这里还要把“如何强制模型真正使用 move-verify-click 方法”写进逻辑。

## 74.1 提示词层约束

系统提示词已经写死：  
小目标不能把 move 和 click 放在同一轮里。  
必须先 move，等新截图，确认后再 click。

## 74.2 执行层约束

为了进一步稳定，增加一个轻量规则：

若一轮工具调用中同时出现 `move_mouse` 和 `click_mouse`，并且这两个工具之间没有 `wait` 且本轮也没有显式说明是大目标，后端不拦截。  
第一版仍允许，因为硬拦截太复杂。  
但日志要记录“同轮 move+click”。后续可通过评估观察模型是否经常违规。

真正保证准确性的核心仍然是提示词和自动截图机制。

## 74.3 模型上下文中的反馈闭环

每次 move 后，自动截图消息都会告诉模型：

现在鼠标在哪里。  
这张图上鼠标被标出来了。  
坐标空间是多少。

这样模型即使第一次给错坐标，也能根据“当前错误位置”继续二次修正。这正是你需求里想要的双重反馈机制。

## 77. 这一部分的实现结论

到这里，Agent 逻辑已经固定：

使用 OpenAI 兼容 Chat Completions。  
不显式暴露截图工具，而是由 harness 自动提供截图。

服务依赖图形桌面会话。单纯 `After=graphical.target` 不一定保证用户已经登录桌面。  
因此最佳实践是：

让程序由“自动登录进入桌面”的用户会话环境启动，或者  
写成用户级 systemd 服务，在图形登录后启动。

如果部署时能确保虚拟机自动登录桌面，系统级服务也能工作，只要 `.Xauthority` 和 `DISPLAY` 正确。

启用方式：

```bash
sudo systemctl daemon-reload
sudo systemctl enable easyagentcu
sudo systemctl start easyagentcu
```

## 82. 服务重启恢复策略

这是稳定性关键点，必须落地。

程序重启后，数据库里可能残留以下 run 状态：

`queued`  
`running`

这些 run 已经不可能继续，因为内存中的 `modelMessages`、取消上下文、当前轮状态全丢了。  
所以程序启动后必须执行一次恢复修正：

把所有 `queued` 和 `running` 的 run 统一改为 `failed`。  
错误信息写：

`service restarted during execution`

同时把对应 conversation 状态修正为：

如果该会话存在 `waiting_user` run，则 `waiting_user`。  
否则 `idle`。

这个修正不需要发 SSE，因为程序刚启动时没有客户端连接。  
但如果此时已有客户端连接，也可以发 `run.failed` 与 `conversation.updated`，属于增强项。

## 87. 开发优先级建议

虽然文档已经足够完整，但实际开发还是按下面顺序最稳：

先打通后端最小链路：创建会话、发消息、假模型、完成 run。

## 90. `migrations/001_init.sql` 完整稿

下面这份 SQL 可以直接作为首版初始化脚本使用。

```sql
PRAGMA foreign_keys = ON;

CREATE TABLE IF NOT EXISTS schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS conversations (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'idle'
    CHECK (status IN ('idle', 'running', 'waiting_user')),
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  archived INTEGER NOT NULL DEFAULT 0
    CHECK (archived IN (0, 1))
);

CREATE INDEX IF NOT EXISTS idx_conversations_updated_at
ON conversations(updated_at DESC);

CREATE INDEX IF NOT EXISTS idx_conversations_status
ON conversations(status);

CREATE TABLE IF NOT EXISTS messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL
    CHECK (role IN ('user', 'assistant')),
  text_content TEXT NOT NULL DEFAULT '',
  run_id TEXT,
  client_message_id TEXT,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation_created
ON messages(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_messages_run_id
ON messages(run_id);

CREATE UNIQUE INDEX IF NOT EXISTS idx_messages_conversation_client_message_id_unique
ON messages(conversation_id, client_message_id)
WHERE client_message_id IS NOT NULL AND client_message_id <> '';

CREATE TABLE IF NOT EXISTS attachments (
  id TEXT PRIMARY KEY,
  message_id TEXT NOT NULL,
  kind TEXT NOT NULL
    CHECK (kind IN ('image', 'file')),
  file_name TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  file_size INTEGER NOT NULL CHECK (file_size >= 0),
  storage_path TEXT NOT NULL,
  thumbnail_path TEXT,
  image_width INTEGER,
  image_height INTEGER,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_attachments_message_id
ON attachments(message_id);

CREATE TABLE IF NOT EXISTS runs (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  assistant_message_id TEXT NOT NULL,
  status TEXT NOT NULL
    CHECK (status IN ('queued', 'running', 'waiting_user', 'completed', 'failed', 'cancelled')),
  phase_text TEXT NOT NULL DEFAULT '',
  final_answer TEXT NOT NULL DEFAULT '',
  needs_user_action INTEGER NOT NULL DEFAULT 0
    CHECK (needs_user_action IN (0, 1)),
  user_action_label TEXT,
  live_ready INTEGER NOT NULL DEFAULT 0
    CHECK (live_ready IN (0, 1)),
  live_session_id TEXT,
  error_message TEXT,
  seq_counter INTEGER NOT NULL DEFAULT 0 CHECK (seq_counter >= 0),
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  started_at DATETIME,
  finished_at DATETIME,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (assistant_message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_runs_assistant_message_id
ON runs(assistant_message_id);

CREATE INDEX IF NOT EXISTS idx_runs_conversation_created
ON runs(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_runs_status
ON runs(status);

CREATE TABLE IF NOT EXISTS bash_blocks (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  status TEXT NOT NULL
    CHECK (status IN ('pending', 'running', 'success', 'error')),
  command_text TEXT NOT NULL,
  stdout_text TEXT NOT NULL DEFAULT '',
  stderr_text TEXT NOT NULL DEFAULT '',
  exit_code INTEGER,
  started_at DATETIME,
  finished_at DATETIME,
  duration_ms INTEGER,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_bash_blocks_run_created
ON bash_blocks(run_id, created_at ASC);

CREATE TABLE IF NOT EXISTS screenshots (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  progress_id TEXT,
  raw_path TEXT NOT NULL,
  model_path TEXT NOT NULL,
  annotated_path TEXT NOT NULL,
  desktop_width INTEGER NOT NULL CHECK (desktop_width > 0),
  desktop_height INTEGER NOT NULL CHECK (desktop_height > 0),
  model_image_width INTEGER NOT NULL CHECK (model_image_width > 0),
  model_image_height INTEGER NOT NULL CHECK (model_image_height > 0),
```

```go
package db

import (
	"context"
	"database/sql"
	"fmt"

	_ "modernc.org/sqlite"
)

func Open(path string, busyTimeoutMS int) (*sql.DB, error) {
	dsn := fmt.Sprintf("file:%s?_pragma=busy_timeout(%d)", path, busyTimeoutMS)
	db, err := sql.Open("sqlite", dsn)
	if err != nil {
		return nil, err
	}

	db.SetMaxOpenConns(1)
	db.SetMaxIdleConns(1)
	db.SetConnMaxLifetime(0)

	pragmas := []string{
		"PRAGMA journal_mode = WAL;",
		"PRAGMA synchronous = NORMAL;",
		"PRAGMA foreign_keys = ON;",
		"PRAGMA temp_store = MEMORY;",
	}

	ctx := context.Background()
	for _, stmt := range pragmas {
		if _, err := db.ExecContext(ctx, stmt); err != nil {
			_ = db.Close()
			return nil, err
		}
	}

	return db, nil
}
```

### `internal/db/migrate.go`

```go
package db

import (
	"context"
	"database/sql"
	"embed"
	"fmt"
	"sort"
	"strconv"
	"strings"
)

//go:embed ../../migrations/*.sql
var migrationFS embed.FS

func Migrate(ctx context.Context, db *sql.DB) error {
	if _, err := db.ExecContext(ctx, `
		CREATE TABLE IF NOT EXISTS schema_migrations (
			version INTEGER PRIMARY KEY,
			name TEXT NOT NULL,
			applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
		);
	`); err != nil {
		return err
	}

	entries, err := migrationFS.ReadDir("../../migrations")
	if err != nil {
		return err
	}

	type item struct {
		Version int
		Name    string
		Path    string
	}

	var items []item
	for _, e := range entries {
		name := e.Name()
		if !strings.HasSuffix(name, ".sql") {
			continue
		}
		parts := strings.SplitN(name, "_", 2)
		if len(parts) != 2 {
			return fmt.Errorf("invalid migration file name: %s", name)
		}
		v, err := strconv.Atoi(parts[0])
		if err != nil {
			return fmt.Errorf("invalid migration version: %s", name)
		}
		items = append(items, item{
			Version: v,
			Name:    name,
			Path:    "../../migrations/" + name,
		})
	}

	sort.Slice(items, func(i, j int) bool { return items[i].Version < items[j].Version })

	for _, m := range items {
		var count int
		if err := db.QueryRowContext(ctx, `SELECT COUNT(1) FROM schema_migrations WHERE version = ?`, m.Version).Scan(&count); err != nil {
			return err
		}
		if count > 0 {
			continue
		}

		body, err := migrationFS.ReadFile(m.Path)
		if err != nil {
			return err
		}

		tx, err := db.BeginTx(ctx, nil)
		if err != nil {
			return err
		}

		if _, err := tx.ExecContext(ctx, string(body)); err != nil {
			_ = tx.Rollback()
			return err
		}

		if _, err := tx.ExecContext(ctx, `INSERT INTO schema_migrations(version, name) VALUES(?, ?)`, m.Version, m.Name); err != nil {
			_ = tx.Rollback()
			return err
		}

		if err := tx.Commit(); err != nil {
			return err
		}
	}

	return nil
}
```

## 99. 一处实现细节必须按这里处理

`messages.run_id` 和 `screenshots.progress_id` 都不要加外键。  
这是刻意设计，不是漏写。

原因如下：

`messages` 和 `runs` 之间存在“助手消息壳先创建、run 再创建、消息再回填 run_id”的顺序。  
`progress_items` 和 `screenshots` 之间存在“截图先创建、进展再创建、截图再回填 progress_id”的顺序。  
若两边都加 FK，会制造循环依赖，事务实现复杂且容易出错。

所以正确做法是：

保持一边有 FK，另一边仅保留普通关联字段和索引。  
业务层自己维护一致性。
