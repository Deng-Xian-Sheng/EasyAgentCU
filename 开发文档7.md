# EasyAgentCU 开发文档（第 7 部分）

## 55. Agent 运行架构

这一部分定义后端最核心的执行链路。实现时不要偏离。

整体结构如下：

`MessageService` 负责创建用户消息、助手消息壳和 run。  
`AgentManager` 负责启动、取消、跟踪 run。  
`Runner` 负责单个 run 的模型循环。  
`ModelClient` 负责调用 OpenAI 兼容 Chat Completions API。  
`ToolExecutor` 负责执行模型返回的工具调用。  
`EventEmitter` 负责落库并广播运行事件。  
`ContextBuilder` 负责给模型构造消息上下文。  
`SnapshotBuilder` 负责截图、缩放、标注、base64。

一个 run 从创建到结束，只允许由一个 `Runner` goroutine 持有。  
一个会话同一时刻只允许一个 `queued/running` run。  
整个系统同一时刻只允许一个 run 实际操作桌面。这个全局桌面锁必须落实在 `AgentManager` 或 `DesktopService` 层。

## 56. 对前文的一处修正

前文写过“会话存在 `waiting_user` 的 run 时，发送新消息返回冲突”。这里修正为更可靠的规则：

`queued` 和 `running` 才是阻塞新消息的活动态。  
`waiting_user` 不阻塞。它表示上一轮已经暂停，正在等用户下一条消息。  
因此用户点击“我已经操作”后，前端发送普通用户消息 `我已经操作`，后端应允许创建新的 run。

也就是说，状态语义改为：

`queued/running`：当前 Agent 还在执行，不能并发发新任务。  
`waiting_user`：上一轮已经停下，允许用户继续发消息，新的消息会创建新的 run。  
`completed/failed/cancelled`：结束态。

这个修正必须同步落实到 `POST /api/conversations/:id/messages` 的冲突判断逻辑。

## 57. Run 状态机

run 的后端状态机固定如下：

`queued -> running -> completed`  
`queued -> running -> waiting_user`  
`queued -> running -> failed`  
`queued -> running -> cancelled`  
`queued -> cancelled`

不允许从结束态回到运行态。  
不允许复用旧 run 继续执行。  
用户后续任何补充说明，都是新建一个 run，而不是恢复旧 run。

状态切换时必须同步更新：

`runs.status`  
`runs.updated_at`  
必要时更新 `started_at/finished_at`  
`conversations.status`  
对应的 SSE 事件

## 58. AgentManager 设计

推荐接口：

```go
type AgentManager interface {
    StartRun(ctx context.Context, runID string) error
    CancelRun(ctx context.Context, runID string) error
    IsRunActive(runID string) bool
}
```

内部维护两个层面的并发控制：

第一层是 `activeRuns map[runID]*RunningHandle`，用于取消和避免同一个 run 被重复启动。  
第二层是 `desktopExecMutex`，用于全局桌面互斥。

`RunningHandle` 推荐包含：

```go
type RunningHandle struct {
    RunID      string
    Ctx        context.Context
    Cancel     context.CancelFunc
    StartedAt  time.Time
}
```

`StartRun` 逻辑固定如下：

先查 run 是否存在且状态为 `queued`。  
若已在 `activeRuns` 中，直接返回幂等成功。  
创建 run 级 context。  
注册到 `activeRuns`。  
起 goroutine 执行 `Runner.Run(ctx, runID)`。  
goroutine 退出时，无论成功失败取消，都从 `activeRuns` 删除。

`CancelRun` 逻辑固定如下：

若 run 不在 `activeRuns` 中，返回幂等成功。  
若存在，则调用其 `Cancel()`。  
不要在这里直接写数据库状态为 `cancelled`。真正状态由 runner 收到取消信号后统一落库。

## 59. Runner 的职责边界

`Runner` 只负责单个 run 的完整生命周期。  
它必须是一个可重复测试的纯业务编排器，不直接持有 HTTP、SSE 连接。

推荐结构：

```go
type Runner struct {
    Runs          RunRepository
    Messages      MessageRepository
    Conversations ConversationRepository
    Emitter       EventEmitter
    ModelClient   ModelClient
    Context       ContextBuilder
    Tools         ToolExecutor
    LiveView      LiveViewService
    Snapshot      SnapshotBuilder
    Logger        zerolog.Logger
    Config        config.Config
}
```

`Run(ctx, runID string)` 的职责只有这些：

把 run 从 `queued` 切到 `running`。  
确保实时画面可用，并尽早发 `run.live.ready`。  
抓取初始截图。  
构造第一轮模型上下文。  
循环调用模型，执行工具，补充截图，再继续下一轮。  
在完成、等待用户、失败、取消时做统一收尾。

## 60. 模型交互的关键技术决策

这里做一个重要且必须照做的决定：

第一版不向模型暴露“截图工具”。  
截图由 harness 自动完成，不作为函数工具让模型自己调用。

原因很简单：

OpenAI 兼容 Chat Completions 的 function calling 无法像 Responses API `computer` 工具那样原生在工具输出里直接塞图片。  
如果把截图也设计成工具，实际仍然需要后端额外再补一条带图片的消息给模型。  
既然如此，不如直接由 harness 在 run 开始和每轮工具批执行后自动截图，这样更稳定，也更接近真正的 computer-use loop。

所以这一版的“屏幕观察相关能力”不是显式函数，而是“自动注入当前截图 + 当前鼠标坐标 + 当前分辨率说明”。

## 61. 模型接口约定

固定调用 OpenAI 兼容 Chat Completions API：

```http
POST {MODEL_BASE_URL}/chat/completions
Authorization: Bearer {MODEL_API_KEY}
Content-Type: application/json
```

请求参数固定建议：

```json
{
  "model": "Qwen3.5-397B-A17B",
  "messages": [],
  "tools": [],
  "tool_choice": "auto",
  "parallel_tool_calls": false,
  "temperature": 0.2,
  "max_tokens": 4096,
  "stream": false
}
```

说明如下：

`parallel_tool_calls` 固定为 `false`。因为桌面动作有强顺序性，并发工具调用几乎一定会把桌面搞乱。  
`stream` 固定为 `false`。第一版只做“事件流式 UI”，不做“模型 token 流式输出”。  
`temperature` 保持低值，优先稳定和可复现。  
`max_tokens` 不要太小，否则模型可能在多轮工具调用后输出不完整。

兼容性细节必须处理：

有些厂商返回 `message.content = null`。允许。  
有些厂商返回 `tool_calls[].id` 为空。后端要自己补一个 UUID。  
有些厂商返回 `function.arguments` 前后带代码块标记。后端要先 strip 再解析。  
如果 `arguments` 不是合法 JSON，不直接 fail run，而是生成一个工具错误结果回给模型，让它重试。

- 不要写“我在推理”或暴露内部思考。

  - 页面已经加载到热搜榜列表

- request_user_action 的 answer 要直接说明：

  - 用户完成后点击“我已经操作”

## 64. 工具执行返回与错误分类

工具执行不能“出错就立即结束整个 run”。  
必须区分可恢复错误和致命错误。

可恢复错误包括：

工具参数缺失。  
坐标超出当前截图范围。  
窗口标题搜索不到。  
按键名称非法。  
bash 命令返回非零但环境本身正常。

这类错误要包装成工具结果回给模型，例如：

```json
{
  "ok": false,
  "error": {
    "code": "invalid_tool_arguments",
    "message": "x must be within current model image width 1600"
  }
}
```

模型收到后有机会自行修正。

致命错误包括：

截图失败。  
桌面环境失效。  
模型上下文构建失败。  
数据库写事件失败。  
run 被取消。  
全局桌面不可用。

这类错误直接结束 run，状态设为 `failed` 或 `cancelled`。

## 65. ContextBuilder 设计

## 65.1 基本思路

给模型的上下文分成两部分：

第一部分是“会话文本历史”。  
第二部分是“当前 run 的实时工具链上下文”。

不要每一轮都从数据库把整个历史全量重放给模型。  
要做稳定裁剪。

## 65.2 文本历史选取规则

第一版固定规则如下：

从当前会话中，按时间倒序取最近 12 条用户/助手轮次。  
再反转为正序放入模型消息列表。  
其中：

用户消息保留文本；若有图片附件，把图片也作为多模态内容一并放入。  
用户普通文件附件不直接发给模型图片输入，而是把文件名、MIME、可访问本地路径写入文本说明。模型如需读取，可调用 `run_bash`。  
助手历史消息不放阶段文本和进展，只放每个旧 run 的最终回答；若旧 run 是 `waiting_user`，则放它当时对用户的接管说明；若旧 run 是 `failed/cancelled`，可放一条简短系统化说明。

这样做的目标是保留多轮任务延续语义，同时不把旧的视觉步骤、旧截图、旧工具日志塞爆上下文。

## 65.3 当前 run 的实时上下文

当前 run 不走数据库重建，而是在 runner 内存里维护一份 `modelMessages`。

推荐结构：

```go
type ModelMessage struct {
    Role       string
    Content    any
    ToolCalls  []ToolCall
    ToolCallID string
    Name       string
}
```

runner 在启动时先把“系统提示词 + 裁剪后的会话历史”构造成基础消息。  
之后每一轮模型输出、工具结果、系统自动截图消息，都只追加到这份内存消息链上。

第一版不支持进程崩溃后恢复中途 run。  
如果服务意外退出，启动后应把所有 `queued/running` run 标为 `failed`，原因写“service restarted during execution”。这一点放到部署部分再写，但这里的架构前提要明确。

## 66. 自动截图消息格式

每次 run 开始，以及每次工具批执行完成后，runner 会自动把“当前截图”作为一条合成消息追加给模型。  
这条消息角色用 `user`，因为 OpenAI 兼容 Chat Completions 对图片输入最稳定的写法就是 `user content[]`。

但为了避免模型把它误解为新的用户要求，消息文本必须固定写明这是系统自动提供的信息。

模板如下：

```text
[系统自动提供的环境观察，不是新的用户要求]
请基于这张最新屏幕截图继续执行当前任务。
图中已经用红圈、箭头和“鼠标指针”标出了鼠标位置。
当前桌面逻辑分辨率：1920x1080
当前模型截图分辨率：1600x900
当前鼠标桌面坐标：(286, 102)
你返回的所有坐标必须基于 1600x900 这张模型截图的坐标空间。
如果目标很小，请先移动鼠标，再等待下一张截图确认位置，然后再点击。
```

对应消息结构示例：

```json
{
  "role": "user",
  "content": [
    {
      "type": "text",
      "text": "[系统自动提供的环境观察，不是新的用户要求]\n请基于这张最新屏幕截图继续执行当前任务。\n..."
    },
    {
      "type": "image_url",
      "image_url": {
        "url": "data:image/png;base64,...."
      }
    }
  ]
}
```

为兼容更多国产 OpenAI 兼容服务，第一版不传 `detail` 字段。  
如果后续实测某个模型兼容 `detail: "high"` 且效果更好，再做配置开关。

## 68. Runner 主循环

这是必须按代码实现的核心流程。

推荐伪代码如下：

```go
func (r *Runner) Run(ctx context.Context, runID string) error {
    // 1. 进入 running
    // 2. 确保 live view 可用，必要时发 live.ready
    // 3. 捕获初始截图，入库并发 screenshot.created
    // 4. 构造基础模型消息
    // 5. for step := 1; step <= maxSteps; step++ {
    //      a. 调用模型
    //      b. 若返回 tool_calls:
    //           - 追加 assistant(tool_calls) 到 modelMessages
    //           - 顺序执行每个工具
    //           - 追加每个 tool result 到 modelMessages
    //           - 如果执行过程中 finish_task/request_user_action 已终止，则 return
    //           - 如果本轮执行了桌面动作，或开启了每步截图，则自动截图
    //           - 追加合成截图消息到 modelMessages
    //           - continue
    //      c. 若返回普通 assistant 文本:
    //           - 作为 finish_task 的兜底，写 final_answer 并 completed
    //           - return
    //      d. 若既无工具也无文本:
    //           - 向模型补一条工具错误/格式错误提示，重试一次
    //    }
    // 6. 超过最大步数则 failed
}
```

## 69. 单轮模型调用的实现

## 69.1 正常轮次

每一轮请求都用当前 `modelMessages` 全量调用一次模型。  
因为 Chat Completions 是无状态 API，不能只发增量。

模型返回后取第一条 `choice.message`。  
处理规则：

若有 `tool_calls`，进入工具执行分支。  
若 `tool_calls` 为空但 `content` 有文本，则进入文本兜底分支。  
若两者都没有，视为异常输出。

## 69.2 文本兜底分支

虽然系统提示词要求模型用 `finish_task` 或 `request_user_action`，但兼容模型不一定完全遵守。  
因此必须有兜底逻辑：

若 assistant 返回普通文本且无工具调用，默认把它当作最终回答，等价于 `finish_task(answer=text)`。  
但如果这段文本明显是在要求用户操作，比如包含“请您登录”“请手动完成验证码”“完成后点击我已经操作”，则可作为 `request_user_action` 兜底。  
第一版可以简单保守：普通文本一律当 `finish_task`。真正的接管暂停尽量通过提示词引导模型走结构化工具。

## 69.3 无效输出修复

当模型返回：

无工具调用。  
无普通文本。  
或者工具参数 JSON 解析全部失败。

不要直接 fail run。  
先追加一条内部修复消息再重试一次：

```text
你刚才的输出无效。请只做以下两种事情之一：
1. 返回合法的工具调用
2. 使用 finish_task 或 request_user_action 完成本轮
不要输出空响应。
```

若修复后仍无效，再把 run 标为 `failed`。

## 70. 工具批执行策略

## 70.1 顺序执行

一个 assistant 消息可能返回多个 `tool_calls`。  
后端必须严格按数组顺序执行。  
不得重排。  
不得并发执行。

## 70.2 每轮工具数量上限

为了防止模型一次吐出几十个小动作，把桌面带飞，第一版限制：

单轮最多执行 8 个工具调用。  
超过则只执行前 8 个，并给模型一个工具错误结果：

```json
{
  "ok": false,
  "error": {
    "code": "too_many_tool_calls",
    "message": "at most 8 tool calls are allowed in one turn"
  }
}
```

## 70.3 哪些工具算“桌面动作”

下面这些工具执行后，下一轮前必须自动截图：

`move_mouse`  
`click_mouse`  
`double_click_mouse`  
`scroll_mouse`  
`type_text`  
`press_keys`  
`drag_mouse`  
`activate_window`  
`wait`

`run_bash` 是否自动截图，按配置决定。  
第一版建议统一自动截图，只要 `AGENT_SCREENSHOT_ON_EVERY_STEP=true`。这样最稳定。

`set_phase_text`、`report_progress`、`finish_task`、`request_user_action` 不触发自动截图。

---

## 71. `report_progress` 的调用时机约束

系统提示词里虽然要求模型“看到新截图后描述进展”，但光靠提示词还不够，runner 也要给出稳定环境。

具体做法：

run 开始后的初始截图入库后，不立即自动创建 progress。要让模型自己调用 `report_progress`。  
每次工具批执行后自动截图，也不自动创建 progress。仍由模型在看到新截图后的下一轮自己决定是否 `report_progress`。  
这样进展文本就是真正基于模型视觉理解，而不是工具名拼接出来的假进展。

同时，为了避免 UI 太空，runner 在以下情况下可以插入极短默认阶段文本，但不插入 progress：

run 刚启动，还没从模型拿到任何输出时，可给前端默认显示“正在思考”。  
这只是前端骨架文案，不落库。

---

## 72. `finish_task` 与 `request_user_action` 的后端落地

这两个工具不是普通工具，而是终止型工具。  
一旦执行，就不再继续当前 run 的模型循环。

## 72.1 `finish_task`

执行顺序固定如下：

更新 `runs.final_answer`。  
更新 `runs.status=completed`。  
更新 `runs.finished_at`。  
发送 `run.answer.final`。  
发送 `run.status.changed`。  
更新 `conversations.status=idle`。  
发送 `conversation.updated`。  
return。

注意前端的 UI 规则要求一旦 final answer 出现，就隐藏 phase/bash/progress/screen。  
所以 `run.answer.final` 必须先发，状态更新事件随后发也可以。

## 72.2 `request_user_action`

执行顺序固定如下：

更新 `runs.final_answer`。  
更新 `runs.needs_user_action=true`。  
更新 `runs.user_action_label`，默认“我已经操作”。  
更新 `runs.status=waiting_user`。  
更新 `runs.finished_at`。  
发送 `run.waiting_user`。  
发送 `run.status.changed`。  
更新 `conversations.status=waiting_user`。  
发送 `conversation.updated`。  
return。

这时旧 run 就停住了。  
用户点“我已经操作”后，新建的是下一条用户消息和下一次新的 run。

## 73. 用户接管与安全策略

为了满足需求文档和 OpenAI 的 computer use 安全建议，系统提示词和后端规则都要明确这些停手场景。

模型必须调用 `request_user_action` 的场景：

验证码、滑块、人机验证。  
短信验证码、邮箱验证码、一次性口令。  
支付提交、下单确认、金融转账。  
对外发送邮件、聊天消息、帖子、评论、工单、表单最终提交。  
删除文件、删除数据、清空记录、修改权限。  
安装新软件、执行来源不明脚本、打开浏览器开发者工具粘贴代码。  
密码修改的最终确认。  
明显可疑的钓鱼页面或页面上的恶意提示。

对于登录这件事，第一版建议保守处理：

可以自动打开登录页、定位输入框、甚至填写非敏感用户名。  
但一旦涉及密码、验证码、扫码确认，统一 `request_user_action`。

这样虽然比最激进自动化更保守，但可靠且安全。

## 74. 鼠标点击鲁棒性在 Agent 逻辑中的落地

前面桌面部分已经解决了“如何提供带标注截图”。  
这里还要把“如何强制模型真正使用 move-verify-click 方法”写进逻辑。

## 74.1 提示词层约束

系统提示词已经写死：  
小目标不能把 move 和 click 放在同一轮里。  
必须先 move，等新截图，确认后再 click。

## 74.2 执行层约束

为了进一步稳定，增加一个轻量规则：

若一轮工具调用中同时出现 `move_mouse` 和 `click_mouse`，并且这两个工具之间没有 `wait` 且本轮也没有显式说明是大目标，后端不拦截。  
第一版仍允许，因为硬拦截太复杂。  
但日志要记录“同轮 move+click”。后续可通过评估观察模型是否经常违规。

真正保证准确性的核心仍然是提示词和自动截图机制。

## 74.3 模型上下文中的反馈闭环

每次 move 后，自动截图消息都会告诉模型：

现在鼠标在哪里。  
这张图上鼠标被标出来了。  
坐标空间是多少。

这样模型即使第一次给错坐标，也能根据“当前错误位置”继续二次修正。这正是你需求里想要的双重反馈机制。

## 77. 这一部分的实现结论

到这里，Agent 逻辑已经固定：

使用 OpenAI 兼容 Chat Completions。  
不显式暴露截图工具，而是由 harness 自动提供截图。  
工具面固定为阶段文本、进展、鼠标键盘、等待、窗口激活、bash、完成、请求用户操作。  
每轮按“模型输出工具 -> 后端顺序执行 -> 自动截图 -> 再给模型”的闭环运行。  
旧 run 不恢复，用户后续补充说明统一创建新 run。  
`waiting_user` 是可继续发新消息的暂停态，不是阻塞态。  
小目标点击依靠“move -> 新截图确认 -> click”的多轮方式提升准确性。
