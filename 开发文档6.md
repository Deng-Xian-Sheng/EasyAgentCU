# EasyAgentCU 开发文档（第 6 部分）

## 40. 桌面控制与实时画面实现总原则

这一部分决定 Agent 是否真正“能操作电脑”，所以实现上必须保守而稳定。

第一版统一采用 X11 原生命令链，不接入 Playwright、不接入 Selenium、不接入 OCR 框架、不接入第二套浏览器自动化体系。所有桌面操作都通过本机 X11 会话执行，所有实时画面都从当前 X11 桌面采集。

固定技术路径如下：

桌面动作执行使用 `xdotool`。  
屏幕信息读取使用 `xrandr` 和 `xdotool getmouselocation`。  
截图使用 `scrot`。  
截图后处理、缩放、鼠标标注使用 Go 图片库。  
实时桌面画面使用 `x11vnc + websockify + noVNC`。  
Go 服务负责进程守护、健康检查、WebSocket 代理、文件持久化。  
所有外部命令都必须继承同一套 X11 环境变量。

这一版有一个非常重要的工程约束：虽然“每个会话同一时刻只允许一个运行中的 Agent 任务”已经足够满足需求，但由于整个系统只有一个真实桌面，所以后端还必须加一个全局桌面执行锁。也就是不管来自哪个会话，同一时刻只能有一个 run 进入桌面动作执行区。这样可以彻底避免两个任务抢鼠标和键盘。

## 41. X11 运行环境约束

EasyAgentCU 不是在纯服务环境里运行，而是在已经登录进入 Deepin V25 图形桌面的用户会话中运行。只有在这个前提下，`xdotool`、`scrot`、`x11vnc` 才能稳定工作。

Go 服务启动后，所有需要访问 X11 的子进程都必须带上这两个环境变量：

```text
DISPLAY=:0
XAUTHORITY=/home/<user>/.Xauthority
```

其中 `DISPLAY` 来自配置项 `X11_DISPLAY`。  
`XAUTHORITY` 若配置为空，则优先按下面顺序自动推断：

先读取环境变量 `XAUTHORITY`。  
再尝试 `${HOME}/.Xauthority`。  
再尝试 `/run/user/<uid>/gdm/Xauthority` 这类路径不推荐，因为 Deepin 场景不稳定。

程序启动期必须执行一轮桌面自检。自检内容不是只检查命令存在，而是要确认当前显示会话真的可用。推荐顺序如下：

先执行 `xrandr --current`，确认能取到 `current WxH`。  
再执行 `xdotool getmouselocation --shell`，确认能拿到鼠标位置。  
再执行一次 `scrot` 到临时目录，确认截图成功。  
再检查 `x11vnc` 和 `websockify` 命令可执行。  
最后删除自检截图。

只要其中任意一步失败，程序直接启动失败。不要让系统在用户第一次发消息时才发现桌面不可用。

## 42. 外部命令执行封装

所有系统命令都必须走统一封装，不能在业务代码里直接 `exec.Command(...)` 到处写。

推荐定义一个公共执行器：

```go
type CmdRunner interface {
    Run(ctx context.Context, name string, args []string, opts CmdOptions) (CmdResult, error)
}

type CmdOptions struct {
    Env          map[string]string
    Dir          string
    Timeout      time.Duration
    CaptureStdout bool
    CaptureStderr bool
}

type CmdResult struct {
    ExitCode int
    Stdout   string
    Stderr   string
    Duration time.Duration
}
```

这个执行器要负责四件事。

第一，统一注入 X11 环境变量。  
第二，统一记录日志，包括命令名、参数、耗时、退出码。  
第三，支持超时和进程组杀死。  
第四，把常见错误转成可分类错误，例如 `desktop_unavailable`、`command_not_found`、`timeout`。

超时处理不能只 kill 主进程，必须 kill 整个进程组。因为 bash、websockify 这类命令经常会再派生子进程。Go 侧需要给命令加：

```go
cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}
```

超时后执行：

```go
syscall.Kill(-cmd.Process.Pid, syscall.SIGKILL)
```

这样才不会残留僵尸子进程。

## 43. DesktopService 设计

桌面动作不要散落在工具执行器里，必须集中到一个服务里。

推荐接口如下：

```go
type ScreenInfo struct {
    DesktopWidth     int
    DesktopHeight    int
    ModelImageWidth  int
    ModelImageHeight int
    ScaleX           float64
    ScaleY           float64
}

type MouseLocation struct {
    X int
    Y int
    Screen int
    Window string
}

type ScreenshotFiles struct {
    RawPath       string
    ModelPath     string
    AnnotatedPath string
}

type ScreenshotBundle struct {
    ID            string
    ScreenInfo    ScreenInfo
    Mouse         MouseLocation
    Files         ScreenshotFiles
    CreatedAt     time.Time
}

type DesktopService interface {
    EnsureReady(ctx context.Context) error
    GetScreenInfo(ctx context.Context) (ScreenInfo, error)
    GetMouseLocation(ctx context.Context) (MouseLocation, error)
    CaptureScreenshotBundle(ctx context.Context, conversationID, runID string) (*ScreenshotBundle, error)

    MoveMouse(ctx context.Context, x, y int) error
    Click(ctx context.Context, button string) error
    DoubleClick(ctx context.Context, button string) error
    Scroll(ctx context.Context, direction string, amount int) error
    TypeText(ctx context.Context, text string) error
    KeyPress(ctx context.Context, keys []string) error
    Drag(ctx context.Context, fromX, fromY, toX, toY int) error
    Wait(ctx context.Context, d time.Duration) error
}
```

实现时必须注意：`MoveMouse`、`Click`、`TypeText`、`Drag` 等实际会改变桌面的动作都必须经过全局桌面互斥锁；`GetScreenInfo`、`GetMouseLocation`、`CaptureScreenshotBundle` 虽然是读操作，也建议共享同一把锁，避免在拖动中途截到半帧或坐标和截图不一致。

## 44. 屏幕尺寸与鼠标位置获取

## 44.1 获取桌面逻辑尺寸

不要从前端画布尺寸推导。  
也不要从图片尺寸反推。  
后端必须直接从 X11 取当前桌面逻辑尺寸。

推荐命令：

```bash
xrandr --current
```

解析第一行：

```text
Screen 0: minimum 8 x 8, current 1920 x 1080, maximum 32767 x 32767
```

取其中的 `current 1920 x 1080` 作为桌面逻辑尺寸。

这一版按单显示器处理。即使 VM 中未来出现多显示器，也统一以 X11 的整个根窗口尺寸为准。

## 44.2 获取鼠标位置

推荐命令：

```bash
xdotool getmouselocation --shell
```

输出类似：

```text
X=286
Y=102
SCREEN=0
WINDOW=56623112
```

直接解析成结构体。

这里不要使用“先截图再估计鼠标位置”。  
鼠标位置必须以 X11 当前真实坐标为准，然后再画到截图上。

## 45. 截图实现

## 45.1 原始截图采集

原始截图一律截整个桌面，不截局部窗口。  
推荐命令：

```bash
scrot /tmp/easyagentcu-shot-xxxx.png
```

执行前先创建唯一临时文件名。  
截图成功后，把临时文件交给 Go 代码继续处理。

这里不要启用 scrot 自带鼠标指针渲染。原因有两个。

第一，不同桌面主题下指针样式和大小不一致。  
第二，需求里明确要求“圈和箭头指向鼠标指针的位置，并写上‘鼠标指针’四个字”，这必须由我们自己的标注层来做。

## 45.2 模型输入图尺寸规则

模型输入图不一定等于原始桌面尺寸。  
缩放规则固定如下：

若原始尺寸宽高都不超过 `MODEL_IMAGE_MAX_WIDTH` 和 `MODEL_IMAGE_MAX_HEIGHT`，则不放大，只保持原尺寸。  
若超出，则按等比缩放，把图压到限制框内。  
推荐默认限制为 `1600x900`。

缩放后记录：

`desktop_width`  
`desktop_height`  
`model_image_width`  
`model_image_height`  
`scale_x = desktop_width / model_image_width`  
`scale_y = desktop_height / model_image_height`

这些值必须和 screenshot 记录一起入库。

## 45.3 文件生成顺序

每次截图生成固定三份文件。

第一份 `raw`。原始整屏 PNG，不做标注，不改尺寸。  
第二份 `model`。给模型发送的那张图，等比缩放。  
第三份 `annotated`。给前端历史进展回放和模型视觉理解辅助使用的标注图。

推荐处理顺序如下：

先落 `raw`。  
再从 `raw` 生成 `model`。  
再从 `model` 生成 `annotated`。  
因为标注图应该和模型看到的坐标空间一致，这样模型看到的箭头、圈、坐标文本与它实际要输出的坐标空间完全一致。

也就是说，箭头和“鼠标指针”标注应画在 `model` 尺寸空间里，而不是画在 `raw` 上再缩放。否则细线条可能因为再次缩放变模糊。

## 45.4 图片编码策略

`raw` 固定保存 PNG。  
`annotated` 固定保存 PNG。  
`model` 默认保存 PNG；若编码后文件过大，导致 base64 预计超过 `MAX_IMAGE_BASE64_MB`，则自动退化为 JPEG。

具体规则如下：

先编码 PNG。  
若文件大小乘以 `4/3` 后超过配置上限，则改用 JPEG 质量 90。  
若仍超限，依次降到 85、80、75。  
降到 75 仍超限，则继续降低模型输入尺寸，例如按 0.9、0.8 比例再次缩小，直到满足上限。  
整个过程必须记录日志。

这样可以同时满足“尽量清晰”和“不能超过模型接口上限”。

## 46. 鼠标标注图实现

这是项目需求中的明确重点，必须具体实现，不能只写在提示词里。

## 46.1 标注元素

每张 `annotated` 图都必须包含三个视觉元素。

第一个元素是一个圆圈，圆心就是当前鼠标坐标。  
第二个元素是一根箭头，箭头指向圆圈。  
第三个元素是文字标签“鼠标指针”。

颜色统一使用高对比的红色系。推荐：

圆圈和箭头主色：`#ff3b30`。  
文字白色描边后填充 `#ff3b30` 或直接白字配深色底。  
线宽随图片尺寸自适应，不能太细。

## 46.2 标注布局算法

不要把箭头和文字永远画在固定位置，否则遇到鼠标靠边时会被裁切。

建议算法如下：

先把鼠标坐标从桌面空间映射到模型图空间。  
再根据鼠标所在象限，选择标签落点方向。  
若鼠标在画面左上区域，标签放右上。  
若鼠标在画面右上区域，标签放左上。  
若鼠标在画面左下区域，标签放右下。  
若鼠标在画面右下区域，标签放左下。  
再做边界裁切修正，确保标签框完全落在图片内。

圆圈半径建议为：

```text
max(18, min(modelWidth, modelHeight) * 0.018)
```

箭头长度建议为圆圈半径的 3 到 4 倍。  
文字标签建议带浅色半透明底，避免压在复杂背景上看不清。

## 46.3 标注内容附加到模型文本上下文

除了给模型图片本身，后端还必须在发图时附带一段说明文字，例如：

```text
这是当前屏幕截图。图中已经用红圈和箭头标出了鼠标指针位置，标签文字为“鼠标指针”。
当前桌面逻辑分辨率为 1920x1080。
发送给你的这张模型截图分辨率为 1600x900。
当前鼠标在桌面坐标系中的位置为 (286, 102)。
你返回的坐标必须基于模型截图 1600x900 的坐标空间。
```

这样就实现了需求中说的“双管齐下”：既提供视觉标注，又提供明确坐标文本。

## 47. 动作执行实现

## 47.1 MoveMouse

命令：

```bash
xdotool mousemove --sync <x> <y>
```

这里的 `<x> <y>` 必须是桌面逻辑坐标，不是模型坐标。  
所以工具执行器在调用 DesktopService 前，必须先完成一次坐标映射。

`--sync` 必须保留，确保命令返回时鼠标已经到位。  
执行完成后默认等待 `80ms` 到 `120ms`，避免桌面渲染还没跟上就继续下一步。

## 47.2 Click

左键、中键、右键分别映射为 `1/2/3`。

命令：

```bash
xdotool click --delay 80 1
```

点击前不自动移动鼠标。  
模型若要点击某坐标，应先单独调用 `move_mouse`，再调用 `click`。这和需求中的鲁棒性设计一致。

## 47.3 DoubleClick

命令：

```bash
xdotool click --repeat 2 --delay 120 1
```

双击前也不自动移动。  
仍然要求模型先 move，再观察确认，再 double_click。

## 47.4 Scroll

滚动不使用鼠标拖拽，而是使用滚轮按钮号。  
上滚是 `4`，下滚是 `5`。

例如向下滚 5 次：

```bash
xdotool click --repeat 5 --delay 80 5
```

前端或模型参数中的 `amount` 统一解释为“滚轮刻度数”，不是像素。  
这样最稳定，也最容易和网页/应用兼容。

## 47.5 TypeText

命令：

```bash
xdotool type --delay 20 --clearmodifiers -- "<text>"
```

必须加 `--clearmodifiers`，否则当系统残留 Ctrl/Shift/Alt 修饰键时，输入会错乱。  
文本参数一定要走 argv 传递，不能拼接 shell，避免转义问题。

对于特别长的文本，建议按 200 到 500 字一段切块输入，每段后等待 `50ms`。  
否则某些应用输入框可能丢字。

## 47.6 KeyPress

快捷键使用：

```bash
xdotool key --clearmodifiers ctrl+l
xdotool key --clearmodifiers alt+Tab
xdotool key --clearmodifiers Return
```

工具层参数建议统一成字符串数组，例如：

```json
["ctrl", "l"]
```

执行时拼成 `ctrl+l`。  
这样 schema 更稳定，也更容易校验。

## 47.7 Drag

实现顺序固定：

先 `mousemove` 到起点。  
再 `mousedown 1`。  
再 `mousemove --sync` 到终点。  
最后 `mouseup 1`。

命令序列：

```bash
xdotool mousemove --sync <fromX> <fromY>
xdotool mousedown 1
xdotool mousemove --sync <toX> <toY>
xdotool mouseup 1
```

每一步之间留出很短的等待，例如 `50ms`。  
不要用一步到位的简化命令，否则某些应用里拖拽识别不稳定。

## 47.8 Wait

等待使用 Go 自己的 `time.Sleep` 即可，不需要调用外部命令。  
支持毫秒级。  
模型参数若给秒数，后端转换为 `time.Duration`。

## 48. 实时画面实现

## 48.1 总体结构

实时画面不依赖截图记录。  
它是一条独立链路：

```text
X11 桌面(:0) -> x11vnc -> websockify -> Go WebSocket 代理 -> 浏览器 noVNC
```

这里的目标不是录屏，而是让用户始终能看到当前真实桌面，并且在需要时可以接管操作。

## 48.2 x11vnc 启动参数

推荐固定命令：

```bash
x11vnc \
  -display :0 \
  -rfbport 5900 \
  -localhost \
  -forever \
  -shared \
  -nopw \
  -xkb \
  -noxdamage \
  -repeat
```

参数含义按实现结果理解即可：

`-localhost` 表示只监听本机。外部浏览器不直接连它。  
`-forever` 表示客户端断开后不退出。  
`-shared` 表示允许多个观察者连接。  
`-nopw` 因为只绑定本机，真正暴露面在 Go 服务层。  
`-noxdamage` 可以减少某些合成器环境下的残影与黑块。  
`-xkb` 和 `-repeat` 有助于键盘事件稳定。

## 48.3 websockify 启动参数

推荐命令：

```bash
websockify 127.0.0.1:6080 127.0.0.1:5900
```

这里只负责把 TCP VNC 转成 WebSocket。  
noVNC 静态文件由 Go 服务或前端包自己提供，不依赖 `websockify --web`。

## 48.4 LiveViewService 设计

推荐接口：

```go
type LiveViewService interface {
    EnsureStarted(ctx context.Context) error
    HealthCheck(ctx context.Context) error
    WebSocketTarget() string
    SessionID() string
}
```

第一版 `SessionID` 固定返回 `desktop-default`。  
因为整个系统只有一个真实桌面，没有每个 run 独立桌面会话。

`EnsureStarted` 的逻辑必须是幂等的。  
也就是多次调用不会重复启动多个 `x11vnc` 或 `websockify` 进程。

推荐做法是：

程序启动时调用一次 `EnsureStarted`。  
每次 run 创建时再调用一次，作为保险。  
内部先探测端口和进程是否健康，健康则直接返回。  
不健康才重启。

## 48.5 进程守护

不要把 `x11vnc` 和 `websockify` 完全交给 systemd 外部守护。  
EasyAgentCU 自己也必须能发现它们异常退出并自动重启。

推荐实现：

Go 内部保存两个受管进程句柄。  
每个进程 `Start()` 后起一个 goroutine `Wait()`。  
一旦异常退出，记录日志，进入指数退避重启。  
退避序列建议 `1s -> 2s -> 4s -> 8s`，上限 `30s`。  
重启成功后恢复健康状态。

这样用户不需要手动修复实时画面链路。

## 48.6 Go WebSocket 代理

为了让浏览器始终访问同源地址 `/api/live/ws`，Go 服务必须提供 WebSocket 代理。  
这里需要新增一个后端依赖：

```text
github.com/gorilla/websocket
```

实现原则是：

浏览器连到 `/api/live/ws`。  
Go 在服务端把这个连接升级为 WebSocket。  
再作为客户端去连 `ws://127.0.0.1:6080/`。  
之后双向拷贝二进制帧和文本帧。  
任一侧断开，则关闭另一侧。

这个代理不需要理解 VNC 协议内容。  
它只是转发。

## 49. 前端实时画面接入细节

前端 `NoVncLiveView` 组件只负责三件事：

获取 `/api/live-view` 配置。  
创建 noVNC `RFB` 实例并连接 `/api/live/ws`。  
向父组件暴露连接状态，例如 `connecting/connected/disconnected/error`。

默认连接参数建议：

关闭本地光标绘制冲突。  
允许剪贴板但第一版不对外暴露。  
全屏接管态下允许发送键盘和鼠标。  
普通嵌入态只读显示也可以允许点击，因为需求里点击实时窗口要进入全屏，实际操作放到全屏再进行。

当连接失败时，屏幕窗口不要整个消失。  
而是显示一个固定大小的深色占位框，中心显示“实时画面连接失败”，并保留重试按钮。  
这样页面结构不会跳动。

## 50. bash 工具实现

## 50.1 工具定位

bash 在本项目中不是附属工具，而是模型的兜底能力。  
当纯鼠标键盘操作低效、页面复杂、系统级任务需要执行时，模型可以直接用 bash。

因为项目运行在用户专门准备的虚拟机里，第一版不对 bash 做权限限制。  
但为了稳定性，仍然必须做下面三件事：

限制单次执行超时。  
限制 stdout/stderr 最大返回量。  
把执行过程完整记录为 bash block，并通过 SSE 推给前端。

## 50.2 BashService 接口

```go
type BashExecResult struct {
    ExitCode   int
    Stdout     string
    Stderr     string
    Duration   time.Duration
    TimedOut   bool
    Truncated  bool
}

type BashService interface {
    Run(ctx context.Context, command string) (*BashExecResult, error)
}
```

## 50.3 执行方式

固定使用：

```bash
/bin/bash -lc "<command>"
```

不要直接 `sh -c`。  
也不要逐词拆成 argv，因为模型给出的就是一整段 bash 命令。

执行时要继承 EasyAgentCU 的环境变量，至少包括：

`DISPLAY`  
`XAUTHORITY`  
`HOME`  
`PATH`

这样模型在 bash 中调用 `xdotool`、`curl`、`python`、`ls` 等命令时环境一致。

工作目录建议设为当前用户的 home 目录。  
因为很多桌面场景下用户文件、下载目录、桌面目录都在 home 下，模型更容易找到资源。

## 50.4 输出截断策略

`stdout + stderr` 合计不能无限增长。  
建议读取时就做流式限长，而不是等执行完再整体读入内存。

推荐上限就是配置项 `BASH_MAX_OUTPUT_KB`。  
超过后：

继续允许进程执行。  
但不再继续累积内存输出。  
最终在截断位置追加：

```text
...[output truncated]
```

同时返回 `Truncated=true`。

## 50.5 超时处理

bash 超时后，必须杀死整个进程组，不允许留后台子进程。  
结果写成：

`ExitCode = -1`  
`TimedOut = true`  
`Stderr` 末尾补一行 `command timed out`

bash block 最终状态记为 `error`。

## 50.6 bash block 和 SSE 的生命周期

bash 一旦开始执行，后端立刻创建 `bash_blocks` 记录，状态是 `running`，并发送 `run.bash.started`。  
执行过程中若输出很长，可以每隔 500ms 到 1000ms 刷新一次数据库和 SSE，发送 `run.bash.updated`。  
执行结束后统一更新状态、退出码、耗时，并发送 `run.bash.completed`。

这样前端能像原型一样看到 bash 调用块逐步出现和变化。

## 51. 截图、实时画面、动作之间的协同顺序

这一段决定执行稳定性，必须固定。

当 run 刚开始时：

先确保 LiveViewService 正常。  
一旦正常，发 `run.live.ready`。  
然后进入模型循环。  
也就是说，用户发消息后界面能先出现实时窗口，再等模型开始分析。

当模型需要新截图时：

先取当前桌面逻辑尺寸。  
再取当前鼠标坐标。  
再截 `raw`。  
再生成 `model` 和 `annotated`。  
再入库并发 `run.screenshot.created`。  
若本次截图会对应一条进展，再创建 `progress_item` 并发 `run.progress.created`。

当模型要求移动鼠标并检查位置时：

执行 move。  
等待短暂稳定时间。  
重新截图。  
模型查看新的带标注图。  
若确认位置正确，再继续 click。  
这个流程在工具和提示词两层都要支持，后面 Agent 部分会把它写死到执行策略里。

## 52. 错误处理与降级策略

## 52.1 截图失败

只要截图命令失败，这一步不能静默跳过。  
因为模型的视觉上下文就断了。

处理规则：

当前工具步骤立即失败。  
记录错误日志。  
run 状态改为 `failed`。  
发送 `run.failed`，错误文案对前端可读，例如“屏幕截图失败，无法继续分析当前桌面”。

不要让模型在没有新截图的情况下盲操作。

## 52.2 实时画面失败

实时画面失败不一定需要让整个 run 失败。  
因为模型操作依赖的是截图，不是 VNC。

处理规则：

若 `x11vnc/websockify` 在 run 开始前不可用，且截图、自检都正常，则 run 仍可继续。  
但前端实时窗口显示“实时画面不可用”。  
同时后端继续后台重启 live view 进程。  
若重启成功，可在后续 run 或本次 run 中再次发 `run.live.ready`。

不过为了符合你给的产品体验，推荐生产部署时把 live view 视为强依赖，在程序启动期就确保可用。

## 52.3 桌面命令偶发失败

比如 `xdotool` 因为窗口短暂切换导致返回非零。

处理规则不要过度激进。  
对于动作命令，可以允许一次轻量重试。  
推荐：

`mousemove/click/type/key/drag/scroll` 失败时，等待 `300ms` 再重试一次。  
第二次仍失败，则工具失败。

但截图和 bash 不建议自动重试太多。  
截图失败一般意味着桌面环境出了问题；bash 失败一般是业务错误，不应盲重试。

## 53. Deepin V25 X11 下的稳定性建议

这部分虽然不是 API 代码，但必须在实现中默认考虑，否则运行非常不稳定。

系统必须禁用自动锁屏、自动睡眠、自动熄屏。  
桌面缩放和分辨率允许变化，但每次截图都要重新取最新尺寸，不能缓存。  
不要假设浏览器始终全屏，也不要假设窗口始终固定位置。  
不要假设鼠标指针样式固定。  
不要假设用户不会手动干预桌面。用户全屏接管后，桌面状态可能和模型预期完全不同，所以每一步都要依赖最新截图，而不是依赖上一步推测。

## 54. 本部分的落地结论

到这里，桌面能力的落地方案已经固定：

桌面读写统一走 X11 原生命令。  
实时画面统一走 `x11vnc + websockify + noVNC`。  
历史画面统一走后端持久化截图。  
鼠标鲁棒性通过“移动后复查截图 + 标注鼠标 + 提供坐标文本”三件事一起实现。  
bash 完全开放，但执行过程有超时、截断、事件化展示。  
所有命令都经统一执行器封装，并继承一致的 X11 环境变量。  
所有实时事件都先落库后广播。
