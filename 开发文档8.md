# EasyAgentCU 开发文档（第 8 部分）

## 78. 构建与打包方案

项目最终交付物是一个 Go 可执行二进制。  
但前端仍然按 Next.js 项目开发。  
因此构建流程固定分两段：

第一段构建前端静态站点。  
第二段把前端产物嵌入 Go 二进制并构建后端。

## 78.1 前端构建要求

`frontend/next.config.ts` 必须配置为静态导出模式。  
不要使用需要运行时 Node server 的能力。

核心要求：

`output: "export"`  
全部页面为客户端组件或可静态导出页面  
不使用 Next API routes  
不依赖服务端动态渲染

构建命令：

```bash
cd frontend
npm install
npm run build
```

构建产物输出到：

```text
frontend/out/
```

然后由脚本复制到：

```text
web-dist/export/
```

推荐脚本 `scripts/build-frontend.sh`：

```bash
#!/usr/bin/env bash
set -euo pipefail

cd frontend
npm install
npm run build

rm -rf ../web-dist/export
mkdir -p ../web-dist/export
cp -r out/* ../web-dist/export/
```

## 78.2 Go 嵌入前端静态文件

`internal/static/embed.go` 使用 `embed`：

```go
package static

import "embed"

//go:embed all:web-dist/export
var FrontendFS embed.FS
```

HTTP 层统一从嵌入文件系统里提供静态资源。  
开发环境可以通过配置切换成本地目录读取，但生产默认用 embed。

前端静态资源路由规则：

请求 `/assets/*`、`/_next/*`、静态文件时，直接返回嵌入文件。  
请求任意非 `/api`、非 `/files`、非 `/novnc` 路径时，统一回退到 `index.html`。  
因为前端只有单页面路由，回退逻辑很简单。

## 78.3 后端构建命令

推荐：

```bash
go build -o dist/easyagentcu ./cmd/easyagentcu
```

完整打包脚本 `scripts/package.sh`：

```bash
#!/usr/bin/env bash
set -euo pipefail

./scripts/build-frontend.sh
mkdir -p dist
go build -trimpath -ldflags="-s -w" -o dist/easyagentcu ./cmd/easyagentcu
```

## 79. 运行环境准备清单

虚拟机操作系统固定为 Deepin V25 X11。  
部署前必须人工准备以下内容。

安装基础依赖：

```bash
sudo apt update
sudo apt install -y \
  xdotool \
  xrandr \
  scrot \
  x11vnc \
  websockify \
  curl \
  jq \
  bash
```

如果 `websockify` 在系统源里名称不同，按实际包名安装。  
若系统源没有合适版本，可通过 Python 安装，但最终部署文档里必须写死安装方式，不能让程序员猜。

还要确保：

图形桌面为 X11，而不是 Wayland。  
部署用户已经登录进入桌面。  
存在可访问的 `.Xauthority`。  
关闭自动锁屏、自动睡眠、自动熄屏。  
网络可访问模型 API。  
宿主机可通过虚拟机 IP 访问 `APP_PORT`。

## 80. systemd 自启动方案

因为需求明确说要在虚拟机中部署并自启动，所以给出标准 systemd 方案。

假设：

程序路径 `/opt/easyagentcu/easyagentcu`  
运行用户 `easyagent`  
工作目录 `/opt/easyagentcu`

创建环境文件：

```bash
sudo mkdir -p /etc/easyagentcu
sudo tee /etc/easyagentcu/easyagentcu.env >/dev/null <<'EOF'
APP_ENV=production
APP_HOST=0.0.0.0
APP_PORT=8080
PUBLIC_BASE_URL=
DATA_DIR=/opt/easyagentcu/data
TEMP_DIR=/opt/easyagentcu/data/tmp
SQLITE_PATH=/opt/easyagentcu/data/app.db

MODEL_BASE_URL=https://your-openai-compatible-endpoint/v1
MODEL_API_KEY=your_api_key
MODEL_NAME=Qwen3.5-397B-A17B
MODEL_TIMEOUT_SEC=180
MODEL_MAX_RETRIES=2
MODEL_TEMPERATURE=0.2
MODEL_MAX_TOKENS=4096

SCREENSHOT_TOOL=scrot
MODEL_IMAGE_MAX_WIDTH=1600
MODEL_IMAGE_MAX_HEIGHT=900
MAX_IMAGE_BASE64_MB=10

X11_DISPLAY=:0
XAUTHORITY_PATH=/home/easyagent/.Xauthority

VNC_ENABLED=true
VNC_HOST=127.0.0.1
VNC_PORT=5900
VNC_WEBSOCKET_HOST=127.0.0.1
VNC_WEBSOCKET_PORT=6080

BASH_SHELL_PATH=/bin/bash
BASH_TIMEOUT_SEC=120
BASH_MAX_OUTPUT_KB=512

AGENT_LOOP_MAX_STEPS=60
AGENT_WAIT_AFTER_ACTION_MS=800
AGENT_SCREENSHOT_ON_EVERY_STEP=true

LOG_LEVEL=info
EOF
```

创建服务文件：

```ini
[Unit]
Description=EasyAgentCU
After=network-online.target graphical.target
Wants=network-online.target

[Service]
Type=simple
User=easyagent
WorkingDirectory=/opt/easyagentcu
EnvironmentFile=/etc/easyagentcu/easyagentcu.env
ExecStart=/opt/easyagentcu/easyagentcu
Restart=always
RestartSec=3
KillMode=process
TimeoutStopSec=15

[Install]
WantedBy=multi-user.target
```

但这里有一个现实问题：  
服务依赖图形桌面会话。单纯 `After=graphical.target` 不一定保证用户已经登录桌面。  
因此最佳实践是：

让程序由“自动登录进入桌面”的用户会话环境启动，或者  
写成用户级 systemd 服务，在图形登录后启动。

如果部署时能确保虚拟机自动登录桌面，系统级服务也能工作，只要 `.Xauthority` 和 `DISPLAY` 正确。

启用方式：

```bash
sudo systemctl daemon-reload
sudo systemctl enable easyagentcu
sudo systemctl start easyagentcu
```

## 81. 程序启动期检查表

程序启动时必须做一轮完整自检，并把结果打到日志里。  
建议实现 `PreflightCheck()`，失败则直接退出。

检查项固定如下：

配置合法。  
`DATA_DIR`、`TEMP_DIR` 可创建。  
SQLite 可打开并迁移成功。  
`xdotool` 存在。  
`xrandr` 存在。  
`scrot` 存在。  
`x11vnc` 存在。  
`websockify` 存在。  
`DISPLAY` 可访问。  
`XAUTHORITY` 可访问。  
能读到当前桌面分辨率。  
能读到当前鼠标位置。  
能成功截一张测试图。  
若 `VNC_ENABLED=true`，则能启动或连接到 live view 组件。  
模型基础配置存在。

建议启动日志示例：

```text

INFO preflight sqlite ok
INFO preflight desktop resolution=1920x1080
INFO preflight screenshot ok
INFO preflight live_view ok ws=127.0.0.1:6080
INFO service started addr=0.0.0.0:8080
```

## 82. 服务重启恢复策略

这是稳定性关键点，必须落地。

程序重启后，数据库里可能残留以下 run 状态：

`queued`  
`running`

这些 run 已经不可能继续，因为内存中的 `modelMessages`、取消上下文、当前轮状态全丢了。  
所以程序启动后必须执行一次恢复修正：

把所有 `queued` 和 `running` 的 run 统一改为 `failed`。  
错误信息写：

`service restarted during execution`

同时把对应 conversation 状态修正为：

如果该会话存在 `waiting_user` run，则 `waiting_user`。  
否则 `idle`。

这个修正不需要发 SSE，因为程序刚启动时没有客户端连接。  
但如果此时已有客户端连接，也可以发 `run.failed` 与 `conversation.updated`，属于增强项。

## 83. 日志规范

## 83.1 日志级别

统一使用结构化日志。  
日志级别：

`debug` 用于本地开发。  
`info` 用于关键生命周期事件。  
`warn` 用于可恢复异常。  
`error` 用于失败和致命异常。

## 83.2 必打日志点

下面这些事件必须打印 `info` 或更高等级日志：

程序启动。  
配置加载成功。  
数据库迁移成功。  
live view 启动/重启。  
收到用户消息。  
创建 run。  
run 开始、结束、取消、失败。  
每轮模型调用开始与结束。  
每次工具调用开始与结束。  
截图生成成功。  
SSE 连接建立与断开。  
WebSocket 代理连接建立与断开。

日志字段至少包含：

`request_id`  
`conversation_id`  
`run_id`  
`tool_name`  
`duration_ms`  
`status`

## 83.3 不要写入日志的内容

模型 API Key。  
完整图片 base64。  
用户上传文件的敏感原文全文。  
可能包含密码、验证码、Cookie 的屏幕文本。

模型请求体和响应体若落磁盘到 run 日志目录，也要注意权限，仅本地管理员可读。

## 84. 测试方案

第一版至少要覆盖三层测试。

## 84.1 单元测试

重点测这些纯逻辑模块：

配置解析与校验。  
坐标映射。  
截图缩放尺寸计算。  
鼠标标注位置算法。  
SSE 事件序号生成。  
ContextBuilder 历史裁剪逻辑。  
工具参数校验。  
DTO 转换。

例如必须写的测试：

给定桌面 `1920x1080`、模型图 `1600x900`，模型坐标 `(800,450)` 映射成桌面 `(960,540)`。  
给定鼠标在右下角，标签文字不会被画出图片边界。  
给定超过 12 轮历史，只保留最近 12 轮。  
给定 `waiting_user` 会话，仍允许创建下一条用户消息。

## 84.2 集成测试

重点测后端 API 与数据库。

建议覆盖：

创建会话。  
发送纯文本消息。  
发送带图片附件消息。  
发送重复 `clientMessageId` 的消息，确认幂等。  
正在运行时再次发消息，返回冲突。  
`waiting_user` 状态下发新消息成功。  
取消 run。  
列出 messages 与 runs。  
SSE 连接能收到 `run.created`、`run.status.changed` 等事件。

这些测试可以把 `ModelClient` 换成 fake，实现稳定输出。  
例如 fake 模型第一轮总是返回 `set_phase_text + finish_task`，这样可验证整条主链路。

## 84.3 桌面端到端测试

这部分必须人工为主，自动化为辅。  
至少跑下面这些验收任务：

任务一：打开百度热搜，提取最热门 10 条新闻。  
任务二：在浏览器里打开一个网页，滚动到指定位置。  
任务三：上传一张用户标注图，要求模型根据图片说明点击正确输入框。  
任务四：进入需要用户接管的页面，验证 `request_user_action` 流程。  
任务五：执行 bash，如列目录、读取文本文件并总结。  
任务六：用户点击全屏接管，自行操作后点“我已经操作”，系统能继续下一轮任务。

## 87. 开发优先级建议

虽然文档已经足够完整，但实际开发还是按下面顺序最稳：

先打通后端最小链路：创建会话、发消息、假模型、完成 run。  
再做前端基础页面：侧栏、聊天区、输入框、空态。  
再接 SSE 和 run 渲染。  
再接真实模型调用。  
再做截图与进展。  
再做 noVNC 实时画面。  
再做全屏接管与“我已经操作”。  
最后补细节：历史时间条、bash 块、重连补偿、日志与恢复。

这样开发时每一步都可验证，不会一开始就把桌面控制、模型、前端交互全部缠在一起。
