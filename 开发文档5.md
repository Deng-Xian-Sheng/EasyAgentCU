# EasyAgentCU 开发文档（第 5 部分）

## 32. API 总体设计

## 32.1 基本原则

第一版后端 API 统一采用下面几类协议：

普通读接口使用 `application/json`。  
写接口使用 `application/json` 或 `multipart/form-data`。  
事件推送使用 `text/event-stream`。  
文件访问使用普通静态文件 HTTP。  
实时桌面 WebSocket 由 Go 反向代理到 `websockify`，前端同源访问。

整套 API 不做鉴权。  
整套 API 只服务单用户、单实例。  
整套 API 统一挂在 `/api` 前缀下。  
文件静态访问统一挂在 `/files`。  
noVNC 静态资源统一挂在 `/novnc`。  
VNC WebSocket 统一挂在 `/api/live/ws`。

## 32.2 有状态与无状态划分

从 HTTP 协议角度看，所有请求都是无状态的。  
从产品行为角度看，接口分三类：

第一类是“纯查询接口”，例如列会话、列消息、列 run、健康检查。这类接口不改变系统状态。

第二类是“状态变更接口”，例如创建会话、发送消息、取消 run。这类接口会写数据库、触发异步任务、改变当前会话状态。

第三类是“长连接接口”，也就是 SSE 事件流和 VNC WebSocket。它们本身不保存浏览器 Session，但连接建立后会持续订阅当前系统状态变化，所以逻辑上属于弱状态连接。

程序员实现时不要误解成“因为没有登录就没有状态”。  
本项目最大的状态就是会话、run、桌面当前画面和 Agent 当前执行位置。

## 32.3 统一返回格式

普通 JSON 接口统一返回：

成功：

```json
{
  "requestId": "req_01JXXXX",
  "data": {}
}
```

失败：

```json
{
  "requestId": "req_01JXXXX",
  "error": {
    "code": "invalid_argument",
    "message": "text or files is required",
    "details": {}
  }
}
```

不要混用有的接口直接返回对象、有的接口返回包裹对象。  
所有 JSON 接口都统一包一层 `requestId + data/error`。

## 32.4 请求 ID

每个 HTTP 请求进入后端时都生成一个 `requestId`，写入：

响应 JSON。  
日志。  
错误日志。  
模型调用日志上下文。

SSE 事件不复用 HTTP `requestId`，SSE 自己使用 `eventId`。

## 33. 错误码规范

错误码固定如下，前后端都按这个集合处理。

| HTTP 状态码 | error.code               | 场景                                     |
| ----------- | ------------------------ | ---------------------------------------- |
| 400         | `invalid_argument`       | 参数缺失、格式错误、非法状态切换         |
| 404         | `not_found`              | 会话、消息、run、文件不存在              |
| 409         | `conflict`               | 会话已有运行中任务，不能并发发送         |
| 409         | `run_busy`               | 指定 run 正在执行中且不允许新操作        |
| 413         | `payload_too_large`      | 上传文件过大、图片超过限制               |
| 415         | `unsupported_media_type` | 不支持的文件类型                         |
| 422         | `desktop_unavailable`    | X11、VNC、截图命令不可用                 |
| 424         | `upstream_model_error`   | 模型接口失败或返回异常                   |
| 499         | `client_cancelled`       | 客户端主动取消连接，可只记日志不对外暴露 |
| 500         | `internal_error`         | 未分类服务端错误                         |

错误 `message` 给前端看，要写成人能读懂的话，不要把 Go 错误堆栈直接暴露出去。  
详细栈信息只写日志。

## 34. 路由清单

第一版路由固定如下：

```text
GET    /api/healthz
GET    /api/live-view

GET    /api/conversations
POST   /api/conversations
GET    /api/conversations/:conversationId

GET    /api/conversations/:conversationId/messages
POST   /api/conversations/:conversationId/messages

GET    /api/conversations/:conversationId/runs
GET    /api/runs/:runId
POST   /api/runs/:runId/cancel

GET    /api/conversations/:conversationId/events/stream

GET    /files/*
GET    /novnc/*
WS     /api/live/ws
```

额外补充一个数据库索引，放进 `001_init.sql` 里一起创建：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_messages_conversation_client_message_id_unique
ON messages(conversation_id, client_message_id)
WHERE client_message_id IS NOT NULL AND client_message_id <> '';
```

这个索引用来保证前端消息发送重试时不会重复创建两条用户消息。

## 36. 各接口详细文档

## 36.1 `GET /api/healthz`

用途是服务健康检查。  
前端通常不用，但部署和自启动脚本要用。

响应示例：

```json
{
  "requestId": "req_xxx",
  "data": {
    "status": "ok",
    "time": "2026-03-08T12:00:00Z",
    "version": "0.1.0"
  }
}
```

处理逻辑：

检查 HTTP 服务本身正常即可。  
不要在这个接口里调用模型、截图、VNC。  
否则健康检查会过重。

---

## 36.2 `GET /api/live-view`

用途是给前端 noVNC 组件提供连接配置。

响应示例：

```json
{
  "requestId": "req_xxx",
  "data": {
    "enabled": true,
    "novncBaseUrl": "/novnc",
    "websocketUrl": "/api/live/ws",
    "desktopName": "EasyAgentCU Desktop"
  }
}
```

处理逻辑：

只返回当前服务配置，不做实时探测。  
真正是否可连，由 `run.liveReady` 和 WebSocket 连接结果决定。

---

## 36.3 `GET /api/conversations`

查询参数：

`q`，可选，按标题做模糊搜索。  
`limit`，可选，默认 100，最大 500。

响应：

```json
{
  "requestId": "req_xxx",
  "data": {
    "items": [
      {
        "id": "conv_xxx",
        "title": "提取热搜新闻",
        "lastMessagePreview": "我正在打开百度热搜页面",
        "status": "running",
        "createdAt": "2026-03-08T12:00:00Z",
        "updatedAt": "2026-03-08T12:05:00Z"
      }
    ]
  }
}
```

处理逻辑：

只返回 `archived=0` 的会话。  
按 `updated_at desc` 排序。  
`lastMessagePreview` 优先级固定如下：

若该会话最后一个 run 有 `final_answer`，取最终回答前 60 个字符。  
否则若最后一个 run 有 `phase_text`，取阶段文本前 60 个字符。  
否则取最后一条用户消息前 60 个字符。  
都没有则返回空字符串。

## 36.7 `POST /api/conversations/:conversationId/messages`

这是最核心的写接口。  
请求必须使用 `multipart/form-data`。

字段定义：

`text`：字符串，可空。  
`clientMessageId`：字符串，可选，前端生成，用于幂等。  
`files`：可重复字段，0 到多个文件。

请求满足下面条件才合法：

文本非空，或者 files 至少一个。  
总上传体积不超过 `MAX_UPLOAD_SIZE_MB`。  
图片类文件扩展名和 MIME 必须在允许范围内。  
普通文件扩展名也必须在允许范围内。

成功响应：

```json
{
  "requestId": "req_xxx",
  "data": {
    "userMessage": {
      "id": "msg_user_xxx",
      "conversationId": "conv_xxx",
      "role": "user",
      "text": "打开百度热搜，提取最热门的10条新闻。",
      "attachments": [],
      "createdAt": "2026-03-08T12:00:00Z"
    },
    "assistantMessage": {
      "id": "msg_asst_xxx",
      "conversationId": "conv_xxx",
      "role": "assistant",
      "runId": "run_xxx",
      "createdAt": "2026-03-08T12:00:01Z"
    },
    "run": {
      "id": "run_xxx",
      "conversationId": "conv_xxx",
      "assistantMessageId": "msg_asst_xxx",
      "status": "queued",
      "phaseText": "",
      "finalAnswer": "",
      "needsUserAction": false,
      "userActionLabel": "",
      "liveReady": false,
      "liveSessionId": "",
      "screenMode": "live",
      "createdAt": "2026-03-08T12:00:01Z",
      "updatedAt": "2026-03-08T12:00:01Z",
      "errorMessage": "",
      "bashBlocks": [],
      "progressItems": [],
      "screenshots": []
    }
  }
}
```

冲突响应示例：

```json
{
  "requestId": "req_xxx",
  "error": {
    "code": "conflict",
    "message": "current conversation already has a running task",
    "details": {
      "runningRunId": "run_current_xxx"
    }
  }
}
```

处理逻辑必须按下面顺序实现：

第一步，校验 conversation 存在。  
第二步，检查该会话是否存在 `queued/running/waiting_user` 的 run。只要存在就返回 409。第一版不允许并发。  
第三步，若请求携带 `clientMessageId`，先查 `(conversation_id, client_message_id)` 是否已存在。若存在，直接返回那次已创建的 `userMessage + assistantMessage + run`，不要重复创建。  
第四步，在一个事务里完成：
创建用户消息。  
保存附件元数据。  
保存附件文件到磁盘。  
创建助手消息壳。  
创建 run，初始状态 `queued`。  
更新 conversation 的 `status=running`、`updated_at=now`。  
若会话标题还是默认“新聊天”或空标题，用本次用户文本前 30 个字符更新标题。  
第五步，事务提交。  
第六步，异步触发 AgentManager 启动该 run。  
第七步，立即返回 JSON，不等待模型完成。

这里有一个落地细节：  
文件保存要么在事务前完成、失败则不入库；要么先进临时目录，事务成功后再 move 到正式目录。推荐第二种，最稳。

用途是单个 run 补拉。

## 36.10 `POST /api/runs/:runId/cancel`

请求体为空即可。

成功响应：

```json
{
  "requestId": "req_xxx",
  "data": {
    "runId": "run_xxx",
    "status": "cancelling"
  }
}
```

处理逻辑：

第一步，查 run 是否存在。  
第二步，若状态已经是 `completed/failed/cancelled`，直接返回当前状态，幂等成功。  
第三步，若状态是 `queued/running/waiting_user`，通知 AgentManager 取消该 run 的 context。  
第四步，不要在 handler 里直接改数据库为 `cancelled`。真正状态落库由 runner 在收到取消信号后统一处理并发出 SSE。  
第五步，立即返回 `cancelling`。

这样可以避免 handler 和 runner 同时写状态导致竞态。

## 36.11 `GET /api/conversations/:conversationId/events/stream`

这是当前会话的 SSE 事件流。

响应头固定：

```http
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no
```

服务端推送格式固定为：

```text
id: evt_01JXXXX
event: message
data: {"eventId":"evt_01JXXXX","conversationId":"conv_xxx","runId":"run_xxx","seq":12,"type":"run.progress.created","data":{"progressItem":{...}}}

```

前端只监听默认 `message` 事件即可。  
真正业务分发看 `data.type`。

处理逻辑：

建立连接后，立即发一条 `heartbeat`。  
之后把该会话的事件订阅注册到内存广播器。  
每 15 秒发一次心跳，防止代理层断开。  
客户端断开连接时，及时注销订阅。  
这个接口不主动回放历史事件。断线补偿走 runs 全量补拉。

## 37. SSE 事件结构与载荷

## 37.1 统一结构

```json
{
  "eventId": "evt_01JXXXX",
  "conversationId": "conv_xxx",
  "runId": "run_xxx",
  "seq": 12,
  "type": "run.progress.created",
  "data": {}
}
```

其中：

`eventId` 来自 `run_events.id`。  
`seq` 来自该 run 的单调递增序号。  
`type` 固定取下面枚举之一。  
`data` 是各事件专属载荷。

## 37.2 事件类型与 data 结构

### `conversation.updated`

```json
{
  "conversation": {
    "id": "conv_xxx",
    "title": "提取热搜新闻",
    "lastMessagePreview": "我正在点击热搜页签",
    "status": "running",
    "createdAt": "2026-03-08T12:00:00Z",
    "updatedAt": "2026-03-08T12:00:20Z"
  }
}
```

### `message.created`

```json
{
  "message": {
    "id": "msg_asst_xxx",
    "conversationId": "conv_xxx",
    "role": "assistant",
    "runId": "run_xxx",
    "createdAt": "2026-03-08T12:00:01Z"
  }
```

## 38. 处理逻辑的后端实现约束

## 38.1 handler 层不直接返回数据库模型

每个 handler 都必须走 DTO 转换。  
数据库里是 `storage_path`，前端要的是 `url`。  
数据库里是 `text_content`，前端要的是 `text`。  
不要把内部字段原样透给前端。

## 38.2 发消息接口的事务边界

`POST /messages` 必须保证下面三件事要么一起成功，要么一起失败：

用户消息入库。  
助手消息壳入库。  
run 入库。

附件文件若已从临时目录移到正式目录，但事务失败，必须补偿删除。  
不要留下孤儿文件。

## 38.3 SSE 与数据库写入顺序

任何会话事件都必须先写数据库，再发内存广播。  
顺序不能反。

正确顺序是：

更新业务表。  
插入 `run_events`。  
提交事务。  
事务成功后广播给在线 SSE 订阅者。

这样即使广播时进程崩溃，重连后仍能靠数据库状态补齐。  
如果反过来，前端可能看到一个从未真正落库的瞬态事件。

## 38.4 conversation 状态同步规则

后端要在下面节点更新会话状态：

创建新 run 后，conversation 状态改成 `running`。  
run 进入 `waiting_user` 后，conversation 状态改成 `waiting_user`。  
run 进入 `completed/failed/cancelled` 后，若该会话没有别的活动 run，则改成 `idle`。

虽然第一版不允许并发 run，但代码仍应按“检查是否还有其他活动 run”写，便于以后扩展。

## 39. 文件与静态资源路由

## 39.1 `GET /files/*`

用途是前端展示用户附件、截图和缩略图。  
只允许读取 `DATA_DIR` 下白名单目录中的文件。  
必须做路径清洗，防止 `../` 穿越。

推荐逻辑：

先取 URL 中的相对路径。  
`path.Clean` 后确认仍在 `attachments/` 或 `runs/` 下。  
拼接 `DATA_DIR`。  
文件存在则直接 `http.ServeFile`。  
不存在返回 404。

不要把整个 `DATA_DIR` 直接裸 `http.FileServer` 出去。  
必须限制目录前缀。

## 39.2 `GET /novnc/*`

Go 服务把 noVNC 静态资源挂载到 `/novnc`。  
前端一般不直接跳转到 noVNC 页面，而是通过 npm 安装的 noVNC 客户端库嵌入。  
但保留这个路由，方便调试与备用。

## 39.3 `WS /api/live/ws`

Go 服务把这个 WebSocket 路由反代到本机 `websockify`。  
这样浏览器始终同源访问，不需要跨端口、跨源配置。  
前端 noVNC 直接连 `/api/live/ws`。

代理要求：

连接建立前检查 `VNC_ENABLED=true`。  
若未启用，直接拒绝。  
若本地 `websockify` 不可达，返回 502。  
代理过程不做消息级解析，只做字节转发。
