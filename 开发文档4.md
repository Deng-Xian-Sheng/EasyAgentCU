这里有几个硬约束。

`internal/domain` 只放领域结构定义，不写业务逻辑。  
`internal/db/queries` 只做数据库访问，不写流程编排。  
`internal/service` 负责面向 API 的业务编排。  
`internal/agent` 负责模型循环、工具调用、上下文管理。  
`internal/tools` 负责落地执行系统能力。  
`internal/api/handlers` 只做参数解析、调用 service、返回 DTO。  
`internal/static` 负责嵌入前端静态文件。

不要把模型调用、数据库写入、截图逻辑直接混写在 handler 里。  
不要把所有代码塞进 `main.go` 或单个 `service.go`。

## 27. Go 模块职责划分

为了让程序员直接开工，这里把每个核心文件该做什么写死。

## 27.1 `cmd/easyagentcu/main.go`

职责只有四个：

加载配置。  
初始化日志。  
初始化数据库并执行迁移。  
创建 app 并启动 HTTP 服务。

它不包含任何业务逻辑，不直接操作 SQLite，不直接注册每一条路由。

## 27.2 `internal/app/app.go`

负责组装整个应用依赖，包括：

配置对象。  
日志实例。  
数据库连接。  
各 service。  
AgentManager。  
HTTP router。  
静态文件服务。

它是依赖注入入口，但不使用复杂 DI 框架，直接手写构造函数。

## 27.3 `internal/config/config.go`

负责读取环境变量并做默认值填充、类型转换、合法性校验。  
配置一旦加载完成，必须变成只读结构体，后续全局传递。

## 27.4 `internal/db/*`

`db.go` 打开 SQLite，设置 pragma。  
`migrate.go` 执行数据库初始化和版本迁移。  
`queries/*.go` 每个文件只负责一个聚合根或一组相关表。

所有 query 层方法都要接受 `context.Context`。  
所有写操作都必须允许接收事务对象。

## 27.5 `internal/service/*`

这是 API 视角的服务层。比如：

创建会话。  
发消息。  
列出消息。  
列出 run。  
取消 run。  
流式事件订阅。

这里负责事务边界和数据拼装，但不直接执行模型主循环。  
例如 `message_service.go` 在收到用户消息后，会：

写入 user message。  
写入 assistant message 壳。  
写入 run。  
提交事务。  
异步通知 AgentManager 启动 run。

## 27.6 `internal/agent/*`

这是整个系统的心脏。

`manager.go` 负责每个会话只允许一个运行中的任务，维护运行 goroutine 生命周期。  
`runner.go` 负责单个 run 的执行循环。  
`prompt.go` 负责系统提示词与开发者提示词模板。  
`context_builder.go` 负责构造 messages 和图片输入。  
`model_client.go` 负责调用 OpenAI 兼容接口。  
`tool_registry.go` 定义模型可见工具 schema。  
`tool_executor.go` 负责按 tool call 执行具体工具并写回结果。  
`event_emitter.go` 负责把运行事件写入数据库并广播给 SSE 订阅者。  
`snapshot_builder.go` 负责截图、鼠标标注、图片缩放、图片 base64 载荷准备。

## 27.7 `internal/tools/*`

每个目录负责一类工具，不要交叉。

`tools/desktop` 管鼠标键盘截图窗口焦点。  
`tools/bash` 管 bash 调用。  
`tools/liveview` 管 noVNC、x11vnc、websockify 的检查与连接地址。  
`tools/files` 管附件和截图的磁盘写入、读取、删除。

## 28. 配置项设计

所有配置只从环境变量读取。  
不做 YAML，不做 TOML，不做命令行子命令参数系统。  
原因是部署目标是虚拟机内自启动服务，环境变量最稳。

## 28.3 配置校验规则

程序启动时，必须校验以下条件，不满足直接失败退出：

`MODEL_BASE_URL` 非空。  
`MODEL_API_KEY` 非空。  
`MODEL_NAME` 非空。  
`APP_PORT`、`VNC_PORT`、`VNC_WEBSOCKET_PORT` 合法。  
`MODEL_IMAGE_MAX_WIDTH` 和 `MODEL_IMAGE_MAX_HEIGHT` 都必须大于 0。  
`MAX_IMAGE_BASE64_MB` 必须小于等于 10。  
`DATA_DIR`、`TEMP_DIR` 父目录可创建。  
`BASH_SHELL_PATH` 存在。  
系统命令 `xdotool`、`xrandr`、`scrot`、`x11vnc`、`websockify` 必须在 PATH 中或有固定路径。

程序不要在第一次任务运行时才报“缺少 xdotool”。  
这些依赖必须在启动期一次性检查。

## 29. 数据库选型与连接策略

数据库固定为 SQLite。  
原因不是“简单”，而是这个项目部署在单台虚拟机、单用户、本地文件存储环境里，SQLite 最匹配。

## 29.1 SQLite 连接初始化

打开数据库后立即执行：

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
PRAGMA temp_store = MEMORY;
```

Go 层建议使用单个 `*sql.DB`，但限制连接池：

`SetMaxOpenConns(1)`  
`SetMaxIdleConns(1)`  
`SetConnMaxLifetime(0)`

这样最稳，避免 SQLite 在多连接下出现不必要锁争用。  
因为本项目吞吐不高，性能完全足够。

## 29.2 迁移策略

第一版不要引入复杂迁移框架。  
直接在 `migrations/001_init.sql` 保存完整建表 SQL，程序启动时检查 `schema_migrations` 表，没有则执行初始化。

以后新增版本时再加：

`002_xxx.sql`  
`003_xxx.sql`

`schema_migrations` 结构固定：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 30. 数据库表结构设计

下面是第一版必须落地的表。

## 30.1 conversations

一个会话就是左侧栏的一条聊天记录。

```sql
CREATE TABLE IF NOT EXISTS conversations (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'idle',
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  archived INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX IF NOT EXISTS idx_conversations_updated_at
ON conversations(updated_at DESC);
```

字段规则：

`id` 使用 UUID。  
`title` 初始可取第一条用户消息前 30 个字符，后续允许后台自动更新。  
`status` 只允许 `idle/running/waiting_user`。  
`archived` 第一版不暴露删除接口时也保留，便于后续软删除。

## 30.2 messages

消息表统一存用户消息和助手消息壳。

```sql
CREATE TABLE IF NOT EXISTS messages (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL,
  text_content TEXT NOT NULL DEFAULT '',
  run_id TEXT,
  client_message_id TEXT,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation_created
ON messages(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_messages_run_id
ON messages(run_id);
```

字段规则：

`role` 只允许 `user` 或 `assistant`。  
用户消息的 `run_id` 为空。  
助手消息的 `run_id` 必填，并与 runs 表一对一。  
`client_message_id` 用于前端临时消息去重。第一版建议支持。

不要把阶段文本、最终回答塞在 `messages.text_content` 里。  
`messages.text_content` 对用户消息有意义，对助手消息壳可为空。  
真正的阶段文本、最终回答放在 `runs` 表。

## 30.3 attachments

附件既支持用户上传图片，也支持普通文件。

```sql
CREATE TABLE IF NOT EXISTS attachments (
  id TEXT PRIMARY KEY,
  message_id TEXT NOT NULL,
  kind TEXT NOT NULL,
  file_name TEXT NOT NULL,
  mime_type TEXT NOT NULL,
  file_size INTEGER NOT NULL,
  storage_path TEXT NOT NULL,
  thumbnail_path TEXT,
  image_width INTEGER,
  image_height INTEGER,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_attachments_message_id
ON attachments(message_id);
```

字段规则：

`kind` 只允许 `image/file`。  
`storage_path` 存相对 `DATA_DIR` 的相对路径，不存绝对路径。  
图片附件若生成缩略图，则写 `thumbnail_path`。

## 30.4 runs

每次用户发消息触发一个 run。  
一个 run 对应一个 assistant message 壳。

```sql
CREATE TABLE IF NOT EXISTS runs (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  assistant_message_id TEXT NOT NULL,
  status TEXT NOT NULL,
  phase_text TEXT NOT NULL DEFAULT '',
  final_answer TEXT NOT NULL DEFAULT '',
  needs_user_action INTEGER NOT NULL DEFAULT 0,
  user_action_label TEXT,
  live_ready INTEGER NOT NULL DEFAULT 0,
  live_session_id TEXT,
  error_message TEXT,
  seq_counter INTEGER NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  started_at DATETIME,
  finished_at DATETIME,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (assistant_message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_runs_assistant_message_id
ON runs(assistant_message_id);

CREATE INDEX IF NOT EXISTS idx_runs_conversation_created
ON runs(conversation_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_runs_status
ON runs(status);
```

字段规则：

`status` 只允许 `queued/running/waiting_user/completed/failed/cancelled`。  
`phase_text` 存当前最新阶段文本，不存历史版本。  
`final_answer` 存最终回答正文。  
`needs_user_action` 与 `status=waiting_user` 可以一起出现，但逻辑上不强绑定，方便容错。  
`seq_counter` 用于生成该 run 的 SSE 事件序号。

## 30.5 bash_blocks

一个 run 中可能多次调用 bash。

```sql
CREATE TABLE IF NOT EXISTS bash_blocks (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  status TEXT NOT NULL,
  command_text TEXT NOT NULL,
  stdout_text TEXT NOT NULL DEFAULT '',
  stderr_text TEXT NOT NULL DEFAULT '',
  exit_code INTEGER,
  started_at DATETIME,
  finished_at DATETIME,
  duration_ms INTEGER,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_bash_blocks_run_created
ON bash_blocks(run_id, created_at ASC);
```

字段规则：

`status` 只允许 `pending/running/success/error`。  
stdout 和 stderr 都允许截断保存，但数据库中应保留截断后的完整展示内容，不要只存摘要。

## 30.6 screenshots

这是最关键的表。  
每一张保存下来的截图都在这里登记。

```sql
CREATE TABLE IF NOT EXISTS screenshots (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  progress_id TEXT,
  raw_path TEXT NOT NULL,
  model_path TEXT NOT NULL,
  annotated_path TEXT NOT NULL,
  desktop_width INTEGER NOT NULL,
  desktop_height INTEGER NOT NULL,
  model_image_width INTEGER NOT NULL,
  model_image_height INTEGER NOT NULL,
  mouse_x INTEGER,
  mouse_y INTEGER,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_screenshots_run_created
ON screenshots(run_id, created_at ASC);

CREATE INDEX IF NOT EXISTS idx_screenshots_progress_id
ON screenshots(progress_id);
```

注意这里 `progress_id` 可以为空。  
原因是有些截图是纯中间检查帧，还未来得及生成进展文案。  
但一旦某个截图对应一个明确进展，后续应该回填其 `progress_id`。

## 30.7 progress_items

每条进展是一段人类可读文本，绑定某张截图。

```sql
CREATE TABLE IF NOT EXISTS progress_items (
  id TEXT PRIMARY KEY,
  run_id TEXT NOT NULL,
  idx INTEGER NOT NULL,
  text_content TEXT NOT NULL,
  screenshot_id TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE,
  FOREIGN KEY (screenshot_id) REFERENCES screenshots(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_progress_items_run_idx
ON progress_items(run_id, idx ASC);

CREATE UNIQUE INDEX IF NOT EXISTS idx_progress_items_run_idx_unique
ON progress_items(run_id, idx);
```

字段规则：

`idx` 从 1 开始递增。  
一条进展必须绑定一张截图。  
这是前端时间线和时间条的主数据来源。

## 30.8 run_events

为了支持 SSE 重连补偿、调试、问题排查，事件必须持久化。  
不要只在内存广播。

```sql
CREATE TABLE IF NOT EXISTS run_events (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  run_id TEXT NOT NULL,
  seq INTEGER NOT NULL,
  event_type TEXT NOT NULL,
  payload_json TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE,
  FOREIGN KEY (run_id) REFERENCES runs(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_run_events_conversation_created
ON run_events(conversation_id, created_at ASC);

CREATE UNIQUE INDEX IF NOT EXISTS idx_run_events_run_seq_unique
ON run_events(run_id, seq);
```

这张表非常有价值：

前端断线后可以补拉。  
后端问题可重放。  
线上调试时能知道某一步到底发过什么事件。

第一版不做无限保留清理也可以，但建议后续加保留策略。

## 30.9 app_kv

用于保存极少量应用级键值，例如自动生成标题任务状态、VNC 健康检查时间等。

```sql
CREATE TABLE IF NOT EXISTS app_kv (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at DATETIME NOT NULL
);
```

第一版用得不多，但预留着比以后硬改方便。

## 31. 文件存储设计

数据库只存元数据。  
真正的附件和截图必须落磁盘。

## 31.1 数据根目录结构

`DATA_DIR` 下固定采用下面结构：

```text
data/
  app.db
  tmp/
  attachments/
    2026/
      03/
        msg_xxx/
          att_xxx_original.png
          att_xxx_thumb.png
  runs/
    conv_xxx/
      run_xxx/
        screenshots/
          shot_001_raw.png
          shot_001_model.png
          shot_001_annotated.png
          shot_002_raw.png
          ...
        logs/
          bash_block_xxx_stdout.txt
          bash_block_xxx_stderr.txt
```

推荐仍以数据库文本字段保存相对路径，例如：

`attachments/2026/03/msg_xxx/att_xxx_original.png`  
`runs/conv_xxx/run_xxx/screenshots/shot_001_annotated.png`

不要把文件都堆在一个目录里。  
消息多了以后目录会很难维护。

## 31.2 附件保存规则

用户上传附件后，服务端保存原文件。  
若是图片，再额外生成缩略图。

命名规则固定：

原文件：`<attachment_id>_original.<ext>`  
缩略图：`<attachment_id>_thumb.webp`

缩略图宽度建议 `480px`，等比缩放。  
图片原图不做二次压缩覆盖，避免用户上传的标注图失真。

## 31.3 截图保存规则

每次截图生成一个 screenshot_id，例如 `shot_xxx`。  
然后固定生成三个文件：

`<screenshot_id>_raw.png`  
`<screenshot_id>_model.png`  
`<screenshot_id>_annotated.png`

文件必须写入 run 专属目录：

`runs/<conversation_id>/<run_id>/screenshots/`

任何地方都不要直接覆盖旧截图。  
截图是时间点快照，必须不可变。

## 31.4 临时目录用途

`TEMP_DIR` 只用于：

上传时的临时 multipart 文件。  
截图处理中间产物。  
模型 base64 临时缓存。  
bash 临时重定向文件。

任务结束后尽量清掉。  
但真正的业务附件和截图一旦入库，必须移动到正式目录。

## 31.5 静态访问 URL 规则

后端需要通过 HTTP 把附件和截图暴露给前端。  
推荐统一挂在：

`/files/*`

例如：

`/files/attachments/2026/03/msg_xxx/att_xxx_original.png`  
`/files/runs/conv_xxx/run_xxx/screenshots/shot_001_annotated.png`

数据库中存的是相对路径。  
API DTO 中返回的是前端可直接访问的 URL。  
URL 拼装规则在 DTO 层处理，不要在数据库层处理。

## 31.6 文件删除策略

第一版不做前台删除会话功能时，可以不实现物理删除。  
但代码层面要预留一个后台清理函数：

删除 conversation 时，先删数据库记录。  
再递归删除 `attachments` 和 `runs/<conversation_id>/` 目录。  
失败时记录错误日志，不影响数据库删除事务提交。
