# EasyAgentCU 开发文档（第 3 部分）

## 14. 前端数据模型设计

这一部分定义前端必须维护的数据结构。程序员不要直接拿后端原始返回对象到处传，必须先归一化，再交给页面渲染层。

前端统一分三类数据：

会话数据。  
消息数据。  
运行数据。

其中“助手消息”不是单纯文本，而是和 run 强绑定的渲染实体。

## 14.1 TypeScript 类型定义

`frontend/lib/types.ts` 直接按下面结构定义。

## 14.2 页面渲染节点模型

虽然前端内部存的是归一化数据，但消息区渲染时不要直接遍历 `messagesById`。  
应该先把会话中的消息整理成一条有序渲染链。

定义一个页面渲染节点：

```ts
export type RenderNode =
  | { type: "user-message"; id: ID; messageId: ID }
  | { type: "assistant-run"; id: ID; assistantMessageId: ID; runId: ID }
```

生成规则很简单：

会话消息按 `createdAt` 升序排列。  
用户消息生成 `user-message` 节点。  
助手消息必须从 `runId` 找到对应 run，生成 `assistant-run` 节点。  
找不到 run 的助手消息不能直接丢弃，要渲染一个错误占位块，并触发后台补拉取。

这样即使 SSE 中途断开，页面仍能用 HTTP 重建渲染结构。

## 15. 前端状态仓库设计

前端使用 Zustand，拆成四个 store，不要做成一个大 store。  
拆分后数据变更边界清晰，也更容易排查 SSE 更新问题。

## 15.1 app-store

负责全局应用态，不存业务实体。

```ts
interface AppStore {

  currentConversationId?: ID
  sidebarCollapsed: boolean
  composerText: string
  pendingAttachments: LocalPendingAttachment[]
  isFullscreenRemoteOpen: boolean
  fullscreenMode: "live" | "image" | null
  fullscreenImageUrl?: string
  setHydrated(v: boolean): void
  setCurrentConversationId(id?: ID): void
  setSidebarCollapsed(v: boolean): void
  setComposerText(text: string): void
  openFullscreenRemote(): void
  openFullscreenImage(url: string): void
  closeFullscreen(): void
  resetComposer(): void
}
```

本地待上传附件类型定义如下：

```ts
export interface LocalPendingAttachment {
  localId: string
  file: File
  kind: "image" | "file"
  previewUrl?: string
}
```

## 15.2 conversation-store

负责会话列表和会话消息索引。

```ts
interface ConversationStore {
  conversationIds: ID[]
  conversationsById: Record<ID, ConversationSummary>
  conversationMessageIds: Record<ID, ID[]>
  messagesById: Record<ID, MessageRecord>
  upsertConversation(conversation: ConversationSummary): void
  upsertConversations(conversations: ConversationSummary[]): void
  upsertMessage(message: MessageRecord): void
  upsertMessages(messages: MessageRecord[]): void
  setConversationMessages(conversationId: ID, messageIds: ID[]): void
  appendMessageToConversation(conversationId: ID, messageId: ID): void
  removeConversation(id: ID): void
  clear(): void
}
```

`conversationMessageIds` 是渲染顺序的主来源。  
不要每次都从 `messagesById` 扫描过滤，那样消息多了以后会反复重算。

## 15.3 run-store

负责运行时相关的复杂实体，包括 bash、进展、截图。

```ts
interface RunStore {
  runIds: ID[]
  runsById: Record<ID, RunRecord>
  bashBlocksById: Record<ID, BashBlockRecord>
  progressItemsById: Record<ID, ProgressItemRecord>
  screenshotsById: Record<ID, ScreenshotRecord>

  upsertRun(run: RunRecord): void
  upsertRuns(runs: RunRecord[]): void

  upsertBashBlock(block: BashBlockRecord): void
  upsertProgressItem(item: ProgressItemRecord): void
  upsertScreenshot(screenshot: ScreenshotRecord): void

  appendBashBlockToRun(runId: ID, bashBlockId: ID): void
  appendProgressToRun(runId: ID, progressId: ID): void
  appendScreenshotToRun(runId: ID, screenshotId: ID): void

  setRunSelectedProgress(runId: ID, progressId?: ID): void
  setRunSelectedScreenshot(runId: ID, screenshotId?: ID): void
  setRunScreenMode(runId: ID, mode: "live" | "history" | "hidden"): void
  mergePhaseText(runId: ID, text: string): void
  setFinalAnswer(runId: ID, text: string): void
  setRunStatus(runId: ID, status: RunStatus): void
  clear(): void
}
```

这里有三个关键点：

第一，`phaseText` 必须支持覆盖式更新，也支持增量拼接。因为后端可能发送 `set`，也可能未来扩展成 `delta`。  
第二，`selectedProgressId` 和 `selectedScreenshotId` 是纯前端 UI 状态，但放进 `RunRecord` 最方便联动。  
第三，历史截图切换和实时画面切换都由 `screenMode` 驱动，不要在组件内部临时维护一套重复状态。

## 15.4 ui-store

## 16. 前端初始化流程

浏览器打开页面后的启动顺序必须固定，不能边想边写。

## 16.1 启动时序

页面加载后按下面顺序执行：

第一步，读取 URL 查询参数中的 `c`。  
第二步，调用 `GET /api/conversations` 拉取会话列表。  
第三步，如果 URL 中存在 `c`，并且该会话存在，则加载该会话。  
第四步，如果 URL 中不存在 `c`，则进入首页空态，不主动创建会话。  
第五步，如果加载了某个会话，则并行执行：

`GET /api/conversations/:id/messages`  
`GET /api/conversations/:id/runs`  
`GET /api/conversations/:id/events/stream`

第六步，把消息和 run 全量归一化后，再渲染消息区。  
第七步，SSE 连接建立后再接受实时更新。

页面一刷新，不需要依赖本地缓存恢复完整数据。可以保留少量 UI 偏好，比如侧栏折叠状态，但消息历史必须以服务端为准。

## 16.2 当前会话切换

点击侧栏会话项时流程如下：

先更新 URL 查询参数。  
再清理当前会话的 SSE 连接。  
再并行拉取新会话的 messages 和 runs。  
拉取完成后建立新的 SSE 连接。  
最后把消息区滚动到底部。

切换会话时不要保留上一个会话的展开进展项和全屏状态。  
这些 UI 状态要重置。

## 17. 用户发送消息的前端事件流

这是第一个真正关键的链路。

## 17.1 发送前校验

点击发送按钮或回车提交时，前端必须按下面顺序判断：

当前是否已有运行中的 run。  
如果有，直接提示“当前任务仍在运行，请先停止当前任务”。第一版不允许并发。  
输入文本是否为空且附件列表是否为空。两者都为空则禁止发送。  
如果附件中存在大于后端限制的文件，直接提示失败。  
如果图片预览 URL 是本地 blob，不必先上传单独附件接口，直接走消息发送的 multipart/form-data 即可。

## 17.2 发送过程

推荐接口为 `POST /api/conversations/:id/messages`，使用 `multipart/form-data`。  
如果当前没有会话，先调用 `POST /api/conversations` 创建会话，再发消息。

前端执行时序如下：

第一步，必要时创建会话。  
第二步，构造本地临时用户消息，立即插入列表。  
第三步，清空输入框和附件。  
第四步，请求服务端发送消息。  
第五步，成功后用服务端返回的正式用户消息替换临时消息。  
第六步，服务端同时会返回新建的 assistant 消息壳和 run 基础信息，前端立即插入。  
第七步，若当前会话 SSE 未连接，则立刻连接。  
第八步，等待 SSE 推送后续运行事件。

临时消息结构需要加一个本地字段：

```ts
type OptimisticStatus = "pending" | "confirmed" | "failed"
```

但这个字段不必进入全局实体模型，可以只在发送 hook 内部维护，再映射到渲染层。

## 17.3 创建助手消息壳子的前端规则

服务端一旦接受消息，前端必须马上让界面里出现一个助手消息容器。  
这条容器默认规则如下：

`status = queued`  
`phaseText = ""`  
`finalAnswer = ""`  
`screenMode = live`  
`liveReady = false`

渲染时看到 `screenMode=live` 且 `liveReady=false`，显示一个带 loading 的屏幕窗口骨架，占住画面空间。  
这样用户不会觉得系统“没反应”。

## 18. 助手运行时的前端渲染逻辑

## 18.1 AssistantRunMessage 组件输入

这个组件只接收 `runId` 和 `assistantMessageId`。  
其余全部数据自己从 store 中取，避免父组件层层透传。

建议写一个 selector：

```ts
interface AssistantRunViewModel {
  run: RunRecord
  bashBlocks: BashBlockRecord[]
  progressItems: ProgressItemRecord[]
  screenshots: ScreenshotRecord[]
  selectedScreenshot?: ScreenshotRecord
  latestProgress?: ProgressItemRecord
}
```

## 18.2 视图推导规则

`selectedScreenshot` 的推导规则必须固定：

如果 `run.screenMode === "live"`，则不选历史图。  
如果 `run.screenMode === "history"` 且 `selectedScreenshotId` 存在，则取它。  
否则默认取最新进展对应截图。  
若连最新进展截图都没有，则取最后一张 screenshot。  
再没有就显示空骨架。

`latestProgress` 取 `progressItems` 的最后一项。  
`bashBlocks`、`progressItems`、`screenshots` 都必须按时间升序排序后再渲染。

## 18.3 显示优先级

这一段是组件最关键的条件分支。

当 `run.finalAnswer` 非空时：

只显示最终回答文本。  
若 `run.needsUserAction` 为真，则在回答下方显示 `HumanActionPanel`。  
绝对不显示阶段文本、bash、进展时间线、屏幕窗口。

当 `run.finalAnswer` 为空时：

若 `run.phaseText` 非空，则显示 `PhaseTextBlock`。  
若 `bashBlocks.length > 0`，则显示一组 `BashToolBlock`。  
若 `run.liveReady` 为真或 `progressItems.length > 0`，则显示 `ProgressTimeline + ScreenWindow`。  
若以上都没有，但 `run.status` 仍是 `queued/running`，则显示默认 loading 文案“正在思考”。

## 19. SSE 事件协议在前端的处理方式

这一部分先从前端角度定义。后端接口文档部分会按这个协议落地。

## 19.1 连接方式

每个已选会话建立一个 SSE 连接：

`GET /api/conversations/:id/events/stream`

服务端持续推送该会话内的消息和 run 事件。  
前端只需要一条连接，不要为每个 run 单独开 SSE。

使用 `EventSource` 即可，不要自己封装 fetch streaming，SSE 更稳。

## 19.2 事件基础格式

所有 SSE data 必须是统一 JSON 对象：

```json
{
  "event_id": "evt_xxx",
  "conversation_id": "conv_xxx",
  "run_id": "run_xxx",
  "seq": 12,
  "type": "run.progress.created",
  "data": {}
}
```

前端收到事件后不要直接假设顺序绝对正确，而要做两层保护：

第一层，`event_id` 去重。  
第二层，对每个 `run_id` 记录最后处理的 `seq`。若收到的 `seq` 小于等于已处理序号，则丢弃。若出现跳号，比如当前是 12，收到 15，说明中间事件丢失，要立即触发该 run 的全量补拉接口。

这一步非常重要。只靠 SSE 连接本身，页面在弱网下很容易丢中间状态。

## 19.3 前端需要支持的 SSE 事件类型

第一版固定支持下面这些事件：

```ts
type SSEEventType =
  | "conversation.updated"
  | "message.created"
  | "run.created"
  | "run.status.changed"
  | "run.live.ready"
  | "run.phase_text.set"
  | "run.bash.started"
  | "run.bash.updated"
  | "run.bash.completed"
  | "run.progress.created"
  | "run.screenshot.created"
  | "run.answer.final"
  | "run.waiting_user"
  | "run.failed"
  | "run.cancelled"
  | "heartbeat"
```

这里不用做过细的“delta token”事件。  
第一版的设计目标是可靠，不是炫技。

## 19.4 每类事件的前端应用逻辑

### `conversation.updated`

更新侧栏会话标题、更新时间、预览文本、状态。

### `message.created`

把正式消息插入会话消息列表。  
如果之前存在本地临时消息，需要根据客户端生成的 `client_message_id` 或内容哈希进行替换，避免重复渲染两条。

### `run.created`

创建或更新 `RunRecord`。  
同时确保对应 assistant message 已存在。  
若不存在，先插入一个最小 assistant message 占位。

### `run.status.changed`

只更新 `run.status` 和 `updatedAt`。  
若状态变成 `completed/failed/cancelled` 且没有最终回答，则保留已有块并显示错误或完成提示。

### `run.live.ready`

把 `liveReady` 改为 `true`，记录 `liveSessionId`。  
若当前 run 还没有 `finalAnswer`，并且用户没有手动切到历史模式，则把 `screenMode` 设为 `live`。

### `run.phase_text.set`

覆盖 `phaseText`。  
这里采用覆盖，不做拼接。因为阶段回答是完整的一段，不是 token 流。

### `run.bash.started`

创建 bash block，状态为 `running`。  
若当前已经存在同 ID 的 block，则更新为运行中。

### `run.bash.updated`

更新 stdout、stderr、duration。  
输出区域是否展开由 UI store 决定，不受事件影响。

### `run.bash.completed`

更新状态为 `success` 或 `error`，写入退出码和结束时间。

### `run.progress.created`

插入新的进展项。  
若当前 run 处于 `live` 模式，不改变它。  
若当前 run 处于 `history` 模式且用户没有手动锁定到旧进展，仍可自动跳到最新截图；  
但为了交互稳定，建议只在用户没有明确选中过历史进展时自动切换。实现上可以用 `selectedProgressId` 是否为空判断。

### `run.screenshot.created`

插入新的截图记录。  
如果该截图绑定了新的 progress，前端下一轮渲染时自然能显示出来。  
若当前 run 还没有 `selectedScreenshotId`，就把它设成最新截图。

### `run.answer.final`

这是状态切换的分水岭。  
收到后立即：

设置 `finalAnswer`。  
设置 `screenMode = hidden`。  
清除 `selectedProgressId` 和 `selectedScreenshotId`。  
保留原数据，但不再显示 phase/bash/progress/screen。  
之后滚动消息区到底部。

### `run.waiting_user`

把 `status` 改为 `waiting_user`。  
设置 `needsUserAction = true`。  
设置 `screenMode = hidden`。  
这一事件通常应和 `run.answer.final` 同时或先后到达。  
前端必须允许它们任意顺序到达。实现上采用幂等合并，不依赖先后顺序。

### `run.failed`

状态改为 `failed`，写入 `errorMessage`。  
若还没有 `finalAnswer`，界面显示一个错误回答块，例如“任务执行失败：xxx”。

### `run.cancelled`

状态改为 `cancelled`。  
若最后一条 run 处于当前会话且还在显示实时屏幕，则切成隐藏，并提示“已停止”。

## 20. SSE 断线重连与补偿机制

这是可靠性要求，不是可选项。

## 20.1 重连策略

浏览器 SSE 断开后，前端要自动重连。  
第一次重连等待 `1s`，第二次 `2s`，第三次 `4s`，上限 `15s`。  
当前会话切换时，旧连接必须立即关闭，不参与重连。

重连成功后，不能假设本地状态一定正确，必须做一次补偿同步。

## 20.2 补偿同步

重连成功后，立即调用：

`GET /api/conversations/:id/runs?since=<last_known_update_time>`  
或者更简单，第一版直接重新拉整个会话的 runs 列表。

若发现最后一个 run 的 `updatedAt` 晚于本地，则用服务端数据覆盖本地。  
覆盖原则不是清空重建，而是按主键 upsert，避免界面闪烁。

## 20.3 seq 跳号处理

每个 run 维护一个 `lastSeq`。  
当收到的事件序号不是 `lastSeq + 1` 时：

先把该事件放入短暂缓冲区。  
立即请求 `GET /api/runs/:id` 全量补拉。  
补拉成功后，以服务端快照覆盖该 run 相关所有实体。  
再清空缓冲区。

第一版也可以更简单：一旦跳号，直接忽略当前事件并全量补拉该 run。  
不要尝试自己在前端做复杂事件重排。

## 21. 前端 HTTP 请求封装规范

`frontend/lib/api.ts` 必须封装统一请求层，不允许组件直接 `fetch("/api/...")`。

推荐封装：

```ts
export async function apiFetch<T>(
  path: string,
  init?: RequestInit
): Promise<T> {
  const res = await fetch(path, {
    ...init,
    headers: {
      ...(init?.headers ?? {}),
    },
    credentials: "same-origin",
  })

  if (!res.ok) {
    const text = await res.text().catch(() => "")
    throw new Error(text || `HTTP ${res.status}`)
  }

  return res.json() as Promise<T>
}
```

对于 `multipart/form-data` 请求，不要手动设置 `Content-Type`，交给浏览器。  
错误统一交给上层 hook 捕获，再用 `sonner` 提示。

## 22. 前端业务 Hook 设计

组件不要直接拼装复杂请求逻辑，必须抽成 hook。

推荐定义下面这些 hook：

`useConversations()`  
负责会话列表加载、选择会话、URL 同步。

`useConversationDetail(conversationId)`  
负责该会话 messages 和 runs 的初始化加载。

`useConversationStream(conversationId)`  
负责 SSE 建连、断线重连、事件分发。

`useSendMessage(conversationId)`  
负责新建会话、发送用户消息、插入临时消息、失败回滚。

`useStopRun(conversationId)`  
负责停止当前运行中的 run。

`useAssistantRunViewModel(runId)`  
负责把 store 中的 run/bahs/progress/screenshot 组合成视图模型。

`useNoVnc(runId)`  
负责实时桌面连接地址、连接状态、全屏接管逻辑。

## 23. 前端交互细节要求

## 23.1 自动滚动策略

消息区使用“聪明跟底部”逻辑，不要粗暴每次新消息都强制滚到底。  
规则如下：

用户当前在底部附近时，新增消息或事件更新后自动滚到底。  
用户已经手动向上滚动查看历史时，不要打断。  
此时右下角显示一个“回到底部”按钮。  
最终回答出现时，无论当前是否在底部，都建议轻微提示，但不强制抢滚动。

可以借用 prompt-kit `ChatContainer` 的思路，但本项目建议自行封装，以便兼容固定底部输入区。

## 23.2 历史进展和实时画面的切换

默认 run 开始后，屏幕窗口优先显示实时模式。  
用户点击某个进展后，切到历史模式。  
此时若后端又推送了新进展，不应强制把用户拉回实时模式。  
只有用户点击底部“直播/实时”节点，才重新回到实时模式。

所以前端必须区分两种行为：

系统自动更新最新数据。  
用户主动锁定历史画面。

推荐在 run 上增加一个临时前端字段：

```ts
isHistoryPinnedByUser?: boolean
```

但第一版为了简化，也可以通过 `screenMode === "history"` 且 `selectedProgressId` 非空来推断。

## 23.3 停止按钮行为

底部 Composer 在会话存在运行中任务时，右下角按钮从发送图标切换为停止图标。  
点击停止后：

按钮立即进入 loading。  
调用 `POST /api/runs/:id/cancel`。  
收到成功后，等 SSE 把 run 状态改成 `cancelled`。  
若 5 秒内没收到 SSE 更新，主动补拉该 run。

## 23.4 “我已经操作”按钮行为

用户点这个按钮时，前端不要调用特殊接口。  
统一走“发送一条用户消息”的普通消息接口，内容固定写死为 `我已经操作`。  
这样整个会话历史是完整的，也利于模型继续理解上下文。

## 25. 前端必须避免的实现错误

这一段是强约束。

不要把阶段文本、bash、进展、屏幕窗口分别建成多条 assistant 消息。  
正确做法是：它们都是同一个 assistant run 消息里的不同块。

不要把当前屏幕截图 URL 直接写死在组件本地 state 里。  
必须从 run 的 `selectedScreenshotId` 推导。否则进展点击联动会乱。

不要把实时 noVNC 画布作为模型输入图来源。  
模型输入图只能来自后端主动采集并保存的 screenshot 记录。

不要在前端推断 run 是否完成，必须以后端事件和接口返回为准。  
只要 `finalAnswer` 或 `status=completed/failed/cancelled/waiting_user` 没明确到达，就视为未结束。

不要把 SSE 事件直接写成“收到什么就 setState 什么”的散乱逻辑。  
必须统一经过事件分发器，按 event type 更新对应 store。
