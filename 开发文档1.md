## 1. 项目目标与边界

EasyAgentCU 是一个运行在 Deepin V25 X11 虚拟机中的桌面智能体。用户在宿主机浏览器中访问它的 Web 界面，输入任务后，系统调用支持视觉的多模态大模型，让模型通过截图理解当前电脑画面，再通过工具调用操作这台电脑，最终完成任务。

本项目第一版必须满足下面这些硬性边界：

前端使用 Next.js、TailwindCSS、radix-ui、shadcn/ui、prompt-kit。  
后端使用 Golang。  
模型接口使用 OpenAI 兼容的 Chat Completions API，而不是 OpenAI Responses API。  
项目最终交付形态是一个 Go 可执行二进制，启动后监听 `0.0.0.0`，前端静态资源由该二进制直接提供。  
运行环境固定为 Deepin V25 + X11，不支持 Wayland。  
桌面控制默认不做沙箱限制，bash 可直接调用系统命令。  
系统默认单用户部署，不做登录、权限、多租户。  
聊天记录、截图、附件、运行日志都保存在本机磁盘。  
每个会话同一时刻只允许一个运行中的 Agent 任务。

这里明确一个关键实现决策：前端虽然用 Next.js 开发，但发布时不运行 Node 服务，而是导出为纯静态站点，再嵌入 Go 二进制。这样既满足技术栈要求，也满足“通过可执行二进制运行”的需求。

## 2. 最终技术方案

## 2.1 前端方案

前端采用 Next.js App Router 开发，但只做静态导出能力允许的客户端页面，不依赖 Next Server API。所有数据请求都直接打到 Go 后端。

前端依赖固定如下：

| 类别        | 选型                               |
| ----------- | ---------------------------------- |
| 框架        | Next.js 15 + React 19 + TypeScript |
| 样式        | TailwindCSS                        |
| 基础组件    | radix-ui + shadcn/ui               |
| AI 界面组件 | prompt-kit                         |
| 图标        | lucide-react                       |
| 状态管理    | zustand                            |
| 表单校验    | zod                                |
| 提示消息    | sonner                             |
| VNC 客户端  | noVNC                              |

前端不使用 Next API Route，不使用服务端渲染业务逻辑，不使用服务端 actions。页面统一由客户端加载后通过 REST + SSE 获取数据。

## 2.2 后端方案

后端采用 Go 1.24+。

后端依赖固定如下：

| 类别      | 选型                                 |
| --------- | ------------------------------------ |
| HTTP 路由 | go-chi/chi/v5                        |
| 跨域      | go-chi/cors                          |
| 日志      | rs/zerolog                           |
| 数据库    | SQLite + modernc.org/sqlite          |
| ID        | google/uuid                          |
| 重试      | cenkalti/backoff/v4                  |
| 图片处理  | disintegration/imaging + fogleman/gg |
| 配置      | 直接读取环境变量，自定义 config 包   |
| 并发控制  | 标准库 context / sync / channels     |

后端直接调用系统命令完成桌面控制与截图，不引入复杂桌面自动化框架。固定依赖的系统命令如下：

`xdotool`：鼠标、键盘、窗口激活  
`xrandr`：获取屏幕分辨率与缩放后逻辑尺寸  
`scrot` 或 `import`：截图  
`x11vnc`：把 X11 桌面暴露成 VNC 服务  
`websockify`：把 VNC TCP 转成 WebSocket，供 noVNC 使用  
`bash`：模型自由执行命令  
`ps`, `pgrep`, `pkill`：进程检查与清理

推荐强制使用 `scrot` 做截图，`import` 作为备用。文档后续统一按 `scrot` 编写。

## 2.3 模型交互方案

模型使用支持图片输入与工具调用的 OpenAI 兼容 Chat Completions API。第一版按“非流式模型请求 + 前端事件流”实现。也就是：

模型接口本身不依赖 token 级流式输出。  
后端在任务执行过程中，把阶段文本、工具调用、进展、截图、最终答复通过 SSE 实时推给前端。  
这样可以避开很多国产 OpenAI 兼容接口在“流式工具调用”上的不稳定行为，可靠性更高。

模型请求格式固定采用：

1. 用户文本消息。
2. 需要看图时，追加一个包含说明文字和 `image_url(base64)` 的用户消息。
3. 模型返回 `tool_calls`。
4. 后端执行工具。
5. 以 `role: tool` 回传工具执行结果文本。
6. 再按需要追加新的截图用户消息。
7. 循环直到模型输出最终文本。

## 3. 运行拓扑

系统运行时的真实拓扑不是“浏览器直接连模型”，而是下面这条链路：

```text
宿主机浏览器
   │
   ├── HTTP / SSE ───────────────► EasyAgentCU Go 服务
   │                                 │
   │                                 ├── SQLite / 文件系统
   │                                 ├── OpenAI 兼容模型 API
   │                                 ├── xdotool / xrandr / scrot / bash
   │                                 └── 任务调度与 Agent 主循环
   │
   └── noVNC(WebSocket) ─────────► websockify ─► x11vnc ─► X11 桌面(:0)
```

这里有两个并行的画面来源：

第一种是“实时画面”，由 noVNC 显示。它用于用户观看与接管操作。  
第二种是“截图画面”，由后端主动采集、压缩、加标注、持久化。它用于模型分析，也用于会话中的历史进展回放。

这两种画面必须分开实现，不能混为一体。原因很简单：

实时画面适合交互，但不适合作为稳定的模型输入。  
模型分析需要明确的一帧图片、明确的分辨率、明确的时间点、明确的鼠标坐标。  
而历史回放也必须基于已保存的截图帧，否则进度条无法回看。

## 4. 关键实现决策

## 4.1 只做一个页面路由

前端静态导出后，为了避免动态路由处理复杂化，整个应用只保留一个页面入口 `/`。当前会话 ID 通过查询参数承载，例如：

`/`
`/?c=conversation_xxx`

这样静态导出最稳，Go 服务只需要提供一个前端入口和一组 API。

## 4.2 对话与运行分离

“会话”是长期存在的数据。  
“运行”是一次用户发送消息后触发的 Agent 执行过程。

一个会话里会有多轮消息。  
每次用户发送一条新消息，都会创建一个新的 run。  
run 生命周期只有四种状态：

`queued`  
`running`  
`waiting_user`  
`completed`  
`failed`  
`cancelled`

助手消息的可视化内容，本质上都是某个 run 的渲染结果，而不是一次性纯文本。

## 4.3 屏幕窗口必须先显示，再开始推理

收到用户消息后，前端要立即插入一条新的助手消息壳子。这个壳子默认先显示“实时屏幕窗口”，哪怕此时模型还没拿到截图，哪怕后端还在准备第一轮推理。

这条规则必须在前后端两侧都落实：

前端提交消息成功后，立刻插入本地临时 assistant-run 消息。  
后端创建 run 后马上发出 `run.created` 和 `screen.live_ready` 事件。  
前端一旦收到 `screen.live_ready`，立刻让对应消息进入“实时画面模式”。

## 4.4 模型自由调用工具，但工具面固定

不做工作流编排器，不做固定流程树，不做“先浏览器后 bash”这类强约束。  
但必须给模型一个稳定的工具面。第一版工具面固定为：

屏幕观察相关工具  
鼠标键盘相关工具  
等待与窗口聚焦相关工具  
bash 工具

后续再扩展，但第一版不要混入 Playwright、浏览器 DOM、MCP、RPA 等第二套控制体系。否则上下文和错误恢复会变乱。

## 5. 屏幕截图与坐标体系

这是整个项目最关键、最容易做错的部分。必须一次设计对。

## 5.1 三种尺寸同时存在

系统里同时存在三种坐标尺寸：

第一种是桌面真实逻辑尺寸，也就是 X11 当前桌面的宽高，例如 `1920x1080`。  
第二种是模型输入截图尺寸，也就是为了节约 token 和网络传输而缩放后的图片尺寸，例如 `1600x900`。  
第三种是前端展示尺寸，也就是浏览器里屏幕窗口实际渲染成的 CSS 尺寸，例如 `896x504`。

模型返回的坐标，必须明确约定为“模型输入截图尺寸空间”的坐标，而不是前端展示尺寸，也不是浏览器像素坐标。

## 5.2 坐标映射规则

每次截图时，后端必须记录下面这些元数据：

`desktop_width`  
`desktop_height`  
`model_image_width`  
`model_image_height`  
`scale_x = desktop_width / model_image_width`  
`scale_y = desktop_height / model_image_height`

模型返回 `(x, y)` 后，执行工具前统一映射成：

`desktop_x = round(x * scale_x)`  
`desktop_y = round(y * scale_y)`

模型永远不需要知道浏览器里展示了多大，也不需要知道前端缩放比例。  
前端点历史进展时，只切换图片与时间点，不参与工具执行坐标换算。

## 5.3 截图文件必须保存三份

每个进展节点保存三份文件：

第一份是原始截图 `raw.png`，分辨率等于桌面逻辑尺寸。  
第二份是模型输入图 `model.png`，缩放后、可加轻度压缩、用于 base64 发给模型。  
第三份是带标注的展示图 `annotated.png`，在图上画出鼠标位置圈、箭头和“鼠标指针”四字，用于模型观察与前端历史回放。

实时画面不用这三份文件，它直接走 noVNC。

## 5.4 鼠标鲁棒性优化必须落地到实现

不是只写在提示词里，而是要形成固定机制。

每次提供给模型的截图，后端都同时附带：

1. 一张带鼠标标注的截图。  
2. 一段文本元数据，明确写出当前鼠标坐标、桌面分辨率、模型截图分辨率。

模型提示词中再强制要求：  
凡是点击小控件、输入框、关闭按钮、页签、复选框、下拉箭头等高精度目标，必须先移动，再看图确认，再点击。不能盲点。

## 6. 页面骨架与原型还原要求

这一部分先定义整体外观骨架，组件细节在下一部分展开。

整个应用是深色主题。背景不是纯黑，而是偏暖的深灰黑，主背景建议从 `#161616` 到 `#1d1d1d` 做极轻微纵向渐变。所有边框都很弱，接近 `rgba(255,255,255,0.08)`，不能出现亮色边框。页面整体视觉要接近你给的原型：安静、厚重、圆角大、留白大。

页面分成左右两栏。

左栏是会话侧边栏，展开状态宽度固定 `384px`。背景略深于主内容区，顶部有一个窄工具栏，里面只有折叠按钮。折叠后宽度变成 `64px`，只保留图标。左栏内部从上到下依次是：新聊天按钮、搜索聊天按钮、“你的聊天”标题、会话列表。会话项是圆角矩形，当前激活项底色比普通项亮一层，文字白色，其余项是低对比度灰白。整个左栏自身可滚动。

右栏是主内容区。顶部固定一个高度约 `72px` 的头部，只显示产品名 `EasyAgentCU`，左对齐，字重中等，字号大约 `24px` 到 `28px`。头部底部有一条很淡的分隔线。头部以下是消息滚动区和底部输入区。

空会话状态要还原原型里的居中首页效果：  
中间是一块大圆角输入框，宽度大约 `960px`，高度大约 `160px` 到 `180px`。输入框下方是一排任务分类胶囊按钮，再下面是一组推荐任务列表项。整体垂直居中略偏上。这个状态下右侧没有任何消息，只显示首页输入区和推荐区。

有对话时，消息区依旧居中显示，消息列最大宽度固定 `960px`。  
用户消息是右对齐的气泡，背景比页面稍亮，圆角很大，靠右悬浮。  
助手消息不是传统单气泡，而是一组“消息段落块”的容器，内部可出现阶段文本、bash 工具块、进展列表、屏幕窗口、最终回答。  
底部输入框固定吸附在页面底部中央，不跟随消息一起滚走。输入框下方有一句小号免责声明：`EasyAgentCU 也可能会犯错。请核查重要信息。`
