# EasyAgentCU 开发文档

## 1. 项目定位与最终交付形态

EasyAgentCU 是一个运行在 Deepin V25 虚拟机内的单桌面 Computer Use 智能体。用户从宿主机浏览器访问虚拟机中的 Web 服务，和智能体对话，智能体通过大模型理解屏幕，再调用鼠标、键盘、bash 来操作桌面。

最终交付物不是“前后端分别启动”的开发态，而是“一个 Go 可执行二进制 + 一个数据目录”。前端使用 Next.js 开发，但生产环境导出为静态资源并嵌入 Go 二进制，由 Go 统一对外提供 HTTP 服务，监听 `0.0.0.0`。

这个项目是单用户、单桌面、单活动任务系统。聊天历史可以有很多，但任意时刻只允许一个活动任务真正控制桌面。

---

## 2. 总体技术方案

### 2.1 架构决策

采用下面这套结构，落地最稳：

| 层           | 技术                                                              | 说明                                                           |
| ------------ | ----------------------------------------------------------------- | -------------------------------------------------------------- |
| 前端         | Next.js + React + TailwindCSS + shadcn/ui + Radix UI + prompt-kit | 做单页应用，静态导出                                           |
| 后端         | Go                                                                | 对外 HTTP API、SSE、VNC WebSocket 代理、Agent 运行器、数据存储 |
| 数据库       | SQLite                                                            | 本地持久化聊天、消息、截图索引、运行状态                       |
| 文件存储     | 本地磁盘                                                          | 截图、附件、bash 输出、日志                                    |
| 屏幕捕获     | X11 + `import`/`scrot` + `xdotool`                                | DeepinV25 是 X11，直接走系统工具最稳                           |
| 用户远程操作 | `x11vnc` + 浏览器 noVNC 客户端                                    | 用户全屏接管桌面时使用                                         |
| LLM 接口     | OpenAI Chat Completions 兼容协议                                  | 兼容 Qwen3.5-397B-A17B 及同类模型                              |

### 2.2 为什么这样做

前端不单独跑 Node 服务，而是静态导出后嵌入 Go，这样才能满足“通过一个可执行二进制运行”的要求。  
屏幕显示分成两路：一路是给前端看的实时预览图；一路是给模型看的高质量截图。两者不要混用。  
用户接管桌面时不用自己造远程桌面协议，直接使用 noVNC 方案，Go 只做 WebSocket 到本地 VNC TCP 的桥接。  
模型不使用 OpenAI Responses 的 computer-use-preview，而是使用 Chat Completions + 自定义工具调用循环，兼容国产 OpenAI 风格模型。

### 2.3 实际运行流

1. 用户发送消息。
2. 前端立刻插入一个“助手运行中”的消息块，并显示实时屏幕窗口骨架。
3. 后端抓取当前桌面截图，构造多模态消息发给模型。
4. 模型返回一段进展文本，以及一个工具调用。
5. 后端执行工具，执行后重新截图，再把工具结果和新截图继续发给模型。
6. 循环执行，直到模型输出最终回答，或者调用“请求用户操作”工具。
7. 如果需要用户操作，前端显示“我已经操作”按钮，用户打开全屏桌面操作，完成后点击按钮，前端自动发一条“我已经操作”。

## 3. 运行环境与部署要求

## 3.1 虚拟机环境

固定目标环境：

| 项       | 要求                           |
| -------- | ------------------------------ |
| 系统     | Deepin V25                     |
| 图形系统 | X11                            |
| 运行用户 | 普通桌面用户，不要用 root      |
| 启动方式 | 用户登录桌面后自启动           |
| 访问方式 | 宿主机浏览器访问虚拟机 IP:端口 |

这个程序必须运行在拥有桌面会话权限的用户下，因为它需要访问 `DISPLAY=:0`，调用鼠标键盘，抓取屏幕。

## 3.2 系统依赖

需要安装这些系统包：

| 包            | 用途                                 |
| ------------- | ------------------------------------ |
| `x11vnc`      | 浏览器全屏远程操作                   |
| `xdotool`     | 鼠标、键盘、窗口信息                 |
| `imagemagick` | 屏幕抓图，推荐用 `import`            |
| `xclip`       | 通过剪贴板粘贴中文和长文本           |
| `sqlite3`     | 仅调试方便，程序本身使用 SQLite 驱动 |
| `wmctrl`      | 可选，窗口控制                       |
| `xrandr`      | 获取屏幕尺寸                         |
| `python3`     | 仅保留，非必需                       |
| `bash`        | bash 工具运行环境                    |

安装命令直接写成部署脚本：

```bash
sudo apt update
sudo apt install -y x11vnc xdotool imagemagick xclip wmctrl xrandr sqlite3
```

## 3.3 端口规划

| 端口   | 用途                 | 对外                   |
| ------ | -------------------- | ---------------------- |
| `8080` | EasyAgentCU Web 服务 | 是                     |
| `5900` | x11vnc 本地 TCP      | 否，只监听 `127.0.0.1` |

## 3.4 自启动

推荐使用桌面自启动，而不是 system service。原因很简单：它必须依附已登录的图形会话。

`~/.config/autostart/easyagentcu.desktop`

```ini
[Desktop Entry]
Type=Application
Name=EasyAgentCU
Exec=/opt/easyagentcu/easyagentcu --config /opt/easyagentcu/config.yaml
X-GNOME-Autostart-enabled=true
Terminal=false
```

## 5. 页面与原型还原说明

这个项目前端只有一个实际页面壳子 `/`，但内部有多种界面状态。不要拆成多个 URL 页面；聊天切换用 query 参数 `?conversationId=` 即可，这样才能稳定支持静态导出。

## 5.1 整体壳子布局

整个页面是深色主题，背景不是纯黑，而是非常深的灰黑色，接近 `#18181b`。  
左侧是固定宽度侧边栏，右侧是主聊天区。整体和原型一致，视觉上像 ChatGPT / Claude 的深色工作台，但更强调“正在操作电脑”的过程展示。

### 侧边栏

侧边栏宽度 320px，深色，比主区域更暗一些。顶部有一个收起按钮。  
按钮下面依次是：

1. “新聊天”入口，带笔形图标。
2. “搜索聊天”入口，带放大镜图标。
3. 一个浅色小标题“你的聊天”。
4. 聊天会话列表，按更新时间倒序排列。

当前选中的会话项是一个圆角矩形高亮块，背景比其他项亮一层。  
列表项文字不大，中文优先，不显示复杂副信息。  
侧边栏底部不做复杂固定区，直接滚动。

### 主区域

主区域顶部只有一个左对齐标题 `EasyAgentCU`，下面一条很细的分隔线。  
消息列表在中间区域垂直滚动。  
底部是一个悬浮感很强的输入框，大圆角、半透明深灰背景、轻微描边，固定在底部中央。  
最底部有一行免责声明：`EasyAgentCU 也可能会犯错。请核查重要信息。`

## 5.2 空状态页面

对应最后一张原型图。

页面中部偏下是一个超大输入框卡片，宽约 720px，高约 160px，圆角很大。  
输入框左下角有一个加号按钮，用于上传文件和图片。  
右下角是圆形发送按钮，默认灰色，输入有内容时高亮。  
输入框 placeholder 是“描述任务”。

输入框下方有一排建议分类按钮，圆角胶囊样式，文字例如“建议 / 报告 / 操作 / 电子表格 / 演示文稿”。  
再下面是一组建议任务列表，每行左边是小图标和任务文字，右边是圆形播放按钮。点击播放按钮会把建议文本直接发送。

空状态时消息区域不显示任何历史消息。

## 5.3 普通会话页面

对应第二、第三、第五、第七张原型图。

### 用户消息

用户消息右对齐，最大宽度 640px。  
文本气泡是深灰色胶囊矩形，圆角大，边框非常弱。  
如果用户消息带图片，图片显示在文字气泡上方，宽度不超过 420px，圆角 16px，有浅描边。  
如果用户消息带文件但不是图片，显示文件卡片，包含文件名、大小、图标。

### 助手消息

助手消息左对齐，不显示头像。  
助手消息不是单一气泡，而是一个“容器块”，里面按状态显示不同子块。

子块顺序固定如下：

1. 阶段回答文本
2. bash 工具调用块
3. 进展文本 + 屏幕画面窗口
4. 最终回答文本
5. 等待用户操作按钮

并且有一个非常关键的显示规则：

- 最终回答文本一旦出现，阶段回答文本和 bash 工具块必须隐藏。
- 普通完成态下，最终回答文本出现后，进展文本和屏幕画面窗口也隐藏。
- 只有“等待用户操作”这一种特殊状态下，最终回答文本下方继续保留一个可点击的实时屏幕窗口，用于进入全屏远程操作。

### 阶段回答文本

它是助手收到新要求后的第一句确认，比如“收到，我先打开百度热搜。”  
这个块没有气泡感，像左侧的自然段文本，字号比普通正文略大一些，颜色偏白。  
当最终回答出现后，这个块直接不渲染。

### bash 工具调用块

这个块用于展示 bash 调用，只有 `bash_exec` 工具会进入这个块。  
视觉上是一个圆角深色卡片，标题行显示终端图标、`bash`、执行状态。  
正文默认显示命令本身，下面可折叠显示 stdout/stderr。  
stdout/stderr 区域使用等宽字体，最大高度 240px，超出滚动。  
当最终回答出现后，这个块直接不渲染。

### 进展文本

这个块在屏幕窗口上方。默认折叠，仅显示最新一条进展，右侧有展开箭头。  
文案例如：“等待页面加载…”、“我正在点击”、“我已打开百度热搜页，正在读取前十项标题”。

点击后展开成一个垂直步骤列表，列表中的每一步：

- 有时间顺序
- 左侧有竖线连接
- 每一项是可点击的文本
- 当前选中的进展高亮
- 点击某项后，下方屏幕窗口跳转到该项对应截图

展开态视觉参考第八张原型图：一个深色滚动容器，内部是纵向步骤流。

### 屏幕画面窗口

对应第四张原型图。

它是一个大圆角“窗口卡片”，外框有轻微阴影。  
里面显示屏幕截图。  
截图下方是一条时间线进度条。时间线左到右是各个进展点，最后一个固定点叫“直播”。  
用户可以：

- 拖动进度条
- 点击某个进展点
- 点击“直播”点回到实时画面

当选择历史进展点时，显示对应截图，不再自动刷新。  
当选择“直播”时，窗口每秒刷新一次实时画面。  
当用户通过展开的进展列表点击某条进展时，时间线必须同步移动到对应点。

这个窗口本身可点击。  
普通运行态点击窗口时，打开“仅查看”的大图预览即可。  
等待用户操作态点击窗口时，必须打开全屏可交互远程桌面。

## 5.4 助手思考态

对应第一张原型图。

用户发出消息后，如果首帧实时画面和模型第一条回复都还没返回，助手区域要先显示一个简洁占位：

- 左侧有“正在思考”
- 右上保留刚发送的用户消息
- 下方实时窗口先显示骨架屏或“正在连接桌面…”

这个状态只存在很短时间，任何一个实时屏幕帧回来后就替换成实时窗口。

---

## 5.5 等待用户操作态

这是特殊态，不能和普通完成态混为一谈。

当模型判断需要用户接管时，助手消息显示：

1. 最终回答文本，告诉用户要做什么。
2. 一个主按钮：`我已经操作`
3. 一个仍然可见的实时屏幕窗口

用户点击屏幕窗口后，全屏进入远程桌面。  
全屏顶部有一条深色工具栏，左侧显示“手动操作模式”，右侧有关闭按钮。  
底部固定显示一条提示：`完成后点击“我已经操作”继续。`

用户点击“我已经操作”后，前端不要只是改状态，而是必须自动发出一条新的用户消息：`我已经操作`。这个消息进入会话历史，后端据此恢复 Agent。

## 6.4 前端组件职责

### `app-shell.tsx`
整个页面壳子。左边 sidebar，右边主区。负责 query 参数里的 `conversationId`。

### `sidebar.tsx`
显示新聊天、搜索、会话列表。  
点击新聊天时调用创建会话 API，并切换到新会话。  
会话项显示标题，当前高亮。  
会话列表支持滚动。

### `search-dialog.tsx`
shadcn `Dialog + Command` 实现。  
输入关键字后请求后端搜索 API。  
回车打开对应会话。

### `conversation-pane.tsx`
根据会话状态渲染空状态或消息流。  
底部固定 composer。  
顶部固定标题。

### `assistant-turn.tsx`
整个助手消息容器。根据 `status` 决定显示哪些子块。核心逻辑写死，不要散落在多个地方。

伪代码：

```ts
if (turn.status === 'waiting_user') {
  showFinalText()
  showWaitingButton()
  showLiveScreenWindow()
} else if (turn.finalText) {
  showFinalText()
} else {
  showPhaseText()
  showBashCalls()
  showProgressHeader()
  showScreenWindow()
}
```

### `progress-header.tsx`
默认仅显示最新进展。  
点击展开后展示所有进展步骤列表。  
点击步骤项时更新 `selectedProgressId`。

### `screen-window.tsx`
显示截图主体和底部时间线。  
如果 `selectedProgressId === 'live'`，调用 `use-live-screen.ts` 每秒刷新图片。  
否则显示历史截图 URL。  
点击窗口触发查看大图或进入全屏远程桌面。

### `timeline-scrubber.tsx`
基于 Radix `Slider` 或 shadcn `Slider` 改造。  
要支持“带离散点”的时间轴，而不是普通连续滑块。  
最后一个点固定显示文本 `直播`。

### `fullscreen-remote-desktop.tsx`
封装 noVNC。  
等待用户操作时为交互模式，其他情况可只读。  
组件挂载时连接 `/api/vnc/ws`，卸载时断开。

### `composer.tsx`
输入区。  
支持：

- 输入文本
- 粘贴图片
- 粘贴文件
- 拖拽文件
- 选择文件
- 发送
- 停止

交互规则：

- 当前无运行任务时，右下是发送按钮。
- 当前有运行任务且输入框为空时，右下是停止按钮。
- 当前有运行任务但用户已经输入了新内容时，右下恢复为发送按钮；点击发送时会自动中断旧任务并发新消息。

这条规则必须实现，因为用户可能临时改需求。

## 7.3 核心服务职责

### `run_service`
启动、停止、恢复 Agent 运行。  
维护全局桌面锁。  
一旦某个会话处于运行中，其他会话启动任务会返回 409。

### `agent/runner`
真正的循环执行器。  
一条用户消息启动一次 run。  
每个 run 绑定一个 assistant turn。

### `desktop`
封装所有桌面行为：

- 抓屏
- 获取分辨率
- 获取鼠标坐标
- 鼠标移动
- 点击
- 拖拽
- 粘贴文本
- 按键
- 滚动
- 获取当前窗口标题

### `bash/manager`
按会话维度维护持久 bash 会话。  
每个会话一个 shell，不要每次 `exec.Command` 新开 bash，否则 `cd` 等状态会丢。

### `vnc/manager`
确保本地 `x11vnc` 已启动。  
只允许监听 `127.0.0.1:5900`。  
Go 负责把浏览器 WebSocket 数据转发到这个 TCP。

### `POST /api/conversations/{id}/messages`

用途：发送用户消息，并启动或恢复 Agent。  
请求使用 `multipart/form-data`，这样图片、文件、文本一次完成。

表单字段：

| 字段                | 类型   | 必填 | 说明                  |
| ------------------- | ------ | ---- | --------------------- |
| `client_message_id` | string | 是   | 前端生成 UUID，做幂等 |
| `text`              | string | 否   | 消息文本              |
| `files[]`           | file   | 否   | 附件                  |

逻辑：

1. 校验大小和文件数。
2. 若当前有活动 run 且属于同会话，先停止旧 run，原因 `interrupted_by_user`。
3. 若当前有活动 run 且属于其他会话，返回 409。
4. 保存用户消息和附件。
5. 若会话标题仍为“新聊天”，用首条用户消息前 18 个汉字自动命名。
6. 创建 assistant message 和 assistant turn，占位状态 `running`。
7. 启动 run。
8. 立即通过 SSE 推送 `message.created` 和 `run.started`。

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "user_message_id": "m1",
    "assistant_message_id": "m2",
    "assistant_turn_id": "t1",
    "run_id": "r1"
  }
}
```

用途：停止当前运行。  
逻辑：取消 run 的 context，更新状态为 `stopped`，assistant turn 写入一条简短最终文本：`已停止当前任务。`

### `POST /api/assistant-turns/{id}/user-operated`

用途：用户完成手动操作后的恢复。  
请求：

```json
{
  "client_message_id": "uuid"
}
```

逻辑：

1. 校验该 turn 当前必须是 `waiting_user`。
2. 自动创建一条新的用户消息，文本固定为 `我已经操作`。
3. 新建 assistant turn。
4. 启动新 run，从最新屏幕继续。

## 10.2 工具执行规则

每次模型回复最多允许一个工具调用。  
这是系统级硬约束，不只是提示词建议。

原因：  
一个动作后必须重新截图，模型才能稳。  
所以不要允许“一次回复里连续点三下、打一段字、再回车”这种并发动作。

服务端如果收到多个 tool call，执行第一个，其余直接忽略，并把 tool result 里带上错误说明，促使模型修正。

## 10.3 Agent 主循环

固定执行顺序如下：

1. 收到用户消息。
2. 创建 assistant turn，占位状态 `running`。
3. 抓取初始屏幕截图。
4. 构造 LLM 请求。
5. 得到模型响应。
6. 若响应包含 assistant 文本：
   - 第一步时写入 `phase_text`
   - 后续步骤写入 `progress_items`
7. 若响应包含工具调用：
   - 执行工具
   - 保存 tool_call
   - 工具执行后重新抓图
   - 把 tool result + 新截图再次送给模型
8. 若响应没有工具调用，且有文本：
   - 写入 `final_text`
   - turn 状态改为 `completed`
   - run 结束
9. 若工具是 `ask_user_to_operate`：
   - turn 状态改为 `waiting_user`
   - `final_text = message`
   - `requires_user_action = true`
   - run 暂停，不再继续

## 10.4 上下文管理

不要把完整历史和所有截图都发给模型。要固定压缩策略。

每次发给模型的内容由这几部分组成：

1. 系统提示词
2. 会话摘要 `summary_text`
3. 最近 6 条文本消息
4. 最近 3 条工具结果摘要
5. 当前用户最新消息全文
6. 当前用户上传的图片
7. 当前最新屏幕截图
8. 当前鼠标坐标、屏幕尺寸、活动窗口标题、bash 当前目录

压缩规则：

- `summary_text` 最大 1200 字
- 最近工具输出合计最大 3000 字
- bash stdout/stderr 超长时截断，完整内容只存文件
- 历史屏幕截图不继续重复发送，只发最新一张
- 老的用户图片只在摘要里保留文字说明，不反复发原图

摘要更新规则：

- 每 8 步更新一次
- 或者当前上下文拼装后文本长度超过阈值时更新一次

摘要可以继续用同一个模型做，提示词要求输出“只保留任务目标、已完成步骤、未完成事项、重要页面状态、用户约束”。

## 11. 屏幕抓取、实时画面和鼠标鲁棒性

## 11.1 两种屏幕图像

必须严格区分：

### 给前端看的实时画面
用途：聊天界面“直播”窗口。  
要求：体积小、刷新快。  
格式：JPEG。  
刷新：1 秒 1 帧。  
来源：`/api/screens/live.jpg`，走缓存。

### 给模型看的截图
用途：每个动作后让模型继续判断。  
要求：字清晰、坐标可靠。  
格式优先 PNG。  
若 base64 超过 8MB，则转 JPEG 90；仍超限再按 90% 比例递减缩放。  
一旦缩放，必须把“截图尺寸”和“真实屏幕尺寸”都放进提示词。

## 11.2 鼠标位置标注

模型用图必须做鼠标标注，UI 图不标。

标注方案固定：

- 在鼠标坐标中心画一个红色圆圈
- 从圆圈外侧拉一根红色箭头
- 箭头尾部写“鼠标指针”
- 同时把坐标 `(x, y)` 以文字放进提示词

如果鼠标靠边，箭头方向自动调整，避免文字超出图像边界。

## 11.3 坐标体系

所有动作坐标都使用 X11 根窗口物理像素坐标。  
所有模型截图都尽量保持与根窗口同分辨率。  
这样 `xdotool` 的坐标和截图坐标是一致的，尽量减少缩放偏差。

## 12. bash 实现

## 12.1 必须做成有状态 bash

不要每次 `exec.Command("/bin/bash", "-lc", cmd)`。  
那样 `cd`、环境变量、shell 临时文件、脚本上下文都会丢失。

正确方案：每个会话维护一个持久 PTY bash 进程。

## 12.2 bash 会话实现

启动：

```go
exec.Command("/bin/bash", "--noprofile", "--norc")
```

环境变量至少带上：

- `DISPLAY=:0`
- `LANG=zh_CN.UTF-8`
- `LC_ALL=zh_CN.UTF-8`
- `HOME=<当前用户家目录>`

工作目录初始为：

```text
/opt/easyagentcu/workspaces/{conversation_id}
```

执行命令时写入 PTY：

```bash
printf '__EA_BEGIN__%s\n' "$ID"
<用户命令>
printf '\n__EA_END__%s:$?\n' "$ID"
```

Go 侧持续读 PTY，读到 `__EA_END__` 为止，解析退出码。  
stdout/stderr 不必严格拆分成两个 FD，PTY 模式允许混合输出；前端展示时按统一终端输出渲染即可。  
如果确实要区分 stdout/stderr，就不要 PTY，但会丢失 shell 状态。这里优先保 shell 状态，所以前端把终端输出视为单一流即可。

超时处理：超过 `timeout_seconds` 就向 PTY 对应进程组发 `SIGINT`，再不退出就 `SIGKILL`。

## 13. VNC 与全屏用户操作

## 13.1 x11vnc 启动方式

程序启动时检查本地 `127.0.0.1:5900` 是否可连。不可连则启动：

```bash
x11vnc -display :0 -rfbport 5900 -localhost -forever -shared -nopw -noxdamage
```

这个子进程由 Go 托管，程序退出时一起结束。

## 13.2 浏览器端实现

前端使用 `@novnc/novnc` 的 `RFB` 类连接 `/api/vnc/ws`。  
等待用户操作态连接参数用 `readonly=0`。  
普通查看态可用 `readonly=1`。

## 16. 关键实现细节

## 16.1 全局桌面锁

因为只有一个桌面，所以只能有一个活动 run。  
实现一个全局 mutex + 当前活动 run 记录。  
新 run 启动前先占锁。  
run 结束、暂停或报错时释放。

## 16.2 实时画面缓存

`/api/screens/live.jpg` 不要每次请求都真抓屏。  
实现一个内存缓存对象：

- 最近一帧 jpeg bytes
- 帧时间
- 若距离当前时间不足 800ms，直接返回缓存
- 超过才重新抓

## 16.3 输入法与中文

中文输入默认不能依赖逐键模拟。  
必须默认走 `xclip + Ctrl+V`。  
只有在模型明确需要逐字输入时才允许 `method=type`。

## 16.4 停止与恢复

停止 run 只停止当前任务，不清空会话。  
同一会话里下一条用户消息继续从历史上下文开始。

## 16.5 错误恢复

工具失败时不要直接结束整个 run。  
把失败信息作为 tool result 返回给模型，让模型自己决定重试、等待还是换方案。  
只有这些情况直接结束：

- 达到最大步数
- LLM 连续多次返回不可解析工具参数
- 全局桌面不可用
- 用户手动停止
