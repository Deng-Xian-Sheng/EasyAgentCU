# EasyAgentCU 开发文档（第 2 部分）

## 7. 前端页面与路由设计

## 7.1 路由清单

虽然最终只保留一个实际页面路由，但前端要按“页面级状态”来设计，而不是只做一个巨型组件。

实际路由如下：

`/`  
主应用页面。空会话、已选会话、运行中、等待用户操作、查看历史截图、接管全屏，全部都发生在这里。

`/?c=<conversation_id>`  
主应用页面的会话选中态。前端根据查询参数加载指定会话。

除此之外不再新增独立业务路由。搜索聊天、图片预览、全屏接管都做成当前页上的面板或浮层。

## 7.2 页面级状态

`/` 页面必须支持 6 种稳定状态：

第一种是“未选择会话的首页空态”。显示大输入框、分类胶囊按钮、推荐任务列表。对应你给的首页原型。

第二种是“已选择会话但无消息”。右侧头部仍显示 `EasyAgentCU`，消息区为空，底部输入框可直接发消息。

第三种是“会话正常浏览态”。显示用户消息和助手消息历史。

第四种是“运行中态”。在最后一条助手消息中显示实时屏幕窗口或进展截图窗口，同时顶部或中部显示阶段文本、bash 调用、进展文本。

第五种是“等待用户操作态”。最后一条助手消息切换为回答文本，并显示“我已经操作”按钮。屏幕窗口可点开进入全屏接管。

第六种是“全屏接管态”。以全屏浮层展示 noVNC 画面，用户可直接操作远程桌面。顶部提供退出按钮，底部提供“我已经操作”按钮。

## 8. 原型图还原说明

## 8.1 整体视觉

整体采用暗色主题。页面不是纯黑，而是高灰低饱和的深色背景。推荐全局颜色：

`--bg-app: #151515`  
`--bg-sidebar: #111111`  
`--bg-panel: #1c1c1e`  
`--bg-panel-2: #242426`  
`--bg-bubble-user: #2b2b2d`  
`--bg-input: #2a2a2c`  
`--text-primary: rgba(255,255,255,0.92)`  
`--text-secondary: rgba(255,255,255,0.68)`  
`--text-muted: rgba(255,255,255,0.46)`  
`--border-soft: rgba(255,255,255,0.08)`  
`--border-strong: rgba(255,255,255,0.14)`

圆角必须偏大。侧栏项、用户气泡、输入框、屏幕窗口外壳都要有明显圆角，不能尖锐。阴影很弱，主要靠层次和亮度差区分区域，而不是强阴影。

字体使用系统中文无衬线。标题和关键文字用中等字重，正文使用常规字重。不要出现高饱和品牌色，整体以灰、白、少量浅蓝灰为主。

## 8.2 布局尺寸

桌面宽屏下采用固定双栏布局。

左侧边栏展开宽度 `384px`。  
主内容区最小宽度 `960px`，占剩余全部空间。  
聊天消息列最大宽度 `960px`，左右居中。  
顶部栏高度 `72px`。  
底部输入区整体高度约 `190px`，包含输入框和免责声明。  
底部输入框本体宽度 `min(960px, calc(100vw - 120px))`。  
消息区上下留白都较大，首屏看起来疏朗。

## 8.3 首页空态还原

对应你给的首页图：

右侧主内容区中间偏上，放置一个大输入框。输入框是深灰圆角矩形，宽约 `960px`，高约 `160px`。左上角 placeholder 为“描述任务”。左下角有一个加号按钮，圆形或仅图标，表示添加文件。右下角是发送箭头按钮，灰色禁用态时半透明。

输入框下方先放一排胶囊按钮。按钮边框细、圆角完整、高度约 `48px`。文案如“建议 / 报告 / 操作 / 电子表格 / 演示文稿”。当前选中项底色略亮，图标与文字白色，未选中项透明底配细描边。

胶囊按钮下方是一组推荐任务列表。每一项都是一行，左侧是小图标和标题，右侧是圆形播放样式按钮。项与项之间有横向分隔线，整体宽度比输入框略窄，约 `700px` 左右，居中。

## 8.4 会话态还原

对应你给的聊天原型：

左侧边栏顶部是两个大按钮区域。“新聊天”和“搜索聊天”不是纯图标，而是图标加文字，左对齐。按钮底色默认透明，hover 时出现浅底。下方是“你的聊天”标题，字号比正文略小但比辅助文字略大。再下面是会话列表，每个会话项一行，高度约 `52px`，当前选中项有深灰背景和大圆角。

右侧头部只显示 `EasyAgentCU`，左对齐，距离左边界约 `28px`。头部下方一条极细分割线，贯穿内容区。

消息区内，用户消息总在右边，最大宽度约 `520px`。使用圆角气泡，四边内边距充足，文字居中偏左，看起来像悬浮胶囊。

助手消息总在左边，但不是单一气泡。它是一个宽内容容器，内部的不同块垂直排列，中间留出 `12px` 到 `16px` 间距。比如先出现“我正在点击”，下面出现屏幕窗口，再下面可能出现 bash 工具块，或者只出现最终回答文本。

## 10. 组件详细规格

## 10.1 AppShell

作用是整页布局容器。

左侧固定边栏，右侧主区。边栏支持展开与折叠。折叠时只保留图标列，宽度 `64px`；展开时宽度 `384px`。切换动画时长 `200ms`，使用 `ease-out`。主内容区跟随平滑变化，不出现跳闪。

实现基于普通 `div + CSS Grid` 即可，不需要复杂布局库。  
左栏使用 `overflow-y-auto`。  
右栏由 `topbar + chat body + composer footer` 三部分组成。

## 10.2 Sidebar

侧栏由以下子区块组成：

顶部工具栏。高度 `64px`，右上角放折叠按钮。展开态下按钮靠右，折叠态下居中。按钮为幽灵按钮，hover 出现圆角浅底。图标用 Lucide 的 `PanelLeftClose` / `PanelLeftOpen`。

快捷操作区。包含“新聊天”“搜索聊天”。每一项高度 `44px`，左侧图标，右侧文字。展开态显示文字，折叠态只显示图标并用 Tooltip 提示。基础组件使用 shadcn `Button` + `Tooltip`。

会话分组标题。文案是“你的聊天”。样式为小号半透明文字，左边距与按钮对齐。

会话列表。每项单行显示标题，超长省略。当前选中项背景为 `rgba(255,255,255,0.08)`，普通 hover 为 `rgba(255,255,255,0.04)`。边距小，圆角 `18px`。点击切换会话。右键菜单暂不做。

列表滚动使用 shadcn `ScrollArea`。

## 10.3 EmptyHome

这是首页空态核心组件，对应原型里中央大输入框和下方建议区。

上半部分是大输入框。  
中间是一排分类胶囊。  
下半部分是推荐任务列表。  
整个块在主内容区横向居中，纵向位置约在可视高度的 `38%` 附近，不要完全居中，否则太靠下。

输入框本体基于 `prompt-kit` 的 `PromptInput` 二次封装。  
但默认 prompt-kit 样式不满足原型，需要覆盖成大尺寸版本：

输入框圆角 `32px`。  
背景 `--bg-input`。  
内部上下 padding 很大。  
文本域最小高度 `108px`。  
左下角是上传按钮。  
右下角是发送按钮。  
placeholder 为“描述任务”。

分类胶囊建议基于 shadcn `Button` 自定义，而不是直接用 PromptSuggestion，因为这里更像筛选器，不是纯建议 prompt。推荐数据固定写死前端：

`建议`  
`报告`  
`操作`  
`电子表格`  
`演示文稿`

推荐任务列表每项使用自定义 `SuggestionRow`，左侧图标和标题，右侧圆形按钮。标题最多单行，超长省略。点击任意一项，直接把文案填入输入框，不自动发送。

## 10.4 Composer

底部输入区是整个产品最常用的交互部件，必须单独封装，不要散落在页面里。

组成如下：

最外层固定在底部。宽度和消息列一致。上方可有很淡的背景模糊。  
中间是一块大圆角输入容器。  
容器内部上层是附件预览区，下层是文本输入区和按钮区。  
容器下方是免责声明。

输入组件使用 `prompt-kit` 的 `PromptInput`。  
文件拖拽和选择使用 `prompt-kit` 的 `FileUpload`。  
但附件展示必须自定义成 `AttachmentChip` 列表，支持图片缩略图和文件名。

默认状态下按钮为“发送”。  
当前会话存在运行中的 run 时，发送按钮切换为“停止”。  
点击“停止”调用后端取消接口。

输入规则如下：

回车发送。  
Shift+Enter 换行。  
支持粘贴图片和文件。  
支持拖拽文件。  
支持同时携带文本和多个附件。  
发送后清空输入框和本地附件缓存。  
运行中仍允许用户继续输入，但点击发送前要先提示“当前任务仍在运行，是否停止并发送新消息”，第一版可以做成简单阻止，不允许并发发送。

免责声明文字固定：

`EasyAgentCU 也可能会犯错。请核查重要信息。`

样式用小号、居中、低对比度。

## 10.5 UserMessage

用户消息组件尽量简单。用 `prompt-kit` 的 `Message` 思路，但建议自定义实现，因为原型里用户消息没有头像，也不需要标准 message row。

渲染规则：

整体右对齐。  
文本消息显示为单个深灰气泡。  
有图片附件时，图片卡片显示在用户文本上方。  
多附件纵向排列。  
图片卡片支持点击放大。  
文本区支持 markdown 但默认按纯文本展示即可。

用户上传的带标注图片需要忠实还原原型：  
图片卡片宽度约 `420px` 到 `560px`，圆角 `20px`，背景稍亮，内部图片居中显示。下方可有一个小下载按钮或查看原图按钮，但第一版非必需。

## 10.6 AssistantRunMessage

这是最核心的复杂组件。一个运行中的助手消息不是一段文本，而是一个“运行容器”。它内部根据 run 状态渲染不同块。

容器宽度跟消息列一致，但实际内容块偏左布局。容器内部允许以下块按顺序出现：

阶段回答文本  
bash 工具调用块  
进展文本时间线  
屏幕窗口  
最终回答文本  
用户接管操作面板

这些块的显示规则不是任意组合，而是固定状态机，后面会写。

## 10.7 PhaseTextBlock

用于展示阶段性回应，例如“收到您的最新说明，我会按您圈出的输入框进行操作”。

它看上去像一段左对齐的正文，不加外层卡片。字体比普通说明略大，行高舒展。最大宽度建议 `760px`。这块在屏幕窗口上方出现。

一旦最终回答文本出现，这块必须从界面移除，不再显示。

这块可以直接用 `prompt-kit` 的 `Markdown` 渲染内容，但要关闭默认 prose 强样式，改成更轻的聊天正文样式。

## 10.8 BashToolBlock

bash 调用块用于展示模型调用 bash 的过程和结果。推荐基于 `prompt-kit` 的 `Tool` 组件思路，但不要直接使用默认外观，要封装成更贴近原型的样式。

块结构分三层：

头部显示 `bash` 标签、状态徽标、执行时长。  
中部显示命令正文，使用 `prompt-kit` 的 `CodeBlock`。  
底部按需显示标准输出、标准错误、退出码。输出超长时默认折叠，支持展开查看。

状态只有四种：

`pending`  
`running`  
`success`  
`error`

`running` 时头部右侧显示小型 loader。  
`error` 时底部 stderr 区域用暗红色弱提示，但不要鲜红刺眼。

一旦最终回答文本出现，这个块也必须从界面移除。

## 10.9 ProgressTimeline

进展文本位于屏幕窗口上方，默认收起，只显示最新一条。

这个组件不能直接用 prompt-kit 现成组件，必须自定义。建议内部使用 shadcn `Collapsible`。

收起态：  
只显示一行，左侧是状态词或小图标，中间是最新进展文本，右侧是展开箭头。  
整行可以点击展开。  
文本超长单行省略。  
风格接近原型里的“我正在点击 >”“等待页面加载 …”。

展开态：  
显示纵向时间线。每一项左边是小圆点，圆点之间用竖线连接。右边是进展文本。最新项高亮。  
每一项都可点击。点击后，下方屏幕窗口切换到该进展对应的截图。  
当前选中的进展项需要有浅底高亮。  
展开区域最大高度约 `360px`，内部可滚动。

每条进展数据必须绑定 `progress_id` 和 `screenshot_id`，不能只靠数组下标。

## 10.10 ScreenWindow

这是屏幕画面显示组件，是产品的视觉中心。

外层看起来像一个桌面窗口。由三部分组成：

顶部是可选标题区。显示当前进展文案，比如“等待页面加载…”或“我正在点击”。  
中间是画面区。显示截图或 noVNC 实时画面。  
底部是时间进度条。左到右是历史节点，最右侧固定为“直播”。

整体外框是大圆角矩形，背景比页面稍亮，内边距明显。内部真实画面区域再包一层较小圆角。比例固定 16:9，默认最大宽度约 `980px`，高度自适应。页面窄时整体缩小，但始终保持比例。

画面区有两种模式：

历史模式。显示某个进展对应的截图。图片 `object-contain`，上下左右居中。  
实时模式。显示 noVNC 画布。右下角可有一个很淡的“实时”标识。

用户点开全屏时，如果当前是实时模式，进入 noVNC 全屏接管。  
如果当前是历史模式，进入图片预览全屏，不可操作桌面。

## 10.11 ScreenProgressBar

这个组件不能直接用原生 slider 外观，要自定义成原型里的“时间线 + 节点”。

底部是一条细横线，颜色浅灰。  
线上的每个进展节点渲染为一个小圆点。  
当前选中节点变大并高亮。  
最右端固定一个“直播”节点，文字显示“直播”或“实时”。  
可以点击任意节点跳转。  
可以拖动滑块头左右移动，但第一版直接做“点击节点切换”已经足够稳定；拖动可以通过 Radix Slider 做二期增强。若第一版实现拖动，必须支持吸附到最近节点。

点击时间线中的某个点后，进展时间线列表也要同步高亮对应项。  
点击进展时间线列表某项后，底部时间条也要同步切换。

## 11. 助手消息显示状态机

这一段必须严格执行，否则界面会乱。

定义一个助手 run 消息包含这些逻辑字段：

`phase_text`  
`bash_blocks[]`  
`progress_items[]`  
`screen_mode`，值为 `live | history | hidden`  
`final_answer`  
`needs_user_action`，布尔值

前端渲染规则如下：

当 run 刚创建但还没有截图时，显示 `screen_mode=live` 的屏幕窗口，可以只显示加载态 noVNC 容器。此时如果还没有阶段文本，就显示一条简短 loading 文案，例如“正在思考”。

当 run 有阶段文本且还没有最终回答时，显示阶段文本。  
当 run 有 bash 工具块且还没有最终回答时，显示 bash 工具块。  
当 run 有进展项且还没有最终回答时，显示进展时间线和屏幕窗口。此时屏幕窗口默认显示实时画面，用户点击历史节点时切到历史模式。

当 `final_answer` 出现后，必须立即隐藏：

阶段文本  
bash 工具块  
进展时间线  
屏幕窗口

只保留最终回答文本。  
这条规则必须写进前端条件渲染里，不允许同时显示“最终回答”和“进展屏幕”。

当 `needs_user_action=true` 时，在最终回答文本下方显示“我已经操作”按钮。点击后自动发送一条用户消息：`我已经操作`。

## 12. 全屏接管设计

## 12.1 打开方式

用户点击实时屏幕窗口时，打开全屏浮层。  
只有实时模式可以进入可操作接管。历史截图模式点开后只能看大图，不可操作。

## 12.2 全屏浮层样式

浮层覆盖整个 viewport，背景接近黑色。  
顶部有窄工具栏，左侧是“退出全屏”，中间显示当前状态，右侧可显示连接状态。  
中间是最大化的 noVNC 画面，保持比例，黑边填充。  
底部固定一个主按钮“我已经操作”。

## 12.3 行为要求

进入全屏后，前端不拦截鼠标键盘，直接让 noVNC 接收操作。  
用户点击“我已经操作”后：

先退出全屏。  
再自动向当前会话发送一条用户消息，内容固定为 `我已经操作`。  
发送成功后，当前 run 若处于 `waiting_user`，后端应继续执行新的 run 或继续原流程，后面在后端部分定义。

## 13. 基于现有库的组件选型结论

这里给出明确结论，程序员照着选，不需要再判断。

`PromptInput`：使用 prompt-kit，作为 Composer 核心输入框底座。  
`FileUpload`：使用 prompt-kit，处理拖拽与选择文件。  
`Markdown`：使用 prompt-kit，用于阶段文本和最终回答文本。  
`CodeBlock`：使用 prompt-kit，用于 bash 命令与输出。  
`ChatContainer`：可以使用 prompt-kit，但由于本项目有固定底部输入区和复杂消息结构，建议只借鉴逻辑，不直接整套接入。  
`Tool`：只借用思路，不直接用默认样式。封装成 `BashToolBlock`。  
`Loader`：使用 prompt-kit，显示运行中小动画。  
`Dialog`：使用 shadcn/ui，做图片预览和全屏接管容器。  
`Collapsible`：使用 shadcn/ui，做进展文本展开收起。  
`ScrollArea`：使用 shadcn/ui，做侧栏和长进展滚动。  
`Tooltip`：使用 shadcn/ui，做折叠侧栏图标提示。  
`Slider`：使用 Radix Slider，自定义时间轴拖动。第一版也可暂缓拖动，仅保留点击节点切换。  
`Button`、`Input`、`Badge`、`Separator`、`DropdownMenu`：使用 shadcn/ui。
